# D-PLOG 서버 배포 가이드 (AWS & Cloud Deployment Strategy)

현재 대표님의 로컬 PC(Mac/Windows)에서 성공적으로 검증된 **Stealth 우회 크롤러**를 AWS, 카페24, 혹은 클라우드 리눅스 서버에 올릴 때 반드시 고려해야 하는 **4가지 핵심 아키텍처 가이드**입니다.

---

## 1. 클라우드 IP 페널티 및 프록시 (가장 중요 ⭐)
놀랍게도 로컬 PC 코드 그대로 AWS EC2에 올려서 돌리면 차단당할 확률이 매우 높습니다.
* **이유:** 네이버 방화벽은 브라우저 지문만 검사하는 것이 아니라, **접속 IP의 출신지(ASN)** 도 검사합니다. 통신사 일반 가정용 IP(KT, SKT)는 살려두지만, **AWS, GCP, Azure 같은 '데이터센터 가상 IP 대역'**에서 접근하는 트래픽은 사람이 아니라고 간주하여 기본적으로 적대적인 검문(우대 캡차)을 때립니다.
* **서버 배포 시 해결책:** AWS 서버에 올릴 경우, 스크립트에 반드시 **레지덴셜 프록시(Residential Proxy, 가정용 IP 우회 서비스)** 를 연동해야 합니다.
  * *로직 변경:* AWS 서버(명령만 내림) ➡️ 한국 가정용 IP 프록시를 경유 ➡️ 네이버 접속
  * *코드 예시:* `browser.new_context(proxy={"server": "http://가정용IP:PORT"})`

## 2. 권장 서버 스펙 (Sizing & Memory)
서버에 크롤러를 24시간 상주(Cron Job)시킬 때의 권장 하드웨어 제원입니다.
* **최소 사양:** RAM **4GB 이상** (AWS 기준 `t3.medium` 이상 권장)
  * *이유:* `Playwright Chromium`이 백그라운드에서 돌아갈 때 Chrome 프로세스와 탭(Semaphore 15 기준)이 각각 메모리를 소모하므로 RAM 1~2GB짜리 깡통 서버(`t2.micro` 프리티어)에서는 실행 도중 `Out of Memory (OOM)`로 서버가 다운됩니다.
* **권장 사양:** RAM **8GB 이상** (다수의 타겟 키워드 데일리 스케줄링 시)

## 3. Docker(도커) 기반 컨테이너 배포 필수
서버(리눅스)는 윈도우/맥과 달리 화면(GUI)을 그리는 그래픽 라이브러리가 아예 존재하지 않습니다. 그래서 리눅스에 크롤러를 그냥 올리면 "글꼴이 없다, 브라우저 엔진이 없다"며 끝없는 에러를 뱉습니다.
* **해결책:** 마이크로소프트가 공식 지원하는 **Playwright 전용 Docker 이미지**를 사용해 배포해야 합니다.
* **권장 환경:** `mcr.microsoft.com/playwright/python:v1.40.0-jammy`
* 이 컨테이너 안에 폰트, 리눅스용 크롬 엔진, 그래픽 가상 드라이버가 전부 포함되어 있어서 AWS에 도커로 올리기만 하면 로컬처럼 100% 똑같이 동작합니다.

## 4. 백그라운드 스케줄링 (Spring Boot / Lambda)
단순 파이썬 스크립트가 아니라 '시스템'으로 조립하기 위한 아키텍처입니다.
* **파이프라인 구성:**
  1. **Spring Boot (백엔드):** 매일 새벽 4시마다 파이썬 컨테이너 또는 AWS Lambda(또는 ECS)로 크롤링 명령 API를 호출.
  2. **Python Crawler (Stealth):** AWS 서버에서 깨어나 가정용 Proxy IP를 달고 네이버 타겟 키워드를 병렬로 긁어옴.
  3. **DB 적재:** 수집된 순위를 PostgreSQL 데이터베이스에 저장 "예: {date: 2026-03-25, store_id: 1, rank: 3}"
  4. **React Dashboard:** 다음 날 대표님이 D-PLOG 대시보드에 접속하면 아름다운 우상향 그래프 차트 확인 완료.

---
### 💡 요약 결론
현재 확보된 엔진은 극강의 성능을 자랑하지만, 이걸 **AWS에 올리는 순간 "서버 IP"라는 태생적 한계로 인해 차단율이 올라갑니다.** 따라서 프로덕션 서버로 이관하실 때는 **[프록시 IP 도입]과 [Docker 환경 세팅]** 이라는 두 가지 무기를 반드시 장착해야 완벽한 서버리스/자동화 파이프라인이 완성됩니다.

## 🚀 대안 강력 추천: 온프레미스(On-Premise) 미니 PC 전략 (가성비 최강)
AWS 클라우드의 복잡한 도커 세팅과 유료 프록시 비용을 완전히 제거하는 **"가장 현실적이고 강력한 방법"**은 사무실이나 매장에 **미니 PC(Intel NUC, Mac Mini 등)를 24시간 켜두는 것**입니다.

1. **무료 가정용 IP 획득:** 사무실 인터넷(KT/SKT/LG)을 쓰기 때문에 네이버가 볼 때는 **가장 완벽한 '일반 사람'의 IP**입니다. 비싼 프록시 IP 시스템이 전혀 필요 없습니다!
2. **Docker 생략:** 윈도우나 맥 OS가 네이티브로 깔려 있기 때문에 리눅스 그래픽 오류가 없습니다. 즉, 도커를 깔 필요도 없이 지금 제(AI)가 짜드린 파이썬 코드를 윈도우 스케줄러나 크론(Cron)에 올려서 바로 돌릴 수 있습니다.
3. **비용 제로:** 매달 3~5만 원씩 나가는 AWS 서버비나 프록시 통신비가 없습니다. 오직 미니 PC 전기세만 나옵니다.
4. **언제 AWS로 가야 하는가?** 만약 대표님이 "수천 개의 가맹점을 모니터링하는 대규모 SaaS"를 런칭한다면 미니 PC 하나로는 연산이 부족하니 AWS + 프록시 라우팅 구조로 가야 합니다. 하지만 **내 매장과 주변 상권을 모니터링하는 수준이라면 미니 PC를 켜두는 것이 클라우드(AWS)보다 100배 더 똑똑하고 효율적인 아키텍처**입니다.
