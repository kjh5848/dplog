# D-PLOG PRD (외식업 소상공인 플레이스 노출 진단)

- 문서 목적: 외식업(카페/식당/주점 등) 소상공인이 **네이버 플레이스 노출/유입을 개선**할 수 있도록, 진단 → 리포트 → 실행 플랜까지 제공하는 D-PLOG의 제품 요구사항을 정리한다.
- PDF 반영 상태: **미반영** (현재 레포 경로에서 참조 PDF를 찾지 못함)

## 1. 한 줄 소개
플레이스 URL/가게 정보 + 키워드(대표/희망) 입력만으로 **실시간 노출 순위와 근거 기반 개선 리포트**를 만들어 주는 외식업 소상공인용 진단 서비스.

## 2. 문제 정의
- 외식업 사장님은 “무엇부터 바꿔야 노출이 오르는지”를 알기 어렵다.
- 광고/대행 의존도가 높고, 내부 운영(메뉴/사진/리뷰/소개문) 개선의 우선순위가 불명확하다.
- 키워드별 성과를 정량적으로 추적하기 어렵고, 변화가 있어도 원인/근거를 연결하기 어렵다.

## 3. 목표(Goals)
- (G1) 10분 내에 진단 완료: 입력 → 결과(요약)까지 빠르게 제공
- (G2) 근거 기반 리포트: “왜” 해야 하는지 근거(요인/경쟁/콘텐츠 전략)를 함께 제공
- (G3) 실행 가능성: 오늘 바로 실행할 수 있는 체크리스트/우선순위 액션 제공
- (G4) 반복 개선: 리포트 히스토리/비교를 통해 개선 사이클을 돌릴 수 있게 함

## 4. 비목표(Non-goals)
- 노출/순위의 “보장” (정책/환경 영향이 크므로 결과 보장 금지)
- 네이버 플레이스의 자동 수정/대리 운영(자동 포스팅/자동 리뷰 조작 등 정책 리스크)
- 복잡한 마케팅 올인원(초기에는 진단/리포트/실행에 집중)

## 5. 타겟 사용자/페르소나
- P1. 1인 사장(카페/식당)
  - 시간 부족, 마케팅 지식 부족, 즉시 실행 가능한 안내를 원함
- P2. 매장 운영자(직원/점장)
  - 매장 개선 업무를 맡았고, 보고 가능한 지표/리포트를 원함
- P3. 소규모 대행/컨설턴트(선택)
  - 여러 매장을 동시에 관리, 리포트 템플릿/히스토리 비교가 필요

## 6. 핵심 사용자 여정(User Journey)
1. 랜딩 진입
2. “무료로 진단 시작” 클릭
3. 가게 정보 수집
   - (A) 플레이스 URL 입력(권장)
   - (B) 직접 입력(상호/주소/카테고리 등)
   - (C) 이미 등록된 매장 선택
4. 키워드 입력
   - 대표 키워드 5개
   - 희망 키워드 5개
   - 유효성: 중복/금칙어/길이 제한
5. 실시간 순위 조회
   - 외부 순위 수집 서비스(내순이) API 호출
6. 진단 리포트 생성
   - 순위 결과 + RAG 지식 결합
   - 요약/근거/개선 포인트/즉시 실행/우선순위
7. 저장/히스토리
   - 리포트 조회/비교

## 7. 핵심 기능 요구사항 (Functional Requirements)

### 7.1 랜딩/온보딩
- FR-L1: 서비스 소개(외식업 소상공인 대상) 랜딩 페이지 제공
- FR-L2: CTA(무료 진단 시작) → 진단 플로우 진입
- FR-L3: FAQ/보안 안내(민감정보 처리 원칙)

### 7.2 가게 정보(스토어) 수집
- FR-S1: 가게 정보 입력 폼 제공
- FR-S2: 플레이스 URL 기반 메타 수집(연동 가능 시)
  - 수집 예시: 상호명, 카테고리, 주소, 전화번호, 운영 시간, 대표 이미지
- FR-S3: 정규화된 스토어 프로필 저장

### 7.3 키워드 입력/관리
- FR-K1: 대표 키워드 5개 + 희망 키워드 5개 입력
- FR-K2: 유효성 검사
  - 중복 제거
  - 길이 제한
  - 금칙어(운영 정책에 따라)
- FR-K3: 키워드 세트 저장 및 진단 요청 컨텍스트 연결

### 7.4 순위 조회(내순이 연동)
- FR-R1: 키워드별 실시간 순위 조회
- FR-R2: 결과 저장
  - 키워드, 순위, 변동, 노출 위치(가능한 경우)
- FR-R3: 실패/타임아웃 처리
  - 캐시/최근 스냅샷으로 대체 또는 부분 결과 제공

### 7.5 진단 리포트(Agent + RAG)
- FR-P1: 입력 조합
  - 스토어 정보 + 키워드 세트 + 순위 결과
- FR-P2: RAG 지식 검색
  - 노출 요인/경쟁 구도/콘텐츠 전략 등
- FR-P3: 리포트 생성
  - 요약
  - 근거
  - 개선 포인트
  - 즉시 실행 항목
  - 우선순위
- FR-P4: 리포트 저장/조회/히스토리

### 7.6 비동기 잡(권장)
- FR-J1: 진단 요청은 비동기 처리 가능
  - `POST /v1/diagnosis` → `202 Accepted + jobId`
  - `GET /v1/diagnosis/{jobId}` → 결과 조회
  - `GET /v1/diagnosis/{jobId}/status` → 상태 조회
- FR-J2: 상태 전이
  - `PENDING → RUNNING → SUCCESS/FAILED` (+ 필요 시 `PARTIAL`)
- FR-J3: 재시도/실패 큐(운영 단계)

## 8. 비기능 요구사항 (Non-Functional Requirements)
- NFR-1: 보안
  - 외부 API 키/토큰은 환경변수로 관리
  - 클라이언트에 민감정보 노출 금지
- NFR-2: 성능
  - 순위 조회/리포트 생성 타임아웃 설정
  - 응답 지연 시 부분 결과/재시도 설계
- NFR-3: 신뢰성
  - 외부 API 실패 시 캐시/부분 결과
  - 작업 중복 방지(아이템포턴시 키)
- NFR-4: 관측성
  - 요청 ID 전파, 구조화 로그, 에러 트래킹
- NFR-5: 접근성
  - 키보드 포커스, 대비 4.5:1, reduced motion

## 9. 성공 지표(가설)
- Activation: 랜딩 → 진단 완료 전환율
- Time-to-value: 진단 시작 → 요약 리포트까지 소요 시간
- Retention: 재진단(히스토리 비교) 비율
- Outcome(간접): 특정 키워드의 평균 순위 개선(기간/표본 명시 필요)

## 10. 오픈 이슈 / PDF 기반 확정 필요
- PDF에서 확정해야 할 항목
  - 타겟 업종/지역/규모 정의
  - 요금제/과금 정책(무료 범위, 제한)
  - 리포트 템플릿(섹션 구성, 톤)
  - 금칙어/정책(운영 가이드)
  - 경쟁 분석 범위(동일 카테고리/반경 등)
