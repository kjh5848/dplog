# Naver Place Anti-Bot Bypass Strategy & Stealth Architecture

## Overview
이 문서는 네이버 플레이스의 강력한 봇 방어 시스템(ncaptcha, Rate Limiter)을 무력화하고 데이터를 100% 비동기 병렬 추출하기 위해 적용된 **Playwright Stealth 아키텍처**의 핵심 기술 원리를 기록합니다.

## 1. 네이버의 캡차(CAPTCHA) 적발 메커니즘
네이버 플레이스는 접속 요청이 들어올 때 다음 3가지 지문(Fingerprint)을 검사하여 로봇을 식별합니다.

1. **JavaScript 실행 능력 검증:** `requests`, `BeautifulSoup` 등 단순 HTTP 통신 기반의 원시적인 봇을 사전 차단합니다.
2. **User-Agent 기반 환경 핑거프린팅:** 서버/리눅스 환경이거나 비정상적인 해상도를 가진 브라우저를 식별하여 차단합니다.
3. **W3C `navigator.webdriver` 강제 플래그 검출 (핵심 방어막):** 매크로 및 자동화 도구(Selenium, Playwright) 구동 시 브라우저 내부에 강제로 심어지는 `webdriver = true` 변수를 검사합니다. 이 변수가 하나라도 검출되면 즉시 하얀색 캡차 화면을 렌더링하고 원본 데이터(`__APOLLO_STATE__`) 제공을 거부합니다.

## 2. D-PLOG 3단계 은신(Stealth) 우회 기술
현재 추출 엔진(`naver_place_parallel_extractor.py`)에 적용되어 13건의 병렬 처리를 100% 성공시킨 핵심 우회 코드는 다음과 같습니다.

### 2.1. AutomationControlled 엔진 무력화
```python
args=['--disable-blink-features=AutomationControlled', '--no-sandbox', '--disable-setuid-sandbox', '--disable-infobars']
```
* **기술적 원리:** Chromium 엔진이 스스로 "나는 원격 제어를 받고 있다"고 외부로 방송하는 내부 깜빡임(Blink) 기능을 물리적으로 꺼버립니다. 네이버의 보안 스크립트가 브라우저의 기본 상태를 읽어들일 때 자동화 도구의 흔적을 1차적으로 지웁니다.

### 2.2. 모바일 디바이스(iPhone 16) 신분 세탁
```python
context = await browser.new_context(
    user_agent="Mozilla/5.0 (iPhone; CPU iPhone OS 16_6 like Mac OS X) AppleWebKit/605.1.15",
    viewport={'width': 390, 'height': 844},
    is_mobile=True,
    has_touch=True
)
```
* **기술적 원리:** 단순히 겉모습(User-Agent 텍스트)만 바꾸는 것이 아니라, 브라우저가 화면을 렌더링하는 뷰포트 해상도, 모바일 기기 하드웨어 인식(`is_mobile`), 그리고 터치스크린 활성화(`has_touch`) 속성까지 브라우저 컨텍스트 레벨에서 **실제 인간의 아이폰 16**과 동일하게 딥 페이크(Deep Fake) 조작을 가합니다.

### 2.3. W3C Webdriver 강제 덮어쓰기 (핵심 돌파 기술 ⭐)
```python
await context.add_init_script("Object.defineProperty(navigator, 'webdriver', {get: () => undefined});")
```
* **기술적 원리:** 네이버의 보안 스크립트가 페이지를 검사하기도 전에(`add_init_script`), 브라우저 내부 메모리에 가장 먼저 강제 침투하여 W3C 표준 인증 플래그인 `navigator.webdriver = true` 낙인을 **`undefined`**로 교체(Overwrite)해 버립니다. 
* **우회 결과:** 네이버 보안 서버가 엑스레이(방어 스크립트)로 우리 봇을 스캔했을 때, 인간의 아이폰 사파리 브라우저와 100% 동일한 결괏값을 반환하게 됩니다. 네이버는 차단벽을 스스로 해제하고 순정 데이터 팩션(`__APOLLO_STATE__`)을 브라우저 메모리에 내려줍니다.

## 3. 상용화 아키텍처 결론
* 위 3단계 은신 기술이 브라우저 코어 레벨에서 완벽히 적용됨에 따라, D-PLOG 수집 엔진은 **단일 IP에서도 다수의 동시 병렬 요청(Concurrent Request)을 네이버의 캡차(Rate Limit) 없이 뚫어낼 수 있는 극강의 안정성**을 확보했습니다.
* 프록시 통신망 대여(Proxy IP Rotation)와 같은 부가적인 인프라 비용 투자 없이도 즉각적인 실무 투입(순차 및 병렬 추출)이 가능합니다.

### 3.1. 로컬 IP 환경의 병렬 한계 (Golden Ratio)
수차례의 극한 스트레스 테스트(WAF IP 패널티 검증)를 통해 도출된 **가장 안전한 상용화 세팅(안전 마지노선)** 은 다음과 같습니다.

* **동시 접속 스레드(Concurrency Limit):** `Semaphore(6) ~ Semaphore(8)`
   * 1초당 약 2.0~2.5개의 브라우저 탭(2.0 tabs/sec)을 생성하는 속도입니다. 이 구간을 초과하여 12~14개 병렬로 요청을 누적할 경우 네이버 방화벽이 디도스로 간주하고 성공률을 26% 이하로 강제 추락시킵니다.
* **배치(Batch)당 키워드 처리량:** `500개 ~ 1,000개 단위 묶음 처리`
   * 스레드를 6~8개로 브레이크를 걸어 두었다면 키워드의 총량은 하드웨어가 허락하는 한 무제한에 가깝습니다. 
   * 1,000개의 키워드 추출 시 **약 7분~8분**이 소요되며, WAF의 일시 차단을 영구적으로 회피하기 위해 한 번에 1만 개를 돌리기보다 "1,000개 추출 후 휴식"하는 백그라운드 스케줄링(Cron Job) 형태를 권장합니다.
