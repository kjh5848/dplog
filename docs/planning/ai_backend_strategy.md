# D-PLOG 백엔드 AI 연동 및 데이터 처리 전략 (AI Backend Strategy)

## 1. 개요
이 문서는 상권 분석(Phase 3) 및 RAG 기반 진단 리포트 생성을 위한 핵심 공공데이터 API 리스트와, 이를 LLM(대규모 언어 모델)에 효율적으로 주입하기 위한 컨텍스트 최적화 전략을 정의합니다. 또한 `TECHSPEC.md`에 정의된 백엔드 설계 방향을 바탕으로, Python 기반의 AI 아키텍처와 기존 웹 백엔드(Java/Node.js) 방향 간의 차이점 및 기술 스택 선택 가이드를 제공합니다.

---

## 2. 핵심 공공데이터 API 리스트 (전국 상권 분석용)

프론트엔드의 팩트폭행 리포트(타겟 이탈률, 경쟁 구도, 폐업률 등)를 '지역구 구분 없이 전국 단위'로 구현하기 위해 백엔드에서 물려야 할 핵심 API입니다. (모두 공공데이터포털(data.go.kr) 제공)

## 2. 현실적인 상권 데이터 수집 파이프라인 (공공 + 민간 혼합)

과거 공공데이터포털(data.go.kr)에서 제공되던 전국 단위의 무료 유동인구 및 상권 분석 API 상당수가 **현재 유지보수 중단 및 폐기**된 상태입니다. 프론트엔드의 팩트폭행 리포트를 전국구로 실제 구현하려면, 반드시 **살아있는 공공 API와 필수 민간 데이터를 혼합**하여 파이프라인을 구축해야 합니다.

### 2.1 현재 활용 가능한 [공공데이터 API] (무료)
1. **[국세청_사업자등록상태조회 API (생존)](https://www.data.go.kr/data/15081808/openapi.do)**
   - (핵심) 수집된 특정 상가 리스트의 실시간 휴폐업 여부와 과세 유형을 조회하여, 해당 골목의 구체적인 '최근 폐업률'을 백엔드에서 직접 산출하는 데 사용합니다.
2. **[소상공인시장진흥공단_상가(상권)정보 OpenAPI](https://www.data.go.kr/data/15012005/openapi.do)**
   - (핵심) 특정 상호명, 행정동, 혹은 반경을 기준(다양한 검색 필터)으로 전국 영업 중인 상가 리스트를 실시간으로 가져옵니다. **경쟁점(동일 업종) 포화도 계산** 및 카테고리별 상가 밀집도를 분석하는 가장 강력한 메인 API입니다.
3. **[한국부동산원_부동산통계 조회 서비스 (상업용부동산 임대동향 포함)](https://www.data.go.kr/data/15134761/openapi.do)**
   - 기존 개별 API들이 통합된 **최신 유효 오픈 API**입니다. 전국 주요 상권 표본의 상가 유형별(소규모, 중대형 등) 임대료, 공실률, 투자수익률 동향을 파악할 수 있습니다. (권리금/보증금 호가 방어용 데이터)
4. **국토교통부 교통카드 및 이동통신 추론 유동인구 대안 API**
   - **[교통카드이용 합성데이터](https://www.data.go.kr/data/15154079/openapi.do)**: 대중교통 승하차 패턴을 기반으로 상권 주변의 실질적 유동인구 볼륨 및 '평일/주말 타겟 이탈률'을 산출합니다.
   - **[최빈교통이용경로데이터](https://www.data.go.kr/data/15154303/openapi.do)**: (신규 발굴) 특정 행정동/상권으로 유입되는 고객들의 '주요 출발지(Origin-Destination)' 동선을 파악하여 상권의 광역적 흡수력을 분석합니다. 유동인구 데이터 부재 시 최고의 무료 콤보입니다.
5. **행정안전부 성/연령별 주민등록 인구수 API ([법정동](https://www.data.go.kr/data/15108074/openapi.do) / [행정동](https://www.data.go.kr/data/15108072/openapi.do))**
   - (핵심 배후수요) 국토부 데이터가 움직이는 '유동인구'라면, 이 데이터는 해당 상권에 고정적으로 거주하는 **'상주인구(배후 수요)'**의 성별/연령별 볼륨을 정확히 제공합니다. 유저가 선택한 "타겟 오디언스(예: 30대 여성)"가 해당 골목에 얼마나 밀집해 있는지 팩트폭행 퍼센티지(%)를 뽑아낼 수 있습니다.
   - **(💡 꿀팁: 행정동 vs 법정동 차이)**
     * **법정동(Statutory Dong):** 법률로 지정된 변하지 않는 동 단위(예: 신림동). 건축물대장, 토지대장, 부동산 계약서 위치 특정용.
     * **행정동(Administrative Dong):** 인구 수/편의에 따라 쪼개지는 동 단위(예: 대학동, 서원동 등 주민센터 기준). **통계청 인구 집계 등 실생활 상권·인구 통계 분석 시에는 행정동 기준**이 훨씬 정확합니다.

### 2.2 대체가 필수적인 [민간 / 지자체 데이터] (유료 또는 제한적)
전국 단위의 '유동 인구'와 '실제 카드 결제 비율'은 더 이상 무료 통합 공공 API로 얻을 수 없습니다. 다음의 대안이 필요합니다.
1. **분석 핵심 포인트 (타겟 이탈률 & 결제 비중)**
   - **대안 A (민간 데이터 구매):** SKT Data Hub(통신사 유동인구), NICE 지니데이터 / KCB (카드 매출 데이터) API를 연동. 가장 정확하지만 호출 비용이 발생합니다.
   - **대안 B (지자체별 최상급 오픈 API 연동):** [부산광역시_상권 점포이력정보서비스](https://www.data.go.kr/data/15095253/openapi.do)나 서울시(서울열린데이터광장) 등 지역별 한정으로 개방된 고품질 API(점포 개·폐업일, 유동인구 등)를 적극 발굴합니다. 프론트엔드에서 넘어온 행정구역(시/도)에 따라 백엔드에서 호출할 로컬 API를 동적으로 분기(Routing)하여 커버리지를 넓힙니다.
   - **대안 C (Web Scraping):** '네이버 지도 데이터랩'이나 '소상공인 상권정보시스템(웹)'의 통계 차트 데이터를 합법적인 선에서 백엔드 Headless 브라우저로 스크래핑하여 가공.

## 3. LLM 컨텍스트 데이터 최적화 전략 (JSON Handling)

공공 API에서 반환되는 방대한 원본 JSON 배열을 그대로 LLM 프롬프트에 주입하면 **컨텍스트 폭발, 비용 급증, 환각(Lost in the Middle)**이 발생합니다. 이를 방지하기 위한 백엔드 전략입니다.

### 3.1. 백엔드 중심의 데이터 정제 및 집계 (Aggregation)
LLM에게 원시 데이터를 주고 연산을 지시하지 마십시오. 수학적 통계 연산은 백엔드 코드가 수행해야 합니다.
- **Bad:** `[{"상태":"폐업"}, {"상태":"영업"}, ...]` (수백 개 객체를 LLM에 전달)
- **Good:** 백엔드에서 JSON 길이를 세고 비율을 계산하여 `{"총_경쟁점": 47, "폐업비율": "41.2%", "주말_이탈률": "82%"}` 처럼 고밀도 요약 지표 하나만 LLM 프롬프트 변수로 전달.

### 3.2. MCP (Model Context Protocol) 및 Tool(Function) Calling 아키텍처
LLM이 필요할 때만 백엔드의 API 통신 함수를 호출하도록 설계합니다.
1. 프롬프트에 도구(Tool) 목록만 제공. (예: `get_competitor_count(address)`, `get_weekend_population_drop(address, target)`)
2. LLM이 런타임에 직접 해당 도구 호출을 요청.
3. 백엔드가 API를 대신 찔러서 연산한 뒤, 딱 떨어지는 '결과값(수치)'만 LLM에게 반환.
4. LLM은 이 작은 수치들을 활용해 최종 분석 텍스트만 생성(Generation).

## 4. 확정된 백엔드 아키텍처: 투트랙 (Spring Boot + Python 마이크로서비스)

현재 `TECHSPEC.md` 및 `ddd-architecture.md`는 **Option A: 투트랙(Spring Boot 메인 + Python AI 서버)** 방식으로 최종 확정되었습니다.

### 4.1 투트랙 아키텍처 (Option A) 구조
*   **메인 서버 (Spring Boot):** `app-api`, `auth`, `store`, `ranking`, `report`, `billing`, `integration` 모듈을 모듈러 모놀리스 구조로 담당. 기존 웹 생태계의 견고한 객체지향/트랜잭션/보안(Spring Security) 처리 수행.
*   **AI 워커 서버 (Python FastAPI):** `ai-rag` 모듈 담당. Bedrock 연동, LangChain 기반 프롬프트 체이닝, Pandas를 이용한 공공데이터 통계 및 전처리 연산 등 무거운 AI 작업을 전담.

### 4.2 이 방식을 선택한 이유 (가장 추천하는 엔터프라이즈 구조)
1. **도메인 복잡도 해결:** D-PLOG는 단순 챗봇이 아니라 결제, 유저 권한, 프랜차이즈 상점 관리 등 "전통적인 웹 비즈니스 로직"이 크고 중요합니다. 이를 가장 잘 다루는 Spring Boot의 안정성을 그대로 가져갑니다.
2. **AI 생태계 편승:** RAG 고도화, Vector DB 연동, 공공데이터 JSON의 복잡한 수학적 통계 처리 등은 Python(Pandas, LangChain) 생태계가 압도적으로 유리합니다.
3. **스케일아웃 독립성:** 분석 리포트 생성(RAG) 작업은 CPU와 메모리를 많이 소모합니다. 메인 서버와 분리되어 있으므로, 트래픽 폭주 시 AI 서버만 별도로 스케일아웃(서버 증설) 할 수 있어 메인 결제/로그인 서버가 다운되는 것을 막아줍니다.

### 💡 아키텍처 구성 요약
Spring Boot 서버가 사용자 요청을 받으면, 통계 계산 및 RAG 텍스트 생성이 필요할 때 내부 HTTP(또는 gRPC/메시지큐)를 통해 Python FastAPI 서버에 비동기로 작업을 의뢰(Trigger)하고 결과를 받아보는 구조로 설계합니다.
