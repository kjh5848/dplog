# 클라우드 스크래핑 인프라 투어 가이드 (AWS & Cloud Options)

본 문서는 로컬 컴퓨터와 단일 IP의 트래픽 한계(IP Rate Limit)를 돌파하고, 수십~수백 개의 스레드(워커)를 무한히 배포하기 위해 사용할 수 있는 **AWS 중심의 클라우드 아키텍처 기술 조사 보고서**입니다.

D-PLOG 크롤링 엔진(Python Playwright)을 어디에 올려서 구동해야 할까요? 각 기술의 장단점과 비용, 실무 적합성을 정리했습니다.

---

## 1. AWS EC2 (Elastic Compute Cloud) + Docker
**가장 직관적이고 표준적인 "서버 한 대 통째로 빌리기" 전략**

클라우드 어딘가에 가상의 컴퓨터 한 대를 24시간 켜두는 방식입니다.
* **작동 원리:** EC2 인스턴스(서버)를 빌려 우분투 리눅스 등을 깐 다음, Playwright 전용 도커 환경을 세팅하여 24시간 내내 크롤링 파이썬 스크립트를 띄워둡니다.
* **👍 장점:** 
  * 로컬(내 맥북)과 환경이 99% 똑같아서 개발 및 디버깅이 매우 편함.
  * 복잡한 클라우드 아키텍처를 배우지 않아도 됨.
* **👎 단점:** 
  * 24시간 내내 켜두면 **서버비(고정 비용)**가 많이 나감. (최소 4GB RAM 이상, t3.medium 사용 시 월 2~3만 원)
  * AWS의 IP는 네이버 방화벽이 1순위로 싫어하는 "데이터센터 가짜 IP"이므로 **비싼 유료 프록시(Residential Proxy) 결제가 반강제**됨.

## 2. AWS Fargate (ECS - Elastic Container Service)
**서버 없이 "크롤링할 때만" 도커 박스 띄우기 (컴퓨팅 렌탈)**

24시간 서버를 켜두지 않고, 크롤링이 필요한 새벽 시간에만 '도커 컨테이너'를 쓱 띄워서 작업이 끝나면 버리는 고급 기술입니다.
* **작동 원리:** "새벽 4시에 1,000개 키워드 크롤링해와!" 라고 스케줄을 걸어두면, Fargate가 백그라운드에서 CPU/RAM을 임시로 할당받아 Playwright를 윙윙 돌리고, 7분 뒤 결과물을 DB에 저장하고 스스로 소멸합니다.
* **👍 장점:** 
  * 일한 시간(7분)만큼만 초 단위로 과금되므로 **유지비가 압도적으로 저렴**함 (월 몇천 원 수준).
  * 트래픽이 몰리면 컨테이너를 복제(Auto Scaling)하여 스레드를 100개, 500개 무한으로 확장하기 유용함.
* **👎 단점:** 
  * AWS ECS 인프라 세팅(Task Definition, VPC 로우타우팅 등)이 처음엔 꽤 복잡함.
  * 역시나 실행 IP가 AWS IP이므로 유료 레지덴셜 프록시 연동 필수.

## 3. AWS Lambda (서버리스 펑션)
**1초 단위로 과금되는 초경량 쪼개기 작업의 제왕**

최근 스타트업에서 가장 많이 쓰는 "함수 단위 서버리스" 기술입니다.
* **작동 원리:** 서버도 없고 컨테이너 박스도 없이, 오로지 이 `fetch_complete_list()` 함수 덩어리 하나만 클라우드에 띄워두고 요청이 올 때마다 0.1초 만에 구동합니다.
* **👍 장점:** 
  * 하루에 10만 번을 호출해도 비용이 캔커피 한 잔 값일 정도로 미친 **초저가 가성비** (100만 건 호출까지 무료 프리티어).
  * 1,000명이 동시에 요청하면 Lambda 함수 1,000개가 동시에 1초 만에 팍! 하고 켜지는 압도적 완전 병렬성 (네이버 트래픽 폭주 분산).
* **👎 단점 (스크래핑 한정 치명타):**
  * Playwright 같은 "진짜 브라우저 엔진(크롬 1~2GB)"은 덩치가 너무 커서 극단적인 다이어트 공사(Lambda Layer 최적화)를 거치지 않으면 **Lambda에 아예 올라가지도 않음**.
  * 1번 실행 시 **최대 15분까지만 동작** 가능. (크롤링이 15분을 넘기면 강제 종료당함).

## 4. AWS SQS + EC2 Auto Scaling (대규모 기업형)
**끊이지 않는 컨베이어 벨트 작업 (대기열 시스템)**

수만 개의 전국의 프랜차이즈 키워드를 돌려야 하는 B2B SaaS가 되었을 때 적용합니다.
* **작동 원리:** 스프링부트 메인 서버가 "조회해야 할 전국 키워드 목록 1만 개"를 SQS(메시지 큐)라는 대기열 상자에 던져둡니다. 그러면 일꾼(EC2 워커)들이 줄을 서서 큐를 하나씩 뽑아가며 크롤링을 수행합니다. 만약 일이 너무 많아 큐가 쌓이면, AWS가 빈 일꾼(서버)을 자동으로 더 고용합니다(Auto Scaling).
* **👍 장점:** 중간에 서버가 터지거나 크롤링이 실패해도, 큐에 키워드가 남아있기 때문에 다른 워커가 이어서 작업하여 안정성이 99.999%에 달합니다.

## 🚀 전문가 요약: 그래서 어떻게 가야 합니까?

초기 D-PLOG가 선택해야 할 가장 효율적이고 강력한 클라우드 성장 로드맵입니다.

### 🥉 [Level 1] 온프레미스 (미니 PC 24시간 풀가동) - **"현재 가장 강력히 추천"**
* **기술:** 그냥 사무실 남는 컴퓨터(윈도우/맥)를 크롤링 봇으로 24시간 켜둡니다.
* **이유:** 값비싼 클라우드 통신비와 '유료 프록시'를 결제할 필요 없이, 가장 완벽한 익명성을 보장하는 **사무실 백업 인터넷(KT/SKT)의 무료 공인 IP**를 그대로 쓸 수 있습니다! 초-중기 모델로 비용 대비 성능비 최상입니다.

### 🥈 [Level 2] AWS EC2 (또는 Lightsail) + 유료 레지덴셜 프록시
* **기술:** 24시간 가동되는 리눅스 서버 한 대와, 크롤링할 때 우회할 글로벌 가정망 프록시(Proxy) 이용권을 월 결제합니다.
* **이유:** 사무실 인터넷이 너무 느리거나 관리가 귀찮을 때, 무중단으로 스케줄링(Cron)을 돌리기 위한 타협점입니다.

### 🥇 [Level 3] Browserbase 등의 SaaS API (클라우드 헤드리스 브라우저)
* **기술:** 가장 최근에 실험한 유료 외부 서비스(Browserbase)입니다. Playwright 호스팅 인프라 구축 자체를 우리가 하지 않고 외주를 줍니다.
* **이유:** 우리가 골머리 앓을 `도커, AWS EC2, 메모리 부족, IP 밴 문제, 프록시 결제` 등을 월 얼마씩 내고 "알아서 안 막히게 해 줘!" 하고 던지는 것입니다. 개발 비용과 스트레스가 가장 적습니다.
