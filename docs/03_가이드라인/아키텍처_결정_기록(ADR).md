# 📚 Architecture Decision Records (ADR)

이 문서는 D-PLOG 프로젝트 개발 중 내려진 **중요한 기술적, 아키텍처적 의사결정의 이력과 그 '이유(Why)'**를 기록합니다. 과거의 결정 맥락을 보존함으로써, 새로운 인원(또는 AI 에이전트)이 이유 없이 예전 방식으로 롤백하거나 잘못된 구조를 제안하는 것을 막습니다.

---

## [ADR-001] Legacy NomadRank 구조 탈피 및 D-PLOG로의 마이그레이션 (투트랙화)

- **결정일**: 프로젝트 초기(Phase 0)
- **배경**: 기존 SpringBoot-React-NomadRank 모노리포 기반 레거시 프로젝트는 화면과 백엔드 로직이 무겁게 스파게티처럼 얽혀 있었음. 특히, 데이터 크롤링 및 NLP 기술이 포함된 SEO 분석 모듈을 Java(Spring)에서 돌리면서 서버 자원 소모가 심각했습니다.
- **결정**: 
  1. 클라이언트와 관리자뷰를 포괄하는 Next.js 앱으로 프론트를 완전 분리 및 재개발.
  2. 단순 비즈니스 로직은 Spring Boot(Main)가 담당.
  3. 무거운 AI 및 NLP 연산, 외부 파이썬 라이브러리 연동 로직은 FastAPI(AI-Worker)로 떼어내는 'Two-Track' 분산 서버 구조를 도입합니다.
- **결과**: 빠른 프론트엔드 모듈 개발 가능 및 머신러닝 처리 트래픽에 대한 병목 현상 해소 예상. 과거의 `views/` 디렉토리나 Jsp/Thymeleaf 의존은 완전히 금지됨.

---

## [ADR-002] 프론트엔드에 FSD (Feature-Sliced Design) 아키텍처 도입

- **결정일**: 프로젝트 레거시 전환(Phase 1) 시점
- **배경**: 여러 페이지에서 반복 사용되는 인증(Auth), 가게 정보(Store), 렌더링 로직의 재사용성을 높이고 싶었습니다. 기존에는 기능 중심인지, UI 요소 중심인지 구문하기 어려운 구조였습니다.
- **결정**: 글로벌 프론트엔드 트렌드인 FSD 방법론 적용. `shared` -> `entities` -> `features` -> `widgets` -> `pages` -> `app` 순서의 단방향 흐름 채택. 
- **결과**: `LoginViewModel`, `SignupForm` 등 도메인-친화적 폴더 배치가 고정되었으며, 프론트 코드 유지보수가 극적으로 향상되었습니다.

## [ADR-003] AI 에이전트 품질 제어를 위한 JSON 상태 연동 및 워크플로우 도입

- **결정일**: 2026-03-24
- **배경**: 개발 단계에서 AI 코딩 비서(커서, 클라인 등)를 활용하며, AI가 전체 로드맵 중 자신의 "현재 주소"를 잊어버리거나 다른 페이즈(Phase)의 구조를 미리 생성해 버리는 환각(Hallucination) 이슈 빈발.
- **결정**: 
  1. 에이전트 전용 글로벌 룰(`.agents/rules/agent-rules.md`) 도입 적용 완료.
  2. 에이전트가 단독으로 업데이트하여 읽을 수 있는 기계-친화적인 `roadmap_status.json` 관리.
  3. 로드맵 수정 시 반드시 `.agents/workflows/update-roadmap.md`를 사용토록 문서화.
- **결과**: 환각 예방, 로드맵 문서들(3개의 마크다운 + 1개의 JSON) 간의 정합성 유지 성공.
