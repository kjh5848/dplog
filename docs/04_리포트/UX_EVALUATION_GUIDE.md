# UX 및 기능 적정성 점검 가이드 (UX Evaluation & Feature Audit Guide)

본 문서는 현재 프로덕트(D-PLOG)의 기능적 과부하(Feature Bloat)나 부족함을 진단하고, UX 관점의 장단점과 개선 방향을 체계적으로 도출하기 위한 업계 표준 기법들을 정리한 가이드입니다.

---

## 1. 기능의 과도함/부족함 진단 (Feature Evaluation)

기능 다이어트와 우선순위 재조정을 위해 데이터와 유저 심리에 기반한 평가를 진행합니다.

### ① 카노 모델 (Kano Model)
유저가 특정 기능이 '있을 때'와 '없을 때' 어떻게 느끼는지를 설문 및 인터뷰하여 기능을 5가지로 분류하는 기법입니다.
- **당연적 기능 (Must-be):** 없으면 불만이지만, 존재한다고 크게 기뻐하지 않는 기능 (예: 검색 기능, 로그인). 부족하면 서비스 자체를 이탈하게 하는 핵심 기반입니다.
- **매력적 기능 (Attractive):** 없어도 불만은 없지만, 제공되면 유저를 열광시키는 기능 (예: AI 기반 리뷰 분석 요약). 프로덕트의 핵심 셀링(Selling) 포인트가 됩니다.
- **일원적 기능 (One-dimensional):** 제공되면 될수록 만족도가 비례하여 올라가는 기능 (예: 순위 업데이트 로딩 속도 향상).
- **무관심 기능 (Indifferent):** 있든 없든 유저가 신경 쓰지 않는 기능. **[과도한 기능 (Feature Bloat)]**이며 즉시 제거 검토 대상입니다.
- **역기능 (Reverse):** 오히려 존재해서 유저를 불편하고 복잡하게 만드는 기능. 최우선 제거 대상입니다.

### ② 기능 감사 (Feature Audit - Intercom 방식)
현재 서비스의 모든 기능을 리스트업한 뒤, **Adoption(얼마나 많은 비율의 유저가 사용하는가?)**과 **Frequency(얼마나 자주 사용하는가?)**를 기준으로 사분면(Quadrant) 모델에 분류합니다.
- 다수가 매일 쓰는 기능(Core)의 UX는 집요하게 개선합니다.
- 소수만 가끔 쓰는 기능은 UI에서 뎁스를 숨겨 메인 화면의 복잡도를 낮추거나 완전히 제거(Deprecate)합니다.

## 2. UX 장단점 및 사용성 점검 (Expert & Flow Evaluation)

내부 기획자/디자이너/개발자가 유저의 리소스 없이 자체적으로 병목을 찾아내는 점검 방법입니다.

### ③ 휴리스틱 평가 (Heuristic Evaluation)
UI/UX 디자인의 10가지 사용성 원칙(Jakob Nielsen)을 기준으로 인터페이스를 자체 점검합니다.
- **시인성 (Visibility of System Status):** 현재 무엇이 진행 중인지 명확히 알 수 있는가? (예: 모호한 로딩 화면 점검)
- **인지 부하 최소화 (Recognition rather than Recall):** 유저가 이전 화면의 정보를 기억해야만 다음 화면을 넘길 수 있는가?
- **자유와 제어권 (User Control and Freedom):** 실수로 키워드를 잘못 등록했을 때 1클릭으로 바로 취소하거나 삭제할 수 있는가?
- **일관성과 표준 (Consistency and Standards):** 대시보드의 특정 버튼의 컬러와 액션이 앱 전반에 걸쳐 동일하게 동작하는가?

### ④ 인지적 워크스루 (Cognitive Walkthrough)
신규 유저의 입장에 완벽히 빙의하여, 설정한 '특정 목표(예: 내 상점의 지역 순위를 확인한다)'를 달성하기 위해 밟는 단계들의 생각 흐름을 따라가는 기법입니다.
- **질문 1:** 사용자가 다음 단계에서 무엇을 해야 할지 직관적으로 아는가?
- **질문 2:** 화면의 버튼이나 텍스트 레이블이 사용자의 일상 언어와 일치하는가?
- **질문 3:** 행동을 취한 직후, 올바르게 행동했음을 피드백으로 알 수 있는가?

## 3. 사용자 관점의 정량화 (User Testing)

데이터와 내부 평가로 가설을 세웠다면, 실제 유저를 대상으로 수치화하는 단계입니다.

### ⑤ SUS (System Usability Scale - 시스템 사용성 척도)
유저에게 10개의 표준화된 5점 척도 문항을 제시하여, 사용성을 100점 만점으로 계량화합니다.
1. 나는 이 시스템을 자주 사용할 것 같다.
2. 시스템이 불필요하게 복잡하다고 느꼈다.
3. 시스템이 사용하기 쉽다고 느꼈다.
4. 이 시스템을 쓰려면 기술적 지원을 받아야 할 것 같다.
(이하 생략)
- **활용:** 업계 평균치(보통 68점) 대비 우리 시스템의 위치를 파악하고, UX 메이저 개편 전후의 점수 차이를 통해 성공 여부를 입증합니다.

## 💡 D-PLOG 적용 액션 플랜

현재 D-PLOG의 고도화를 위해 아래의 순서로 파일럿 점검을 수행하는 것을 권장합니다.

1. **Phase 1 (자체 점검):** 개발팀 내부에서 `휴리스틱 + 인지적 워크스루`를 통해 뎁스가 지나치게 깊은 화면, 혹은 모바일에서 가독성이 떨어지는 정보가 있는지 체크합니다.
2. **Phase 2 (기능 다이어트):** `카노 모델` 관점에서 유저에게 필수적이지 않은 지표(예: 의미 없는 저장 수 지표, 사용되지 않는 분석 탭 등)를 과감히 덜어내고, MUST-BE 기능인 '키워드 추적' 로직의 안정성과 직관성에 모든 UI 리소스를 투자합니다.
3. **Phase 3 (정기 평가):** 릴리즈 이후 얼리 어답터 사장님들을 대상으로 간이 `SUS 설문`을 진행하여 지속적으로 UX 사용성을 트래킹합니다.
