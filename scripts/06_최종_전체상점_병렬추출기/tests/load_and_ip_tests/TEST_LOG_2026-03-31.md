# 🧪 로컬 온프레미스 스케일업 및 부하 한계 테스트 일지 (Load & WAF Threshold Test)

## 📅 테스트 일시
**2026년 3월 31일 오후 1시 30분 ~ 1시 50분**

## 1. 램(RAM) 메모리 부하 테스트 (Memory Load Test)
### 📍 테스트 목적
마케터 다수 접속 상황을 가정하여, Playwright(Chromium 헤드리스)가 네이버 모바일 플레이스 딥 스크롤 시 운영체제의 물리적 메모리를 얼마나 소모하는지 벤치마킹. 클라우드 서버의 필수 사양(RAM Limit) 및 최대 동시 수용(Max Concurrency) 가능치를 도출하기 위함.

### 📜 사용된 스크립트
- `tests/load_and_ip_tests/memory_load_test.py`
- 파이썬 `psutil` 라이브러리를 통해 프로세스와 하위 Chromium 헬퍼 프로세스들의 RSS(Resident Set Size) 메모리의 총합 산출.

### 📊 테스트 결과 스냅샷
- **1개 작동:** 약 `667 MB` 소모
- **3개 병렬:** 약 `1,461 MB` 소모 (인스턴스당 ~487 MB)
- **5개 병렬:** 약 `2,375 MB` 소모 (인스턴스당 ~475 MB)
- **8개 병렬:** 약 `3,747 MB` 소모 (인스턴스당 ~468 MB)

### 💡 아키텍처 결론 (Memory)
- 모바일 가상렌더링 특성상 브라우저 1개당 평균 500MB의 거대한 램 메모리가 소멸됨.
- 클라우드 혹은 로컬 환경의 RAM 용량을 초과하지 않도록 **전역 글로벌 큐(Global Job Queue)를 도입하여 동시 실행 브라우저 개수에 반드시 락(Lock)**을 걸어두어야 함.

## 2. 네이버 안티 봇(WAF) 단일 IP 차단 임계 테스트 (IP Block Threshold Test)
### 📍 테스트 목적
가장 비용이 저렴한 "Mac Mini 온프레미스 (사내 단일 IP 구축 방식)" 환경에서 엔진을 가동했을 때, 어느 수준의 동시 접속(다중 크롤링 병렬 요청)부터 네이버의 캡차(CAPTCHA)나 IP 밴(차단) 방어벽에 걸리는지를 모의 해킹(Penetration) 테스트 함.

### 📜 사용된 스크립트
- `tests/load_and_ip_tests/ip_block_test.py`
- 모바일 Safari User-Agent로 위장. 5개 ➞ 10개 ➞ 15개 ➞ 20개 순차 스파이크 발송 후 HTTP Status/DOM 캡차 텍스트 검출.

### 📊 테스트 결과 스냅샷 (단일 로컬 IP 기준)
| 동시 요청 개수 | 통과/성공 (Success) | 캡차 및 차단 (Ban) | 비고 |
| :---: | :---: | :---: | :--- |
| **5 개** | 5 | 0 | 순식간에 통과 (4.9s) |
| **10 개** | 10 | 0 | 여유롭게 통과 (10.1s) |
| **15 개** | 15 | 0 | 어뷰징 회피 완벽 (14.8s) |
| **20 개** | 19 | 1 | 캡차 없음! (1건은 순수 로딩 타임아웃) |

### 💡 아키텍처 및 비즈니스 결론 (Throughput & Hosting)
- **경이로운 캡차 생존력:** 하나의 공유기(사내 IP)에서 20개의 딥 렌더링 검색을 스파이크 형태로 난사해도 네이버 매크로 차단 로직이 전혀 막지 못함. (캐시 무력호를 위한 더미 쿼리 부여 및 모바일 View 속성 위장이 완벽하게 작동).
- **무임승차(Zero Cloud Cost) 온프레미스 가능 확정:** 비싸게 AWS 서버리스를 쓸 이유가 상실됨. 사무실에 16GB 메모리가 장착된 M1 맥 미니(Mac Mini) 중고를 두고, 앞서 확인된 램 최적 한계점인 **'동시 15개 브라우저 제한(Max Limit 15)'**만 큐(Queue)에 세팅.
- 이 세팅만으로 마케터 10~20명이 하루 종일 황금 키워드를 발굴해도 로컬 IP 차단 없이 영구적으로 0원 호스팅이 가능함.
