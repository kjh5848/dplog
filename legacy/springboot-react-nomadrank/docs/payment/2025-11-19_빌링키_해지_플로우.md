# PortOne 빌링키 구독 해지 백엔드 계획

## 목적
- PortOne V2 빌링키를 사용하는 구독 해지 요청을 안전하게 처리하기 위한 백엔드 흐름을 정리한다.
- 예약 결제, 다중 상점, 예외 응답 등 운영 리스크를 사전에 제거하고 서비스/테스트/운영 체크리스트를 일관되게 유지한다.

## 연관 컴포넌트
- `src/app/api/subscriptions/[subscriptionId]/cancel/route.ts`: Next.js App Route에서 해지 요청을 수신.
- `DELETE /api/subscriptions/{subscriptionId}/cancel`: 프런트엔드에서 호출하는 App Route HTTP 계약.
- `src/lib/services/portoneService.ts`: 서버 전용 서비스, PortOne 클라이언트 호출 및 응답 해석.
- `kr.co.nomadlab.nomadrank.domain.payment.service.PaymentServiceApiV1`: 해지 플로우의 단일 진입점.
- `kr.co.nomadlab.nomadrank.domain.payment.client.PortOneV2Client`: PortOne REST API 호출 담당.
- `kr.co.nomadlab.nomadrank.domain.payment.util.PaymentUtils`: 카드/빌링키 마스킹 등 공통 유틸.

## App Route 요청/응답 계약
- HTTP 메서드/URI: `DELETE /api/subscriptions/{subscriptionId}/cancel`
- Request Body(JSON): `{ "reason": string | undefined }`
- Response Body(JSON): `{ "status": "canceled", "canceledAt": ISO8601 string }`
- 상태 매핑
  - 200: 해지 성공, 내부 DB 갱신 완료
  - 404: 구독/빌링키 미존재 또는 이미 해지된 상태 → UI에서는 이미 처리된 해지로 안내
  - 409: 예약 결제 미삭제 또는 중복 해지 충돌 → 프런트 문구 “예약 결제 삭제 후 다시 시도” 노출
  - 5xx: PortOne 장애 혹은 내부 오류 → 재시도/문의 안내 및 Sentry 경고

## 프로세스 상세

### 1. 접근·컨텍스트 수집
- 구독자가 사용하는 상점(storeId)과 채널(channelId)을 식별한다. 다중 상점 배포 시 상점 간 빌링키 이동을 방지하기 위함이다.
- 해지 대상 `billingKey`와 `customerId`를 `PaymentServiceApiV1`에서 조회하여 PortOne과 내부 DB의 일관성을 확인한다.
- App Route는 server action으로만 동작하도록 하고, `route.ts`에서 `PortOneService`를 호출해 백엔드 계층으로 위임한다. 요청 본문의 `reason`은 PG 요구 사항이 있을 때만 전달한다.

### 2. 예약 결제 확인
- PortOne `GET /v2/payment-schedules/{scheduleId}` API로 저장된 scheduleId(=내부 `subscriptionId`) 기준 예약 존재 여부를 확인한다.
- 스케줄이 존재하면 동일 ID로 `DELETE /v2/payment-schedules/{scheduleId}` 요청을 실행한다. 삭제 실패 시 재시도 전략과 실패 로그를 남긴다.
- 예약 건을 선 제거하지 않으면 `PaymentScheduleAlreadyExistsError`가 발생하고, 이미 제거된 경우 `BillingKeyAlreadyDeletedError`를 방지할 수 있다.

### 3. 빌링키 검증
- `GET /v2/billing-keys/{billingKey}` 또는 `GET /v2/billing-keys?customer.id={customerId}`로 상태와 소유 상점을 재확인한다.
- 응답의 `storeId`가 현재 상점과 일치하지 않으면 즉시 중단하고 운영 알림을 전송한다.
- 이미 해지된 상태(`status=inactive`)면 중복 요청으로 간주하고 UI에는 성공 메시지를 반환한다.

### 4. PortOne 해지 요청
- `DELETE /v2/billing-keys/{billingKey}` API를 호출한다.
- 다중 상점 환경이면 `storeId`를 명시하고, PG에서 사유(reason)를 요구하는 경우(예: 네이버페이) 쿼리 파라미터에 전달한다.
- 인증은 PortOne JWT 또는 API Secret을 사용하며, 실패 시 자격 증명 누락/만료를 Sentry에 경고로 남긴다.

### 5. 응답 처리 및 내부 상태 갱신
- 성공 응답 시 내부 DB의 빌링키 상태를 `deleted`(또는 도메인 규약에 맞는 상태)로 업데이트한다.
- 중복 해지, 미존재 등 4xx 응답은 사용자 가이드를 포함한 메시지로 변환하고, 5xx 응답은 재시도 로직을 적용한다.
- 모든 예외는 PII가 제거된 형태로 Sentry에 기록한다. billingKey, cardNumber 등은 `PaymentUtils`를 이용해 마스킹한다.

### 6. 사용자 통지 및 UI 반영
- 빌링키 해지 완료 후 이메일·푸시 등 고객 통지 수단을 호출한다.
- 프론트엔드 `src/store/subscriptionStore.ts`(또는 대응 Zustand 스토어)와 `CancelModal` 컴포넌트가 기대하는 응답 스키마(`status`, `canceledAt`)에 맞춰 데이터를 반환한다.
- 실패 시 고객센터 전달용 표준 문구를 준비하고, 재시도 혹은 수동 처리 루틴을 운영 문서에 링크한다.

## 프런트엔드 연계 포인트
- `CancelButton` → `CancelModal` → `useCancelSubscription` 흐름은 `DELETE /api/subscriptions/{subscriptionId}/cancel`만 호출하도록 유지한다.
- Optimistic update는 백엔드 응답을 받은 뒤 적용하고, 예약 결제 삭제 실패 시에는 즉시 에러 상태로 전환해 재시도 버튼을 노출한다.
- PG별 안내 문구(`네이버페이 해지 사유 필수` 등)는 백엔드에서 reason 필요 여부를 응답 헤더 혹은 바디 플래그(`requiresReason: true`)로 내려줄 수 있도록 추후 확장성을 고려한다.

## 테스트 전략
- `tests/.../subscription-cancel.spec.ts` Playwright 시나리오에 다음 케이스를 추가한다:
  - 예약 결제가 존재하는 구독 해지 → 예약 삭제 → 빌링키 해지 성공.
  - 예약 결제가 없는 구독 해지 → 빌링키 해지만 수행.
  - PortOne 5xx 발생 시 UI에서 재시도/문의 안내 노출.
- PortOne 호출은 테스트 더블(예: MSW, Mock Service Worker) 또는 API mocking layer로 대체하고, 실패 응답을 강제 주입해 방어 로직을 검증한다.
- 백엔드 단위/통합 테스트에서는 `PortOneV2Client`를 `@MockBean`으로 대체하고, 트랜잭션 롤백 여부를 확인한다.

## 운영 체크리스트
- `npm run lint` 및 `./gradlew test` 실행 결과를 PR 템플릿에 기록한다.
- PR 본문에 PortOne 호출 로그(성공/실패)와 재현 절차를 포함한다.
- PG별 추가 후속 절차(예: 카드사 자동이체 해지 보고)가 필요한 경우 재무팀과 공유한다.
- Sentry·로그 수집기에서 `billingKey` 삭제 이벤트가 정상적으로 수집되는지 확인한다.
- 다중 상점 환경이면 상점-채널 매핑 테이블을 재확인하고, 신규 상점 추가 시 문서를 갱신한다.

## PortOne API 요약

| 단계 | 메서드 & 경로 | 필수 파라미터 | 비고 |
| --- | --- | --- | --- |
| 예약 조회 | `GET /v2/payment-schedules/{scheduleId}` | Path `scheduleId` | 내부 `subscriptionId`와 동일한 scheduleId 사용 |
| 예약 삭제 | `DELETE /v2/payment-schedules/{scheduleId}` | Path `scheduleId` | 예약이 있을 때만 실행 |
| 빌링키 조회 | `GET /v2/billing-keys/{billingKey}` | Path `billingKey` | 다중 상점 검증 |
| 빌링키 해지 | `DELETE /v2/billing-keys/{billingKey}` | Path `billingKey`, 필요 시 `storeId`, `reason` | 해지 본동작 |

- 모든 요청은 PortOne JWT 또는 API Secret 기반 인증이 필요하다.
- 응답 코드 401/403 → 자격 증명 오류, 404 → 빌링키 미존재, 409 → 예약 결제 존재 또는 중복 해지, 502 이상 → PortOne 장애.

## 참고 및 후속 조치
- 서비스 계층은 반드시 `PaymentServiceApiV1` 단일 진입점을 사용하여 다른 서비스 구현체에 직접 접근하지 않는다.
- 외부 연동은 `PortOneV2Client`에서만 수행하고, 서비스 계층은 클라이언트 응답을 해석/변환하는 데 집중한다.
- 운영 배포 전 `application-prod.yml`의 PortOne 자격 증명과 상점 매핑을 재검토하고, 필요 시 피처 토글을 통해 단계적 출시를 고려한다.
