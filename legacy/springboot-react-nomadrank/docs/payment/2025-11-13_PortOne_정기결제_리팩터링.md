# 2025.11.13 PortOne 정기 결제 리팩터링 계획 (V2 결제 예약 API 대응)

## 1. 배경 및 이슈 정리
- 현재 `RecurringPaymentScheduleService` (`src/main/java/.../RecurringPaymentScheduleService.java:35-155`)는 V1 시절의 `billing-reservations` 엔드포인트를 호출하도록 구현돼 있습니다.
- PortOne V2 OpenAPI (MCP `readPortoneOpenapiSchema`, `version=V2`)에 따르면 정기 결제 예약 기능은 `POST /payments/{paymentId}/schedule` 로만 제공되며, Request Body는 `CreatePaymentScheduleBody` (`payment` + `timeToPay`) 구조입니다. `interval`, `intervalCount` 개념이 존재하지 않습니다.
- 그 결과, 현재 구현은 항상 404를 반환하며 첫 결제 이후 예약 생성 단계에서 실패 → 롤백/취소 루틴이 실행됩니다.

## 2. 리팩터링 목표
1. PortOne V2 결제 예약 API (`/payments/{paymentId}/schedule`)를 사용하도록 DTO와 클라이언트 호출을 전면 교체한다.
2. 예약/업그레이드/다운그레이드/해지 로직이 새 API에 맞춰 정상 동작하도록 서비스 계층을 정비한다.
3. 내부 스케줄 전략(선 예약 vs. 주기적 예약)을 명확히 정의해 운영/구독 정책이 일관되게 유지되도록 한다.

## 3. 상세 작업 항목

### 3.1 DTO 및 클라이언트
- `BillingReservationRequest`를 `PaymentScheduleCreateRequest`로 교체.
  - 필드: `paymentId`, `timeToPay`, `amount`, `currency`, `orderName`, `customer`, `billingKey`, `channelKey`, `storeId`, `productCount`.
  - `interval`, `intervalCount`, `scheduleAt` 필드는 제거.
- `BillingReservationResponse`는 `PaymentScheduleCreateResponse`로 교체하고, PortOne 응답(`scheduleId`, `timeToPay`, `status`, `paymentId`)을 그대로 담는다.
- `PortOneV2Client#createBillingReservation` → `createPaymentSchedule`로 이름 변경하고 URL을 `"{base}/payments/{paymentId}/schedule"`로 수정.
  - Request Body는 `{"payment": {...}, "timeToPay": "ISO-8601"}` 형태로 직렬화.
  - 포트원 응답에서 400/404/409 시 `PortOneApiException`을 던져 서비스 계층이 구체적인 원인을 판단할 수 있게 한다.
  - `getPaymentSchedule`/`deletePaymentSchedule`도 동일한 API 네이밍(결제 예약)으로 로그 메시지를 업데이트.

### 3.2 서비스 계층 (`RecurringPaymentScheduleService`)
- 입력 파라미터:
  - `paymentId`: PortOne 사전등록 시 생성한 결제 ID (기존 `orderId`와 별개).
  - `executionTime`: 예약 실행 시각 (`OffsetDateTime`).
- 내부에서 `payment` 객체를 구성하고 `PortOneV2Client#createPaymentSchedule` 호출.
- 기존 `interval/intervalCount` 계산, `generateReservationOrderId`, `resolveIntervalFromCycle`는 더 이상 PortOne API에 전달되지 않음.
  - 다만 비즈니스 정책(구독 정보 조회, 다음 결제일 계산)에는 여전히 필요하므로, `RecurringPaymentScheduleService` 외부로 옮기거나 “예약 생성 시 참고용” 데이터로 사용.
- 응답으로 받은 `scheduleId`를 `SubscriptionEntity` 측에 저장해야 이후 업그레이드/다운그레이드/해지 시 포트원 예약을 정확히 취소/변경할 수 있음.

### 3.3 업그레이드 · 다운그레이드 로직 정비
- 기존 로직(파일 경로는 별도, 예: `PaymentServiceApiV1` 업그레이드/다운그레이드 분기)에서 `RecurringPaymentScheduleService.deleteSchedules`를 호출해 모든 예약을 날린 뒤 신규 예약을 생성하는 방식이었다면,
  - 새 API에서는 `scheduleId`를 알고 있으므로 `DELETE /payment-schedules/{scheduleId}` + `billingKey` 파라미터로 단일 예약만 제거.
- 등급 변경 시 “다음 결제 예정 시각”을 재계산하고, 새로운 `paymentId` + `timeToPay` 조합으로 예약을 다시 건다.
- 업그레이드와 다운그레이드 모두 **기존 예약 → 취소** → **새 결제 생성/사전등록 → 새 스케줄 등록** 순서를 동일하게 맞춰 예외 시 롤백 전략을 단순화.

### 3.4 스케줄 전략 (JIT 단건 예약으로 통일)
- **월간 멤버십**: 결제 성공 직후 PortOne 예약을 한 건 생성하고(`timeToPay = 다음 달 청구일`), 다음 결제가 성공하면 즉시 그 다음달 예약을 생성한다. 항상 “다음 결제일 한 건”만 PortOne에 유지한다.
- **연간 멤버십**: 최초 가입 시 다음 1건만 예약(`timeToPay = 다음 연도 결제일`). 만료 30일 전에 내부 스케줄러가 실행돼 새로운 `paymentId`로 예약을 다시 생성한다. 선(先) 다중 예약 전략은 사용하지 않는다.
- **예약 보강 로직**: 매일 스케줄러가 “다음 결제일까지 D일 이내인데 예약이 없는 사용자”를 조회해 단건 예약을 다시 만들어 JIT 전략이 깨지지 않게 한다.

### 3.5 스케줄 계산 & 스케줄러
- 이미 `PaymentServiceApiV1`에서 `resolveBillingCycleFromMembership`, `resolveNextBillingDate`를 통해 다음 청구일을 계산하고 `RecurringPaymentScheduleService.toOffsetDateTime`으로 변환하고 있음 (`src/main/java/.../PaymentServiceApiV1.java:448-505` 부근).
- 결제 성공 시:
  1. PortOne 사전등록을 새 `paymentId`로 수행.
  2. `timeToPay` = `nextBillingDate` at 00:00(+09:00) (또는 멤버십 종료 직후).
  3. `createPaymentSchedule` 호출.
- 예약 실패 시:
  - `PortOneApiException` status에 따라 사용자 메시지와 운영 알람을 구분 (`404` → 빌링키 만료/삭제, `409` → 중복 예약 등).
- Scheduler/cron:
  - 매일 새벽에 “예약 누락 사용자” 목록을 조회(조건: 활성 구독 + 다음 결제일 - 오늘 <= N일 & 예약 미존재) 후 JIT 예약 보강.

## 4. 매월 1건 예약 운영 가이드

### 4.1 전체 흐름
1. BillingKey 발급 (`PaymentControllerApiV1.completeBillingKey`)  
2. 첫 결제 완료 (`PortOneV2Client.payWithBillingKey`)  
3. PortOne Webhook 확정 (`PaymentWebhookService`, 추후 구현)  
4. Subscription 활성화 (`PaymentServiceApiV1.activateUserMembershipAfterPayment`)  
5. “다음 결제 1건 예약” 생성 (`RecurringPaymentScheduleService.createReservation`)  
6. 한 달 후 PortOne 예약 결제 자동 실행  
7. Webhook으로 성공 이벤트 수신  
8. 다음 결제 예약 생성 … 반복

### 4.2 예약 생성 전 필수 검증
예약은 `RecurringPaymentScheduleService` 또는 예약 스케줄러가 수행한다. 아래 항목을 단계별로 확인한다.

#### 4.2.1 BillingKey 상태
- API: `GET /billing-keys/{billingKey}` (`PortOneV2Client.getBillingKeyInfo`)
- 체크: 존재 여부, 상태(ACTIVE), 채널키 일치, 사용자/고객 식별자 일치.
- 실패 시: 사용자에게 카드 재등록 안내, `billing.max-retry-count` 한도 내 재시도 큐에 적재.

#### 4.2.2 Subscription 상태
- `SubscriptionEntity`가 `ACTIVE`인지, `endDate`가 예약일 이후인지, 업/다운그레이드 처리 대기 상태가 아닌지 확인.
- 실패 시 예약 보류, 운영자 알림 또는 사용자 UI에 수정 요청.

#### 4.2.3 PaymentHistory 점검
- 직전 사이클 결제가 `PAID`인지 확인 (`PaymentHistoryRepository`).  
- `FAILED`면 재시도/카드 변경 요구, `CANCELLED`면 원인 분석 후 수동 개입.
- 실패 시 `PaymentHistory.errorMessage` 갱신 + 사용자 알림.

#### 4.2.4 중복 예약 여부
- API: `GET /payments/{paymentId}/schedules` 또는 DB의 `reservationId` 존재 여부로 확인.
- PortOne 오류코드 `PAYMENT_SCHEDULE_ALREADY_EXISTS`를 사전에 방지.

#### 4.2.5 다음 결제일 계산
- `previousPaidAt + interval(1 month)`을 기준으로 계산하되, 타임존(KST) 및 말일/윤년 보정(`java.time` 사용) 확인.
- `RecurringPaymentScheduleService.toOffsetDateTime`는 시스템 기본 TZ를 사용하므로 UTC↔KST mismatch가 없는지 검증.

#### 4.2.6 업/다운그레이드 시 기존 예약 삭제
- 구 플랜 예약 삭제: `DELETE /payments/{paymentId}/schedules/{scheduleId}` (`PortOneV2Client.deletePaymentSchedule`).
- 누락 시 중복 결제 리스크 발생 → 필수 선행 단계.

### 4.3 결제 실행 후(Webhook) 검증
Webhook(`billingKey.payment.scheduled.*`) 수신 시:
1. HMAC-SHA256 서명/타임스탬프 검증 (`PortOneV2Config.v2.webhookSecret` 활용).
2. 결제 상태에 따른 분기:
   - `PAID`: `SubscriptionEntity` 만료일 갱신 + 다음 예약 즉시 생성.
   - `FAILED`: 사용자 카드 재등록 요청, `retryCount` 증가, 예약 재시도 job 등록.
   - `CANCELLED`: 예약 재생성 중단 후 운영자 확인.
3. 결제 성공 후 업/다운그레이드 플래그 확인 → 필요 시 새 요금제 가격으로 재예약.

### 4.4 업그레이드/다운그레이드 체크리스트
1. 기존 예약 취소 (`PortOneV2Client.deletePaymentSchedule`).
2. 새 Plan 금액/주기를 기준으로 사전등록 & 예약 재생성.
3. BillingKey 변경이 필요한 경우 `BillingKeyEntity` 교체 후 단계 4.2.1~4.2.5 재수행.

### 4.5 실패 및 복구 전략
- **404 BillingKeyNotFoundError**: 발급 직후 전파 지연/오타 가능성 → 1~3초 sleep 후 최대 3회 재시도 → 실패 시 BillingKey 재검증 및 사용자 재등록 유도.
- **409 PaymentScheduleAlreadyExists**: 기존 예약 조회 후 `reservationId`를 DB에 동기화하고 중복 생성 중단.
- **결제 실패(한도/만료/PG)**: 하루 후 1회 재시도 → 재실패 시 카드 재등록 요청, 구독 grace period 3일 유지.
- **Webhook 누락**: 5분 주기로 `GET /payments/{paymentId}` polling, 상태 mismatch 시 강제 동기화.

### 4.6 예약 생성 스케줄러
백그라운드 Job(예: Spring Scheduler/Quartz)이 수행:
1. BillingKey 유효성 재검증.
2. Subscription 상태 확인.
3. 다음 사이클 예약 생성(JIT).
4. 실패한 예약 재시도 및 `retryCount` 관리.
5. 업/다운그레이드 후 예약 재생성.
6. Webhook 누락 결제 보정.

### 4.10 트랜잭션 & 예외 처리 원칙
- **트랜잭션 경계**  
  - `PaymentServiceApiV1.completeBillingKey` 및 `SubscriptionServiceApiV1`의 업/다운그레이드 메서드는 `@Transactional`로 감싸고, PortOne API 호출 전후로 DB 상태를 선 검증한다.  
  - PortOne 호출(사전등록, 예약 생성/삭제)은 외부 I/O이므로 가급적 *트랜잭션 안에서 실행 후 즉시 커밋*하거나, 필요 시 `REQUIRES_NEW`/`NOT_SUPPORTED` 전파를 고려해 롤백 시에도 PortOne 상태와 DB 상태가 불일치하지 않게 한다.
- **검증 실패 시점**  
  - BillingKey/Subscription/PaymentHistory 검증 중 하나라도 실패하면 PortOne API를 호출하지 않고 즉시 `BadRequestException`을 던져 상위 컨트롤러가 사용자 메시지를 반환하게 한다.  
  - 업/다운그레이드에서 `rebuildRecurringReservation(...)` 실행 전 반드시 기존 예약 삭제와 상태 전환이 완료됐는지 확인한다.
- **예외 매핑**  
  - `PortOneApiException`은 상태코드·code·message를 그대로 보존해 로깅하고, 사용자 메시지는 `BadRequestException`으로 감싸 전달한다.  
  - `PortOne` 5xx/네트워크 오류는 `RuntimeException`으로 전파하되, `PaymentServiceApiV1`에서 감싸서 보상(결제 취소 등)을 수행한다.
- **보상 로직**  
  - 예약 생성 실패 시 첫 결제 취소(`PortOneV2Client.cancelPayment`)를 실행하는 현재 구조를 유지하되, 취소도 실패하면 `PaymentHistory`에 상태를 기록하고 운영자 알람을 트리거한다.  
  - 업/다운그레이드 도중 오류 발생 시 새 예약/사전등록이 되돌려졌는지 확인하고, 미정리된 예약은 스케줄러가 발견해 삭제하도록 `reservationStatus`를 업데이트한다.
- **재시도/낙관적 잠금**  
  - `subscriptionRepository` 기반 업데이트는 사용자 단위로 직렬화(예: `SELECT ... FOR UPDATE` 또는 버전 필드)하여 동시 업그레이드 요청 시 중복 예약이 생성되지 않도록 한다.  
  - PortOne API 재시도는 `billing.max-retry-count` 설정을 공유해 전역적으로 관리한다.

## 5. 리스크 및 대응
- **예약 생성/삭제 시점 race condition**: 업그레이드/다운그레이드 중 복수 스레드가 동일한 `billingKey`의 예약을 삭제하지 않도록 낙관적 락 또는 DB 상태 체크 필요.
- **예약 실패 재시도**: PortOne API 429/5xx 시 exponential backoff 도입(기존 `billing.max-retry-count`, `retry-interval-hours` 설정 재활용).
- **데이터 정합성**: `SubscriptionEntity`에 `portOneScheduleId`, `nextScheduleTime` 등을 저장해 실시간 상태 확인 및 재예약 여부 판별.

이 계획에 따라 구현하면 정기 결제 예약이 PortOne V2 사양에 맞게 동작하고, 업/다운그레이드 로직에서도 동일한 예약 관리 체계를 공유할 수 있습니다.

## 7. 구현 현황 & 주의사항

### 7.1 반영된 변경 사항
- DTO 교체: `BillingReservationRequest/Response` → `PaymentScheduleCreateRequest/Response`로 전환하여 PortOne V2 `/payments/{paymentId}/schedule` 스펙과 1:1 매핑.
- 클라이언트: `PortOneV2Client#createPaymentSchedule`이 nested `payment` payload를 구성하고, `scheduleId/timeToPay`를 반환하도록 구현.
- 서비스: `RecurringPaymentScheduleService`, `PaymentServiceApiV1`, `SubscriptionServiceApiV1` 모두 새 DTO를 사용하여 단건 예약을 생성/갱신.
- 문서: 본 문서를 포함한 API 문서(`docs/api/portone-subscription-api.md`)에서 새 메서드명을 안내.
- 스케줄러 인프라: `PaymentSchedulerProperties`, `PaymentIdGenerator`, `RecurringReservationJobService`, `RecurringReservationScheduler`를 추가해 JIT 예약 보강/복구 작업을 자동화했고, `application-dev.yml`에서 검사 구간과 재시도 파라미터를 설정 가능하게 했다.

### 7.2 주의사항
- 테스트 실행 시 Gradle wrapper lock 파일 접근 권한 오류가 발생해(`gradle-8.7-bin.zip.lck`) 로컬 환경에서 `./gradlew test`를 재실행해야 함. CI/운영 전 필수로 확인.
- 현재 구현은 예약 “생성” 단계까지만 완료했으며, JIT 전략을 안정화하려면 예약 누락 보강 스케줄러/재시도 큐/Webhook 연계가 필요하다.
- 업/다운그레이드 시 기존 `subscriptionId`를 다음 예약의 `paymentId`로 재사용하고 있으므로, PortOne 측에서 동일 ID를 허용하는지(또는 별도 `paymentId` 규칙이 필요한지) 운영 환경에서 재확인 바람.
- `SubscriptionReservationResponse`는 여전히 interval 정보를 포함하지만 PortOne API에서는 사용하지 않으므로, 프런트/백엔드 간 계약을 다시 한 번 맞추어 불일치가 없도록 한다.
- `resolveIntervalFromCycle`, `generateReservationOrderId` 등 도우미는 아직 남아 있으나 더 이상 PortOne API에 전달되지 않으므로, 추후 내부 비즈니스 로직에서만 사용하거나 정리한다.
- **첫 결제용 paymentId 재사용 금지**: PortOne 예약 API는 “다음 결제용” 결제 ID를 요구하므로, `PaymentIdGenerator`를 통해 새 paymentId를 생성해 예약에 사용하고 `SubscriptionEntity`/스케줄러에서 공유해야 한다.
- **멤버십 상태 전환 주의**: `MembershipUserEntity`의 상태 흐름은 `ACTIVATE → READY → EXPIRED`이며 “FREE”는 별도 상태가 아니다. Activation 이후 예외가 발생하면 READY로 되돌리는 보상 로직이 아직 없어, 실패 위치에 따라 ACTIVATE 상태가 남을 수 있다 (7.4 참조).

### 7.3 추가 조언
1. **스케줄러 구현**: 매일/매시간 예약을 검증하는 Spring Scheduler 또는 Quartz Job을 추가해, 누락 사용자를 자동으로 재예약하고 `retryCount`를 관리하십시오.
2. **Webhook 우선 전략**: PortOne Webhook을 신뢰 경로로 두고, 예약 결제 성공 시점에만 다음 예약을 생성하도록 이벤트 기반 플로우를 구성하면 동시성 이슈가 크게 줄어듭니다.
3. **관측지표**: 예약 생성/삭제/실패 건수를 Prometheus 등으로 수집해 SLA를 모니터링하면, PortOne API 장애나 설정 오류를 빠르게 감지할 수 있습니다.
4. **통합 테스트**: V2 결제 예약 API를 사용하는 통합 테스트(예: Mockito + WireMock)로 happy-path/에러-path를 검증하여 회귀를 방지하세요.
5. **스케줄러 모니터링**: `RecurringReservationScheduler`의 03:00/매시 Job 결과(성공/실패 건수, PortOne 응답 코드)를 로그/모니터링 시스템에 연동해 SLA 이상 징후를 빠르게 파악해야 한다.
## 8. 예약 스케줄러 서비스 설계 가이드

### 8.1 목표
- PortOne JIT 전략(항상 “다음 결제 1건”만 예약)을 강제하기 위해, 예약 누락 사용자 탐지, 재시도 큐 처리, webhook 누락 보정을 자동화한다.
- 업/다운그레이드 후 설정되지 않은 예약이나 취소되지 않은 이전 예약을 일괄 정리한다.

### 8.2 구성 컴포넌트
| 컴포넌트 | 역할 |
| --- | --- |
| `RecurringScheduleScheduler` (Spring `@Scheduled`) | 주기적으로 메인 워크플로우를 실행 |
| `ReservationInspector` | DB에서 활성 구독과 예약 상태를 비교해 누락/이상 데이터를 추출 |
| `ReservationJobService` | PortOne API 호출(생성/삭제), 재시도 정책 수행, 결과를 DB에 반영 |
| `ReservationRetryQueue` (DB 테이블 또는 메시지 큐) | 실패 항목을 enqueue하고 백오프 전략대로 dequeue |
| `ReservationAuditLogger` | 결과를 Kibana/Prometheus로 전송, 관리자 알림 |

### 8.3 실행 주기
- **일일 실행(예: 03:00)**: 전체 활성 구독을 스캔해 다음 결제일이 `D`일(예: 2~5일) 남은 사용자 중 예약이 없는 케이스를 찾아 생성.
- **시간 단위 실행(예: 매시 10분)**: 실패 큐를 비우고 재시도, webhook 누락 여부 확인, 긴급 경보를 발송.
- **수동 트리거**: 운영자가 특정 사용자 또는 Subscription ID를 넣어 즉시 예약 재생성을 실행할 수 있는 Endpoint.

### 8.4 프로세스 단계
1. **데이터 로드**
   - `SubscriptionRepository`에서 `status = ACTIVE`이면서 `nextBillingDate`가 `기준일 ± N일`에 해당하는 구독을 조회.
   - `PaymentScheduleInfo`와 비교해 PortOne 상 예약이 존재하는지 확인.
2. **검증/분류**
   - 예약 없음 → “생성 대상” 리스트
   - 예약 존재하지만 `timeToPay`가 실제 `nextBillingDate`와 상이 → “갱신 대상”
   - `retryCount`가 임계치 초과 → “사용자 알림 대상”
3. **예약 생성**
   - 각 대상에 대해 `paymentIdGenerator.generate(subscription.getUserId(), subscription.getNextBillingDate())` 호출 → 새 paymentId 확보.
   - `RecurringPaymentScheduleService.createReservation()` 호출 (BillingKey·customer 정보 동시 검증).
   - 성공 시 `Subscription`에 `nextPaymentId`, `reservationId`, `reservationStatus`, `lastVerifiedAt` 갱신.
4. **결과 반영**
   - 성공 로그와 실패 로그를 분리해 저장(`ReservationAudit` 테이블 or ElasticSearch).
   - 실패 항목은 `ReservationRetryQueue`에 push(지수 백오프).
   - 임계치 초과/빌링키 만료 오류는 `NotificationService`를 통해 슬랙/메일 발송 + 사용자를 향한 푸시/이메일.
5. **Webhook 누락 보정**
   - 최근 `N`시간 내 예약 실행 예정이었지만 webhook 이벤트가 없는 paymentId 목록을 조회.
   - PortOne `/payments/{paymentId}` 호출로 실제 상태를 동기화하고, 필요 시 다음 예약을 재생성.

### 8.5 데이터 모델 확장
추천 필드(기존 Subscription + 신규 테이블):
- `subscription.nextPaymentId`: 다음 예약에 사용할 paymentId.
- `subscription.reservationStatus`: `SCHEDULED`, `FAILED`, `RETRYING` 등.
- `reservation_retry` 테이블: `subscription_id`, `payment_id`, `failure_reason`, `retry_count`, `next_retry_at`.
- `reservation_audit` 테이블: 예약 생성/삭제 결과, PortOne 응답 전문, 처리자 정보.

### 8.6 예외 및 재시도 정책
| 오류 유형 | 대응 |
| --- | --- |
| 4xx (BillingKeyNotFound, AlreadyExists) | 즉시 사용자 알림 + 예약 실패 기록 |
| 5xx/네트워크 | `retryCount++`, 다음 스케줄 실행 또는 `next_retry_at`에 따라 재시도 |
| Webhook 미수신 | 5분 간격 polling → 상태 동기화 후 다음 예약 생성 |

### 8.7 구현 로드맵
1. `ReservationJobService` 및 `ReservationRetryQueue` 인터페이스 작성.
2. `RecurringScheduleScheduler`에 일일/시간별 `@Scheduled` 메서드 구현.
3. `paymentIdGenerator`(예: `payment-{subscriptionId}-{yyyyMMddHHmmss}`) 서비스화.
4. 테스트: 예약 누락 시나리오, PortOne mock 서버, 재시도 큐, 업/다운그레이드 후 예약 재구성.
5. 모니터링 대시보드 추가(Grafana/Slack 알림).

### 8.8 설정 변경 가이드 (운영/개발자가 원하는 주기로 조정)

| 설정 위치 | 설명 | 기본값 | 변경 방법 |
| --- | --- | --- | --- |
| `RecurringReservationScheduler.ensureUpcomingReservations` `@Scheduled` | JIT 예약 보강 실행 시각 | `0 0 3 * * *` (매일 03시) | 월간 실행을 원하면 `0 0 3 1 * *`와 같이 크론 표현식을 변경 후 재배포 |
| `RecurringReservationScheduler.recoverOverdueReservations` `@Scheduled` | 지연/누락 예약 복구 주기 | `0 5 * * * *` (매시간) | 필요 없으면 주석 처리 또는 `0 0 4 1 * *`처럼 월 1회로 변경 |

## 9. 실패/롤백 시 멤버십 상태 관리 계획

### 9.1 현재 동작 요약
1. **첫 결제 전**: `MembershipUserEntity`는 `READY` 상태로 대기한다.
2. **첫 결제 성공 → `activateUserMembershipAfterPayment`**:
   - 다른 활성 멤버십이 있다면 `EXPIRED`로 전환 후 로그를 남긴다.
   - 대상 멤버십을 `ACTIVATE`로 변경한다.
3. **첫 결제 또는 예약 과정에서 예외 발생**:
   - PortOne 결제는 `compensateActivationFailure`로 취소하지만, 이미 `ACTIVATE`된 멤버십을 `READY`로 되돌리는 로직은 없다.
4. **업그레이드/다운그레이드**:
   - 업그레이드 결제 전 실패 → 기존 플랜이 유지된다.
   - 결제 후, `updateMembership`가 기존 멤버십을 `EXPIRED`, 새 멤버십을 `ACTIVATE`로 만든 뒤 예외 시 롤백 로직이 없다.

### 9.2 개선 계획
- **Activation 이후 예외 시 롤백**  
  1. `activateUserMembershipAfterPayment`가 성공한 뒤 예약 단계에서 예외가 나면, 기존 READY 멤버십을 다시 ACTIVATE하거나 새로 생성했던 멤버십을 READY로 되돌리는 보상 루틴을 추가한다.  
  2. 이 보상 루틴은 `PaymentServiceApiV1.completeBillingKey`의 `catch` 블록에서 PortOne 결제 취소 후 실행한다.

- **업그레이드/다운그레이드 롤백**  
  1. `applyUpgradeWithPayment`/`applyUpgradeWithoutPayment`/`applyPendingPlanIfNecessary` 내부에 “이전 멤버십 스냅샷”을 저장해 두고, 예외 발생 시 원래 멤버십을 즉시 복구한다.  
  2. PortOne 예약 생성 실패 또는 결제 실패 시, `SubscriptionEntity`의 `membershipLevel`, `amount`, `nextBillingDate`를 이전 값으로 되돌리는 메서드를 추가한다.

- **문서 반영**  
  - 본 섹션(9장)에 “멤버십 상태는 ACTIVATE → READY → EXPIRED 흐름이며 FREE는 별도 상태가 아니다”라는 설명과 함께, 보상 트랜잭션 구현 계획을 기록했다.

### 9.3 프리 멤버십 롤백 로직 구현 계획
1. **보상 헬퍼 추가**  
   - `MembershipRollbackService`를 만들어 “기존 프리 멤버십 스냅샷 저장 → 롤백” 기능을 제공.  
   - 스냅샷에는 `membershipUserId`, `startDate`, `endDate`, `membershipState` 등을 저장해두고, 롤백 시 동일 엔티티를 다시 ACTIVATE로 업데이트.
2. **PaymentServiceApiV1 연동**  
   - `activateUserMembershipAfterPayment` 이후 예외 발생 시, 현재 유지 중인 “첫 결제 취소” 로직을 수행한 뒤 `MembershipRollbackService.rollbackToPreviousPlan(...)`을 호출해 프리를 다시 ACTIVATE로 되돌린다.  
   - 숙려 기간 동안에는 기존 전체 취소 로직을 유지하되, 롤백 성공 여부를 로그에 남겨 추후 장애 분석에 활용.
3. **SubscriptionServiceApiV1 연동**  
   - 업그레이드/다운그레이드 결제 실패 또는 예약 실패 시, `MembershipRollbackService`를 호출해 원래 멤버십(예: 프리)을 ACTIVATE로 되돌린다.  
   - 결제가 성공했는데 예약만 실패한 경우 현재는 전체 취소를 진행하고 있으므로, 이후 정책 변경 시 “결제 유지 + 예약 재시도”로 바뀌면 이 롤백 호출 위치도 함께 조정.
4. **추가 모니터링**  
   - 롤백이 실행되면 `ReservationAudit`와 별개로 “membership rollback” 로그를 남겨 운영자가 쉽게 추적할 수 있도록 한다.  
   - 실패가 반복되면 Slack/메일 알림을 전송해 예약 시스템 상태를 확인한다.
