# 구독 업/다운그레이드 및 환불 정책

본 문서는 현재 백엔드 구현과 도메인 규칙을 기준으로 **업그레이드**, **다운그레이드**, **환불(구독 해지)** 시 적용되는 정책을 정리한 자료입니다. 실 서비스 정책 변경 시 본 문서를 함께 업데이트해 주세요.

---

## 1. 업그레이드 정책

### 1.1 지원 범위
- 현재 활성 구독(`SubscriptionEntity`)이 존재해야 한다.
- 다음 조건 중 하나를 만족할 때만 업그레이드가 가능하다.
  - 멤버십 **레벨 상승** (예: Level 2 → Level 3)
  - 동일 레벨에서 **월간 → 연간**으로 주기 변경
- 그 외(레벨 하향, 연간 → 월간)는 업그레이드로 처리하지 않음. 요청 시 `BadRequestException`.

### 1.2 차액 계산 (Hybrid Model)
- 구현 위치: `UpgradeCalculator`.
- 남은 일수 비율 × 남은 사용량 비율을 곱해 현재 플랜의 잔여 가치를 산출 후, 목표 플랜 가격과 비교한다.
- 의사 코드
  ```java
  double remainingDaysRatio = remainingDays / totalDays;
  double remainingUsageRatio = remainingUsage / totalUsage;
  BigDecimal remainingValue = currentPrice * remainingDaysRatio * remainingUsageRatio;
  BigDecimal difference = max(targetPrice - remainingValue, 0);
  ```
- **남은 사용량이 0**이면 `remainingValue` = 0 으로 간주되어 전액 차액 결제가 발생한다.

### 1.3 결제 및 반영 흐름
1. 활성 구독 / 타겟 멤버십 조회 (`SubscriptionRepository`, `MembershipRepository`).
2. 기존 예약(다운그레이드/변경 요청)이 구현 상 존재하지 않아도 차액 계산은 항상 수행.
3. 차액이 0 이하 → `applyUpgradeWithoutPayment` 호출, 즉시 플랜 변경.
4. 차액이 양수 → `applyUpgradeWithPayment` 호출.
   - 최신 활성 빌링키 검증 (`BillingKeyRepository.findLatestActiveByUserId`).
   - PortOne 빌링키 결제(`PortOneV2Client.payWithBillingKey`)로 차액 결제.
   - 결제 성공 시 구독 금액, 주기, 다음 결제일 업데이트.
5. 모든 단계는 `@Transactional` 블록(`SubscriptionServiceApiV1.upgradeSubscription`) 안에서 수행.
- 결제 실패 시 `compensateActivationFailure`로 보상 취소 로직을 호출해 상태를 롤백한다.

### 1.4 로그 & 기록
- 업그레이드 성공/실패에 따라 `MembershipUserLogEntity`에 이벤트가 저장된다 (`action=UPGRADE_APPLY` 등).
- 결제 내역은 `PaymentHistoryEntity`에 `payment_type = UPGRADE`로 기록된다.

## 2. 다운그레이드 정책

### 2.1 현재 상태
- 백엔드에 **자동 다운그레이드** 기능이 구현돼 있지 않다.
- `upgradeSubscription` 메서드에서 다운그레이드 시도는 `BadRequestException("현재 요금제보다 높은 요금제로만 업그레이드할 수 있습니다.")`로 차단된다.

### 2.2 운영 가이드
- 사용자가 더 낮은 요금제를 원할 경우, 다음 옵션 중 하나를 운영 정책으로 채택해야 한다.
  1. **해지 후 재가입**: 기존 구독 해지 → 새로운 하위 플랜으로 가입.
  2. **수동 처리**: 운영/재무 팀에서 DB 수정 또는 별도 스크립트로 조정 (권장 X).
  3. 추후 개발 과제로 남겨 스케줄형 `subscription_change_request` 테이블 설계 및 스케줄러 반영.

- ### 2.3 향후 개발 권장사항
  - 스케줄러(`ChangeRequestProcessor`) 기반 자동 다운그레이드 예약 시스템 설계 및 구현.

- 다운그레이드 자동화를 도입하려면:
  - `SubscriptionChangeRequestEntity`(예약) 설계 및 상태머신 추가.
  - 기존 예약 취소/추가 로직, 스케줄러(`ChangeRequestProcessor`) 확장.
  - 환불/차액 정산 규칙을 명확히 정의해야 한다.

## 3. 환불(구독 해지) 정책

### 3.1 구독 해지 동작
- 구현 위치: `PaymentServiceApiV1.cancelSubscription`.
- 해지 요청 시 다음을 순차적으로 수행한다.
  1. 대상 구독/빌링키 소유자 검증 (`SubscriptionEntity`, `BillingKeyEntity`).
  2. PortOne 예약 결제를 `scheduleId`(=subscriptionId)로 조회한 뒤 존재 시 삭제 (`PortOneV2Client.getPaymentSchedule`, `.deletePaymentSchedule`).
  3. PortOne 빌링키 삭제 (`PortOneV2Client.deleteBillingKey`). 이미 삭제된 경우(404) 경고만 기록.
  4. 내부 테이블 갱신:
     - `SubscriptionEntity.status = CANCELLED`, `endDate = today`.
     - `BillingKeyEntity.status = DELETED`, `expiredAt = now`.
     - 멤버십 상태를 `EXPIRED`로 로그 남김 (`MembershipUserLogRepository`).

### 3.2 환불 규칙
- **선불 결제 모델**: 이미 결제된 기간 동안은 서비스 이용이 가능하며, 해지하더라도 남은 기간에 대한 환불은 자동으로 발생하지 않는다.
- 해지 시점 이후 추가 청구만 차단한다(예약 삭제, 빌링키 삭제).
- **부분/전액 환불**을 제공하려면 별도 정책과 구현이 필요하다.
  - 포트원 결제 취소 API 호출 (`PortOneV2Client.cancelPayment`) 및 금액 산정 로직 추가.
  - 환불 사유, CS 프로세스, 재무 승인 흐름 등 운영 절차 확립 필수.

### 3.3 예외 처리
- PortOne 409 (예약 존재) → `ConflictException("예약 결제가 남아 있어 구독을 해지할 수 없습니다.")`.
- PortOne 404 → 이미 삭제된 것으로 간주하고 계속 진행.
- PortOne 5xx → `ExternalServiceException`으로 변환, Sentry/로그 경고 발생.
- 모든 예외는 `CommonExceptionHandler`에서 HTTP 404/409/502 응답 코드로 매핑된다.

---

## 4. 향후 과제 & 권장 사항

- **다운그레이드 자동화**: 사용자가 원하는 날짜에 자동으로 요금제를 낮춰주는 예약 시스템 설계.
- **환불 옵션**: 선불 기간 잔여 일수/사용량 기반 환불 정책을 도입할 경우, UpgradeCalculator와 유사한 잔여 가치 평가가 필요.
- **고객 안내**: 프런트/CS FAQ에 “업그레이드/해지 시 과금 & 환불 정책”을 명시해 불필요한 문의를 줄인다.
- **로그/모니터링**: 해지와 업그레이드 이벤트를 Sentry, BI 대시보드에 노출해 운영팀이 추세를 모니터링할 수 있도록 한다.
