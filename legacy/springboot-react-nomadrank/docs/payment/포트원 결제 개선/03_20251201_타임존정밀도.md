# 시간·타임존 정밀도 개선

## 배경 / 문제
- `nextBillingDate`가 `LocalDate`만 보관되어 결제 시각/타임존 정보가 유실되고, Cron 일괄 처리(09시/14시)에 의존.
- 사용자 타임존/선호 청구 시각을 반영할 수 없으며, 예약/알림/사용량 리셋이 어긋날 수 있음.

## 목표
- 청구·예약·사용량 리셋을 **시각 단위 + 타임존** 기준으로 관리.
- 사용자별 선호 청구 시각을 지원하고, PortOne 예약 시간과 DB 시각을 일치시킨다.

## 설계 방향
- `Subscription`에 `nextBillingAt(OffsetDateTime)` 추가, `user.timezone`(IANA) 컬럼 추가.
- PortOne 예약 시간은 UTC로 저장하고, 사용자 타임존을 고려해 예약 시각을 계산.
- 사용량 리셋/알림(예: 결제 3일 전)도 동일 시각 기준으로 스케줄링.

## 작업 항목
- [x] DB 스키마 확장: `next_billing_at` 컬럼 추가 완료(DDL `20251201_recurring_schedule_columns.sql`), 기본값은 `nextBillingDate` + 09:00 KST로 초기화.
- [x] 예약 생성 로직 기본 타임존/시각 설정: `payment.timezone.default`(Asia/Seoul), `payment.timezone.default-billing-time`(09:00) 추가. `RecurringPaymentScheduleService.toOffsetDateTime`가 기본 타임존+09:00을 사용하고 zone/time 오버로드 제공.
- [x] 예약 생성 로직에 사용자 타임존/선호 시각을 받아 계산하도록 변경(프런트 `timezone`/`billingTime`, 서버 오버로드 활용).
- [x] User 엔티티에 `timezone` 컬럼 추가(기본 Asia/Seoul) 및 웹훅 결제 성공 시 사용자 타임존 기반 `nextBillingAt` 재계산.
- [x] 사용량 리셋 서비스가 `nextBillingAt` 기준으로 동작하도록 조정(`UsageResetService`가 `nextBillingAt` → 청구 주기 역산).
- [ ] 알림/이메일 발송 시각을 타임존 기반으로 계산.
- [ ] 마이그레이션 스크립트/배치로 기존 구독의 `nextBillingAt` 채움(초기화 완료, 사용자 타임존 반영은 미완).

## 추가 계획
- `RecurringPaymentScheduleService.toOffsetDateTime` 오버로드에 사용자 타임존/선호 시각을 실제 호출부에서 전달(User.timezone/사용자 입력 연동).
- 알림/이메일/예약 생성 시 모든 시간 계산을 `nextBillingAt` + 사용자 타임존으로 통일.
- 사용자 타임존 컬럼(User) 추가 여부 결정 후, 예약/알림 시각 계산에 반영.

## 프런트 연동 가이드
- API 변경점: `SubscriptionRequest`에 `timezone`(IANA), `billingTime`(HH:mm)이 추가되었고 `scheduleAt`은 선택 입력 가능(입력 시 ISO-8601 OffsetDateTime 권장).
- 권장 시나리오:
  1) 클라이언트가 사용자 타임존과 선호 청구 시각을 알고 있다면 `timezone`/`billingTime`을 전송하고, `scheduleAt`을 비워두면 서버가 기본 청구일 + 시각으로 계산한다.
  2) 정확한 청구 시각을 프런트에서 이미 계산했다면 `scheduleAt`에 타임존 오프셋을 포함해 전송한다(예: `2025-01-02T09:00:00+09:00`). `timezone`/`billingTime`은 보조 정보로만 사용된다.
- 저장 권장: 사용자 설정 화면에서 IANA 타임존과 선호 청구 시각을 저장하고, 모든 결제/구독 요청 시 함께 전송해 일관된 예약 시각을 보장한다.
- 기본값/폴백: 잘못된 타임존/시각이 전달되면 서버가 `payment.timezone.default`(Asia/Seoul)과 `payment.timezone.default-billing-time`(09:00)으로 대체한다.

## 리스크 / 대응
- 시간대 변환 오류로 조기/지연 청구 위험: UTC 기준 저장 + 변환 유닛 테스트 추가.
- 이행 과정에서 null 시각 발생: 기본값(사용자 타임존 미지정 시 Asia/Seoul, 시각 09:00)으로 채움.

## 측정 지표
- 예약 시각과 PortOne 실제 결제 시각 차이 평균/분산.
- 시간대 관련 고객 문의/오류 건수.
