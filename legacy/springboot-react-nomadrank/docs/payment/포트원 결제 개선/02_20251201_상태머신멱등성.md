# 구독 상태 머신 및 멱등성 정리

## 배경 / 문제
- 해지/업/다운그레이드/결제 성공·실패가 `PaymentServiceApiV1`와 `SubscriptionServiceApiV1`에 흩어져 있고, 단순 상태 변경 메서드가 남아 있어 불완전 전이가 가능.
- 웹훅/중복 요청/재시도에 대한 멱등 키 처리가 없어 동일 이벤트가 여러 번 적용될 위험.

## 목표
- 단일 상태 머신과 이벤트 기반 전이 테이블을 정의하고, 모든 경로에서 이를 따르도록 강제.
- 멱등 키 기반으로 중복 호출·웹훅 재전송에도 동일 결과를 반환하도록 한다.

## 설계 방향
- 상태 집합 명시: `PENDING` → `ACTIVE` → (`SUSPENDED` ↔ `ACTIVE`) → `PENDING_CANCEL` → `CANCELLED`/`EXPIRED` 등 전이 그래프 정의.
- 이벤트 타입 정의: `PAYMENT_PAID`, `PAYMENT_FAILED`, `SCHEDULE_CREATED`, `SCHEDULE_DELETED`, `USER_CANCEL_REQUESTED`, `DOWNGRADE_SCHEDULED`, `UPGRADE_REQUESTED`, `BILLING_KEY_DELETED` 등.
- 각 이벤트에 대해 **전이 함수**와 **멱등 키**(예: `eventId`, `paymentId`, `scheduleId+timestamp`)를 요구하고, 처리 결과를 이벤트 로그에 기록.

- [ ] `SubscriptionServiceApiV1` 진입부에서 이벤트 객체를 받아 전이 함수로 위임하도록 리팩터링.
- [x] 멱등 키 테이블 또는 이벤트 로그 테이블 추가(`event_id`, `source`, `payload_hash`, `processed_at` 등) 후 중복 차단.
- [ ] 기존 단순 상태 변경 메서드(`cancelSubscription`, `suspendSubscription` 등) 폐기 또는 이벤트 래핑.
- [ ] 웹훅 핸들러에 멱등 키 검증 추가(PortOne `paymentId`/`scheduleId` 기반).

## 리스크 / 대응
- 기존 호출부와 새 이벤트 인터페이스 불일치: 어댑터 계층을 두고 단계적 마이그레이션.
- 상태 전이 회귀 버그: 전이 테이블 기반 단위/통합 테스트 추가.

## 측정 지표
- 중복 이벤트 차단률(중복 수신 대비 중복 적용 건수).
- 불허 전이 시도 건수 및 오류율.

## 상태 머신 명세 (초안)
| 현재 상태 | 이벤트 | 다음 상태 | 부가 처리 |
| --- | --- | --- | --- |
| `PENDING` | `PAYMENT_PAID` | `ACTIVE` | 멤버십 활성화, `nextBillingAt`/`scheduleId` 갱신 |
| `PENDING` | `PAYMENT_FAILED` | `PENDING` | 실패 카운트 증가, 알림/재시도 큐 적재 |
| `ACTIVE` | `PAYMENT_PAID` | `ACTIVE` | 실패 카운트 초기화, 다음 결제일 계산 |
| `ACTIVE` | `PAYMENT_FAILED` | `ACTIVE` or `SUSPENDED` | 실패 카운트/메시지, 임계값 도달 시 `SUSPENDED` |
| `ACTIVE` | `USER_CANCEL_REQUESTED` | `PENDING_CANCEL` | 해지 예약일 설정, 예약 삭제 |
| `PENDING_CANCEL` | `BILLING_KEY_DELETED` | `CANCELLED` | 즉시 해지, 빌링키 삭제 |
| `PENDING_CANCEL` | `PAYMENT_PAID` | `ACTIVE` | 해지 예약 취소, 결제 성공 처리 |
| `SUSPENDED` | `PAYMENT_PAID` | `ACTIVE` | 실패 카운트 초기화, 예약 재생성 |
| any | `BILLING_KEY_DELETED` | `SUSPENDED`/`CANCELLED` | 예약 삭제, 알림 |
| any | `EXPIRE` | `EXPIRED` | 만료 처리, 빌링키 무효화 |

## 이벤트/멱등 설계 (초안)
- 이벤트 DTO: `SubscriptionEvent { eventId, source(REST|WEBHOOK|SCHEDULER), type, subscriptionId, occurredAt, payloadHash, payload }`
- 멱등 저장소 테이블 예: `subscription_events`(PK=`event_id`), `processed_at`, `status`, `error_message`.
- 키 룰: `paymentId`(결제/웹훅), `scheduleId+status`(예약), `operationId`(업/다운/해지 요청), `billingKey` 이벤트.
- 처리 순서: 이벤트 수신 → 멱등 체크 → 전이 함수 적용 → 결과/부가 처리(예약 삭제 등) → 이벤트 로그 업데이트.

## 구현 계획 (업데이트)
1) 전이 테이블/함수 정의: 상태×이벤트 → 액션/다음 상태 매핑을 코드화(Enums + Map)하고 단위 테스트 작성.
2) `SubscriptionServiceApiV1`에 이벤트 처리 진입점 추가: 기존 메서드는 이벤트 생성 후 전이 처리기로 위임. (업/다운/해지 멱등 적용, operationId 사용)
3) 멱등 테이블/리포지토리 추가 + 공통 멱등 체크 유틸 작성. (완료)
4) 웹훅/스케줄러/REST 컨트롤러 진입부에 멱등 검사 적용(PaymentId, scheduleId 기반).
5) 단순 상태 변경 메서드 제거/래핑: `cancelSubscription`, `suspendSubscription`, `resumeSubscription` 등.
6) 통합 테스트: 중복 이벤트/웹훅 재전송, 순서 뒤바뀜 시 안전성 검증.

## 적용 순서 (리팩터링 단계)
1) 간단 경로부터 적용: `subscribe`, `completeBillingKey`(구독 생성/첫 결제) → `eventId` = `orderId`/`paymentId` 기반으로 `SubscriptionEventProcessor` 감싸기.
2) 해지/업/다운/빌링키 삭제 경로 적용: `eventId` = `operationId` (클라이언트 제공) 또는 `cancel-<subId>-<requestId>` 등 일관된 규칙으로 멱등 처리.
3) 나머지 스케줄러/웹훅/백오피스 경로는 기존 패턴 유지(`webhook-paid/failed-<paymentId>`, `schedule-sync-<scheduleId>-<status>`).

## operationId 프런트 반영 계획
- 해지/업그레이드/다운그레이드/빌링키 삭제 요청 시 프런트에서 `operationId`(고유 requestId)를 생성해 API에 포함.
- 규칙 예시: `op-<userId>-<timestamp>-<uuid>` (클라이언트 생성), 서버는 존재 시 그대로 사용해 멱등 키로 활용.
- 재시도 시 동일 `operationId`를 재사용하도록 가이드하여 중복 실행을 차단.

## 이벤트 ID 규칙 (초안)
- 결제/첫 결제/빌링키 완료: `payment-<paymentId>`
- 정기 예약 생성: `schedule-create-<orderId>`
- 해지: `cancel-<subscriptionId>-<operationId>`
- 업그레이드/다운그레이드: `upgrade/downgrade-<subscriptionId>-<operationId>`
- 웹훅: `webhook-paid/failed-<paymentId>`
- 예약 싱크: `schedule-sync-<scheduleId>-<status>`, 예약 미존재: `schedule-missing-<scheduleId>`

## ID 역할 정리
- `paymentId`: 결제 트랜잭션 고유 키. 첫 결제/정기 결제/업그레이드 결제 등 결제 이벤트의 멱등 키로 사용.
- `orderId`: 정기 결제 예약 생성 시 사용되는 주문 키. `schedule-create-<orderId>`로 예약 중복 생성을 방지.
- `scheduleId`: PortOne 예약 ID. 예약 동기화/삭제 이벤트의 멱등 키(`schedule-sync-*`, `schedule-missing-*`).
- `operationId`: 사용자 주도 요청(해지/업그레이드/다운그레이드/빌링키 삭제 등)의 클라이언트 생성 요청 ID. 동일 요청 재시도 시 같은 `operationId`를 보내 멱등 보장.
- `eventId`: 위 키들을 조합해 만든 멱등 처리용 최종 식별자. 이벤트 처리기에 전달되어 중복 실행을 차단하고 로그에 기록.

## 검증 시나리오 / 로그 체크리스트
- 중복 요청 멱등: 동일 `operationId`로 업/다운/해지 재시도 시 서버 로그에 중복 처리 메시지(예: `[Upgrade] 중복 업그레이드 요청`)가 찍히고 추가 로직 실행이 없는지 확인.
- 결제/웹훅 멱등: 동일 `paymentId` 웹훅 재전송 시 한 번만 처리되고 이벤트 로그에 PROCESSED/중복 표시가 남는지 확인.
- 예약 싱크: `schedule-sync-*` 이벤트가 반복되어도 상태만 최신화되고 중복 처리 없이 로그가 남는지 확인.
- 불허 전이: 허용되지 않은 상태 전이 시 예외 발생 및 이벤트 로그 FAILED 기록 확인.
- 이벤트 로그 적재: `SubscriptionEventLog`에 eventId/source/status/processedAt 등이 쌓이는지 샘플 조회.
- operationId 전달: 프런트 재시도 시 동일 `operationId`가 Body/Idempotency-Key에 전달되는지 HTTP 로그로 확인.

## 다이어그램
[REST/프론트] --(operationId/requestId 포함)--> [컨트롤러]
    |
    v
[SubscriptionServiceApiV1 / PaymentServiceApiV1]
    ├─ eventId 생성 (paymentId / orderId / operationId / scheduleId 조합)
    ├─ SubscriptionEventProcessor.processEvent(eventId, currentStatus, eventType, source, payloadHash, action)
    │      ├─ SubscriptionEventLogService: alreadyProcessed? → 중복이면 반환
    │      ├─ 상태 전이 허용 여부 검사(전이 테이블)
    │      └─ action 실행(예약/결제/해지/업다운 로직) 후 log → PROCESSED/FAILED
    └─ 응답 DTO 생성

[Webhook (PortOne)] --(paymentId/scheduleId)--> PaymentWebhookService
    └─ eventId = webhook-paid/failed-<paymentId>
       → SubscriptionEventProcessor (PAYMENT_PAID/FAILED) → 상태/다음 결제일 갱신

[Scheduler]
    ├─ SubscriptionScheduler (백업 스케줄러: 결제 직접 호출 여부 flag)
    └─ SubscriptionScheduleSyncService (예약 싱크)
        ├─ PortOne 예약 조회(scheduleId)
        ├─ eventId = schedule-sync-<scheduleId>-<status> or schedule-missing-<scheduleId>
        └─ SubscriptionEventProcessor (SCHEDULE_CREATED/DELETED) → scheduleStatus/nextBillingAt 갱신

[이벤트/멱등 로그]
    └─ SubscriptionEventLog (event_id, subscription_id, event_type, source, payload_hash, status, error_message, processed_at)
       └─ SubscriptionEventLogService (recordReceived / markProcessed / markFailed / alreadyProcessed)

[상태 전이 테이블]
    ├─ PENDING: PAYMENT_PAID→ACTIVE, PAYMENT_FAILED→PENDING, USER_CANCEL_REQUESTED→PENDING_CANCEL, BILLING_KEY_DELETED→SUSPENDED
    ├─ ACTIVE: PAYMENT_PAID→ACTIVE, PAYMENT_FAILED→ACTIVE/SUSPENDED(실패 카운트), USER_CANCEL_REQUESTED→PENDING_CANCEL,
    │         DOWNGRADE_SCHEDULED/UPGRADE_REQUESTED→ACTIVE, BILLING_KEY_DELETED→SUSPENDED, MANUAL_SUSPEND→SUSPENDED
    ├─ PENDING_CANCEL: PAYMENT_PAID→ACTIVE, BILLING_KEY_DELETED→CANCELLED, EXPIRE→EXPIRED
    ├─ SUSPENDED: PAYMENT_PAID→ACTIVE, MANUAL_RESUME→ACTIVE, BILLING_KEY_DELETED→CANCELLED
    ├─ EXPIRED: PAYMENT_PAID→ACTIVE
    └─ 예약 이벤트(SCHEDULE_CREATED/DELETED): 상태 유지, scheduleStatus/nextBillingAt만 갱신

[ID 역할]
    - paymentId: 결제 트랜잭션 멱등 키
    - orderId: 정기 예약 생성 키
    - scheduleId: 예약 동기화/삭제 키
    - operationId: 사용자 요청 멱등 키(해지/업/다운/빌링키 등), 프런트가 생성·재사용
    - eventId: 위 키들을 조합한 최종 멱등 처리 식별자
