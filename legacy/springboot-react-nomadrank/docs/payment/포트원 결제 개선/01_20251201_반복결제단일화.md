# 포트원 반복 결제 단일화

## 배경 / 문제
- PortOne 예약(웹훅 기반)과 내부 스케줄러(09시/14시 직접 청구)가 공존해 중복 청구·상태 불일치 위험 존재.
- `SubscriptionEntity.nextBillingDate`만 보고 내부 청구가 실행될 경우 PortOne 예약이 이미 진행 중인지 구분 불가.
- 예약 삭제/재생성 시점이 모호해 업/다운그레이드, 해지 시 이중 예약/실행 가능.

## 목표
- 반복 결제 드라이버를 단일 소스로 통일하고, 예약/취소/재예약 흐름을 명확히 정의.
- 예약 상태를 관측 가능하게 만들어 장애 시 재처리/복구가 용이하도록 함.

## 결정 방향
1. **PortOne 예약 우선**: 반복 청구는 PortOne 스케줄만 사용, 내부 직접 청구는 제거(또는 장애 대응 백업으로 분리)한다.
2. 예약 생성·삭제는 `RecurringPaymentScheduleService` 한 곳에서 수행하고, `Subscription`에 예약 ID·최근 상태를 저장한다.
3. 웹훅이 미도착/실패 시 재조회→재예약/보상 프로세스를 정의한다.

## 현행 흐름 조사 (초안)
- **PortOne 예약 생성**: `PaymentServiceApiV1.subscribe` 및 `completeBillingKey` → `RecurringPaymentScheduleService.createReservation` 호출, 예약 ID를 `SubscriptionEntity.subscriptionId`로 재사용(현재 예약 상태 컬럼 없음).
- **내부 직접 청구**: `SubscriptionScheduler.processDueSubscriptions/processRetryableSubscriptions`가 `SubscriptionServiceApiV1.processSubscriptionPayment`를 호출, `payWithBillingKey`로 바로 결제. PortOne 예약 존재 여부와 무관하게 실행 가능.
- **웹훅 처리**: `PaymentWebhookService`가 `PaymentType.RECURRING` 성공/실패 시 `SubscriptionEntity`의 `nextBillingDate`/`failureCount`/`status`를 갱신하지만 예약 ID/상태는 미추적.
- **예약 삭제/변경 트리거**: 해지(`PaymentServiceApiV1.cancelSubscription`), 업/다운그레이드(`SubscriptionServiceApiV1`), 빌링키 삭제 웹훅에서 `RecurringPaymentScheduleService.deleteSchedulesByBillingKey` 호출. 성공 여부/예약 ID 기록 없음.
- **데이터 구조**: `SubscriptionEntity`는 `nextBillingDate(LocalDate)`만 보유, 예약 시각/타임존/예약 상태를 저장하지 않아 싱크 불일치 감지 불가.

## 설계 고도화 (초안)
- **단일 소스 결정**: PortOne 예약을 기본/유일 실행 경로로 채택. 내부 직접 청구는 기본 비활성화, `use_backup_scheduler=false` 기본값. 장애 대응용 백업 스케줄러는 별도 feature flag + 락 기반으로만 실행.
- **신규 컬럼 스키마**: `Subscription.scheduleId`(예약 ID), `scheduleStatus`(PENDING/CONFIRMED/FAILED/CANCELLED/UNKNOWN), `scheduleLastSyncedAt`(OffsetDateTime), `nextBillingAt`(OffsetDateTime, 타임존 포함). `nextBillingDate`는 마이그레이션 후 유지/폐기 여부 결정.
- **예약 규칙 표(요약)**:
  - 생성: 가입/첫 결제 성공 시 PortOne 예약 생성, `scheduleId/status` 저장.
  - 변경: 업/다운그레이드 적용 시 예약 금액/주기 변경 → 기존 예약 삭제 후 새 예약 생성.
  - 해지: `PENDING_CANCEL`/`CANCELLED` 전이 시 예약 삭제, `scheduleStatus=CANCELLED`.
  - 빌링키 삭제/비활성: 예약 삭제 후 `scheduleStatus=FAILED` 기록, 구독 `SUSPENDED`.
  - 실패 누적 정지: 내부 재시도 정책에 따라 예약 유지/삭제 결정(정책 문서 연계).
- **싱크 배치**: 주기적으로 `scheduleId` 기준 PortOne 예약 조회 → 로컬 상태 업데이트. 불일치/미존재 시 `FAILED`/`MISSING` 상태로 플래그하고 알림·재예약 큐에 적재.
- **알림/모니터링**: 예약 생성/삭제 실패율, `MISSING` 건수, 중복 청구 탐지 지표를 대시보드로 노출.

## 구현 세부 계획
- 1단계: 스케줄러 직접 청구 비활성화 + 스키마 확장
  - `SubscriptionScheduler.processDueSubscriptions/processRetryableSubscriptions`에 feature flag(`payment.recurring.backup-scheduler.enabled=false`) 추가, 기본 비활성화.
  - `SubscriptionServiceApiV1.processSubscriptionPayment`는 flag off 시 즉시 return/log만 수행.
  - `SubscriptionEntity` 필드 추가: `scheduleId`, `scheduleStatus`, `scheduleLastSyncedAt`, `nextBillingAt`. 리포지토리/쿼리 수정.
  - DB 마이그레이션 스크립트 작성(DDL + 기본값 채우기: `scheduleStatus=UNKNOWN`, `nextBillingAt=nextBillingDate at 09:00 KST`).

- 2단계: 예약 생성/변경/삭제 일원화 및 규칙 적용
  - `RecurringPaymentScheduleService.createReservation/deleteSchedulesByBillingKey` 반환값에 예약 ID/결과 코드 포함 후 저장.
  - 가입/첫 결제/업/다운/해지/빌링키 삭제 시 예약 조작 규칙을 코드화(단일 helper 사용).
  - 실패/예외 시 `scheduleStatus=FAILED`와 에러 메시지 로깅, 재시도 큐 적재.
  - `nextBillingAt` 갱신 시 PortOne 예약 시간과 일치하도록 계산 후 저장.

- 3단계: 예약 싱크 배치/알림
  - 잡: `syncPortoneSchedules`(예: 1시간 주기). `scheduleId` 기준 PortOne 조회 → 상태 반영, 미존재 시 `MISSING`.
  - `MISSING/FAILED` 발견 시 Slack/메일 알림 + 재예약 트리거 옵션.
  - 백업 스케줄러를 사용할 경우: 실행 전 락 + `scheduleStatus!=CONFIRMED` 조건에서만 결제 시도.

- 데이터 마이그레이션
  - 모든 `ACTIVE/PENDING` 구독에 대해 `scheduleId` 없음 → PortOne 예약 존재 여부 조회 후 생성/채움.
  - `nextBillingAt` 기본값: `nextBillingDate`가 있으면 09:00 KST, 없으면 `now()+1month` 계산.
  - 마이그레이션 로그: 생성/실패 건수, 실패 사유 csv로 별도 남김.

- 데이터 마이그레이션 스크립트(초안)
  - DDL
    - `ALTER TABLE SUBSCRIPTIONS ADD COLUMN schedule_id VARCHAR(100);`
    - `ALTER TABLE SUBSCRIPTIONS ADD COLUMN schedule_status VARCHAR(30) DEFAULT 'UNKNOWN';`
    - `ALTER TABLE SUBSCRIPTIONS ADD COLUMN schedule_last_synced_at DATETIME NULL;`
    - `ALTER TABLE SUBSCRIPTIONS ADD COLUMN next_billing_at DATETIME NULL;`
  - DML 초기화(예시; 실제는 배치/Java로 실행)
    - `UPDATE SUBSCRIPTIONS SET schedule_status='UNKNOWN', schedule_last_synced_at=NOW() WHERE schedule_status IS NULL;`
    - `UPDATE SUBSCRIPTIONS SET next_billing_at = CONVERT_TZ(CONCAT(next_billing_date, ' 09:00:00'), 'Asia/Seoul', 'UTC') WHERE next_billing_date IS NOT NULL AND next_billing_at IS NULL;`
  - 예약 생성/주입 배치
    - 대상: `status IN (ACTIVE, PENDING)` AND `schedule_id IS NULL`
    - 절차:
      1) PortOne 예약 조회(`scheduleId` 미보유 → `customerId=userId`/`billingKey` 기준 조회 불가하므로 새 예약 생성)  
      2) 새 예약 생성 후 `scheduleId/timeToPay`를 `schedule_id/next_billing_at`에 저장, `schedule_status=PENDING`, `schedule_last_synced_at=NOW()`
      3) 실패/예외 건은 csv 로그(`subscription_id,user_id,reason`)로 적재

- QA/릴리스 체크리스트
  - 유닛: 예약 헬퍼/규칙, feature flag off 시 결제 미실행.
  - 통합: 가입→첫 결제→예약 생성, 업/다운→예약 갱신, 해지→예약 삭제, 빌링키 삭제→예약 삭제.
  - 회귀: 중복 청구 금지 시나리오(PortOne 예약 + 백업 스케줄러) 검증.
  - 모니터링: 예약 실패율, MISSING 건수, 웹훅 처리 지연 지표 대시보드 확인 후 배포.
