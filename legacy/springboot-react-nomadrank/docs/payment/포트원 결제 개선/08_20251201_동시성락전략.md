# 동시성/락 전략

## 배경 / 문제
- 해지/업그레이드/다운그레이드/재시도 등 동시 요청 시 레이스 컨디션을 막을 제어 장치가 없음.
- 스케줄러와 API 호출이 같은 구독을 병렬로 수정할 수 있어 중복 청구/상태 꼬임 가능.

## 목표
- 구독 단위 동시성 제어로 중복 처리 방지, 상태 일관성 유지.
- 장기 실행 작업(PortOne 호출) 중에도 멱등성을 확보.

## 설계 방향
- **분산 락**: `subscription_id` 기준으로 Redisson/DB 락 적용, TTL과 리트라이 설정.
- **낙관적 락**: `Subscription`에 `version` 컬럼 추가, 갱신 시 버전 충돌 감지.
- **작업 토큰**: 업/다운그레이드, 해지 등 요청에 `operationId`를 발급해 동일 토큰 재요청 시 중복 처리 방지.
- 스케줄러/웹훅/사용자 API 모두 동일 락 유틸을 사용하도록 추상화.

## 작업 항목
- [ ] `SubscriptionEntity`에 `version` 추가 및 JPA 낙관락 활성화, 충돌 시 재시도 전략 정의.
- [ ] 공용 `SubscriptionLockService` 작성(Redisson or DB advisory lock).
- [ ] 주요 진입점(`subscribe`, `upgrade`, `cancel`, `processSubscriptionPayment`, `webhook`)에 락 적용.
- [ ] PortOne 호출 전후 크리티컬 섹션 최소화, 타임아웃/서킷브레이커 설정.
- [ ] 락 실패/타임아웃 시 사용자 메시지/재시도 가이드 제공.

## 리스크 / 대응
- 락 경합/타임아웃 증가: 크리티컬 섹션 분리, 재시도 백오프 적용.
- 낙관락 예외 미처리로 사용자 오류 노출: 공통 예외 핸들러로 사용자 친화 메시지 제공.

## 측정 지표
- 락 획득 실패/타임아웃률.
- 버전 충돌 발생률 및 재시도 성공률.
