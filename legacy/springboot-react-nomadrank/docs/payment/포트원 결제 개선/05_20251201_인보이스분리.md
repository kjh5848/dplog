# 인보이스/청구 레이어 분리

## 배경 / 문제
- `PaymentHistory`가 결제 트랜잭션과 청구 주기를 혼합해 기록, 회계/정산/조정에 필요한 단위(인보이스)가 없음.
- 업/다운그레이드, 부분 환불, 조정 내역을 추적하기 어려움.

## 목표
- `Subscription`(계약) ↔ `Invoice`(주기별 청구) ↔ `Charge`(결제 트랜잭션) 3계층으로 분리해 투명한 정산/보고가 가능하도록 한다.
- 조정/할인/환불을 인보이스 레벨에서 관리.

## PaymentHistory vs Charge
- PaymentHistory: 결제 시도/성공/실패 로그 성격, 청구서(Invoice) FK가 불완전했고 charge_type 개념이 없었음.
- Charge: 결제 트랜잭션 단위로 인보이스와 1:N(라인 기준) 관계를 명확히 연결, `charge_type`으로 INITIAL/RECURRING/UPGRADE/REFUND 등 정산 타입을 구분.
- 목표: PaymentHistory를 Charge로 리네이밍·대체해 결제/청구 레이어를 일관 관리.

## 설계 방향
- 엔티티 추가:
  - `Invoice`: `invoice_id`, `subscription_id`, `period_start/end`, `amount_due`, `currency`, `status(PENDING|PAID|VOID|REFUNDED)`, `issued_at`, `due_at`.
  - `InvoiceLine`: 멤버십 요금, 크레딧/할인, 프로레이션 항목 등.
  - `Charge`는 기존 `PaymentHistory`를 대체/확장(`invoice_id` FK 추가).
- 결제 흐름: 인보이스 생성 → Charge 생성/결제 → 성공 시 Invoice `PAID`, 실패 시 재시도/VOID/부분 지불 등 상태 전이.
- 업/다운그레이드 시 인보이스 라인 조정/크레딧 발행으로 기록.

## 왜 해야 하는가 / 미룰 수 있는가
- PaymentHistory → Charge 일관화  
  - 필요성: 결제 트랜잭션과 청구서를 분리해 정산/환불/재시도 시 데이터 정합성 확보. 현재는 부분 적용이라 경로마다 불일치 가능.  
  - 미루면: 기능은 동작하나 보고/정산/환불 시 누락·중복 위험.
- 프로레이션/환불 계산기 라인화  
  - 필요성: 업/다운/환불 금액을 라인(양수/음수)으로 명시해 사용자·회계에 근거 제공.  
  - 미루면: 금액 근거가 불투명할 수 있음(표시/정산 시 설명 어려움).
- 다운/환불/보상 전 경로 라인 연계  
  - 필요성: 모든 보정/환불 금액을 인보이스/라인에 기록해 멱등·정산 일관성 확보.  
  - 미루면: 일부 경로 데이터가 인보이스에 반영되지 않아 불일치/추후 데이터 정리 비용 증가.
- 보고/다운로드 확장  
  - 필요성: 월간 청구 합계/미수금 등 정산 지표 확보. 현재는 기본 목록/상세/CSV/요약만 제공.  
  - 미루면: 운영 보고 급하지 않다면 후순위 가능하나, 향후 정산 시 추가 작업 필요.

## 인보이스 생성 시점 (정의)
- 첫 결제: 결제 성공 직후 Invoice 즉시 생성(`issued_at=now`, status=PAID), line=멤버십 요금.
- 정기 결제: `nextBillingAt` 기준 N일 전(기본 3일 전) PENDING 인보이스 생성 → 결제 성공 시 PAID, 실패 시 상태 유지/VOID.
- 업/다운/프로레이션: PENDING 인보이스는 라인 조정, 이미 발행/결제된 경우 크레딧/추가 청구 인보이스 별도 발행.

## PaymentHistory → Charge 리팩터링(계획)
- PaymentHistory에 `invoice_id`, `charge_type` 컬럼 추가 후 Charge 모델로 명확화.
- Charge 타입 예: INITIAL, RECURRING, PRORATION, ADJUSTMENT, REFUND(음수), CANCELLATION_FEE 등.
- 구독/웹훅/스케줄러에서 결제 성공 시 해당 invoice_id를 갱신(PAID)하고 Charge 기록.

## 프로레이션/환불 계산 (라인 단위, 계획)
- 업그레이드: 남은 기간 금액 크레딧 라인(-), 신규 플랜 일할 계산 라인(+) → 차액 결제(Charge=PRORATION/UPGRADE).
- 다운그레이드: 잔여 금액 크레딧 라인(-)만 발행(즉시 환불/크레딧), 다음 주기부터 새 요금제 라인(멤버십 라인)으로 청구.
- 환불: 원 Charge 기준 환불 금액을 음수 라인(PRORATION/REFUND/CREDIT)으로 기록, 결제 기록에는 chargeType=REFUND로 남김.

## 보고/쿼리 아이디어
- 월간 청구 합계: invoices status=PAID sum(amount_due) by month/user.
- 미수금: status=PENDING/VOID, due_at 지연 건 count/sum.
- 업/다운/프로레이션 영향: invoice_lines where line_type in (PRORATION, CREDIT, DISCOUNT).

## 프런트 연동(안)
- 백엔드 API 예시
  - `GET /api/invoices?status=&from=&to=`: 사용자별 인보이스 목록 조회.
  - `GET /api/invoices/{invoiceId}`: 인보이스+라인 상세.
  - `GET /api/invoices/{invoiceId}/charges`: 해당 인보이스 결제(Charge) 내역.
  - `GET /api/reports/invoices?month=`: 월간 집계(다운로드/표시용).
- 응답 필수 필드: `invoiceId`, `status`, `periodStart/end`, `amountDue`, `currency`, `issuedAt`, `dueAt`, `paidAt`, `lines[{type,description,amount,currency}]`, `charges[{paymentId,amount,status,paidAt}]`.
- 인증/권한: 사용자 본인/관리자만 조회. 프런트는 기존 토큰을 전달.
- 프런트 작업: 테이블(인보이스 목록), 상세 드로어(라인/결제 내역), 다운로드 버튼(CSV/Excel) 구현.

## 리스크 / 대응
- 대규모 마이그레이션으로 기존 결제 데이터 연계 깨짐: 병행 기록 기간 두고 검증 쿼리 작성.
- 인보이스 생성 타이밍 변경에 따른 사용자 경험 영향: 이메일/알림 UX와 함께 배포.
