# 멤버십/사용량 처리 분리 및 이벤트 기반 보상

## 배경 / 문제
- 결제/구독 처리와 멤버십 상태 변경/사용량 초기화가 한 트랜잭션에 묶여 있어 실패 시 롤백 범위가 크고 보상/재실행이 어렵다.
- 업/다운그레이드, 해지 시 사용량 초기화/로그 기록이 직접 호출되어 재처리/멱등성 확보가 어렵다.

## 목표
- 결제/구독과 멤버십/사용량 처리를 느슨하게 결합하고, 이벤트 기반으로 재실행·보상 가능하게 한다.
- 장애 시 보상 트랜잭션으로 멤버십/사용량 상태를 복원할 수 있게 한다.

## 설계 방향
- 도메인 이벤트 정의: `MembershipActivated`, `MembershipExpired`, `SubscriptionCancelled`, `PlanChanged`, `UsageResetRequested` 등.
- 결제/구독 서비스는 이벤트 발행까지 책임, 멤버십/사용량 서비스는 구독 없이 이벤트 구독자로 처리.
- 실패 시 이벤트 상태를 `PENDING/FAILED`로 기록하고, 리트라이 큐/배치에서 재처리.
- 사용량 초기화 정책(`UsageResetService`)을 이벤트 핸들러로 이동, 멱등 키(회원 ID+주기)를 사용.

## 작업 항목
- [ ] 이벤트 엔티티/퍼블리셔/컨슈머 인터페이스 정의(JPA outbox, Kafka 등 선택).
- [ ] 결제/구독 서비스에서 멤버십/사용량 직접 호출 제거 → 이벤트 발행으로 치환.
- [ ] `UsageResetService` 멱등 처리(동일 주기 중복 호출 방지) 및 이벤트 핸들러화.
- [ ] 보상 트랜잭션 정의: 이벤트 처리 실패 시 적용할 롤백/정정 로직 명시.
- [ ] 모니터링: 미처리/실패 이벤트 대시보드, 알림 설정.

## 리스크 / 대응
- 이벤트 전파 지연으로 사용자 상태 반영이 늦을 수 있음: 동기/비동기 경계를 정의하고, 사용자-facing 경로는 동기 처리 유지.
- Outbox/메시지 브로커 운영 복잡도: 초기에는 DB Outbox + 배치 폴링으로 시작 후 필요 시 Kafka 전환.

## 측정 지표
- 미처리/실패 이벤트 건수 및 평균 지연.
- 멤버십/사용량 상태 불일치 발생률.
