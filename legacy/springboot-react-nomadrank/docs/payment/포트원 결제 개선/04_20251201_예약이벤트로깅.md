# 예약/이벤트 로깅 및 관측성

## 배경 / 문제
- PortOne 예약/삭제/실행 결과와 내부 상태가 불일치할 때 추적 수단이 부족.
- 웹훅 원본 payload, 처리 결과, 재시도 여부를 남기지 않아 장애 원인 파악과 보상이 어려움.

## 목표
- 예약/웹훅/보상 처리 흐름을 이벤트 로그로 남겨 재현 가능성을 높이고, 모니터링 지표를 제공.
- 예약 누락·중복·실패를 조기 감지하는 대시보드/알림을 구축.

## 설계 방향
- 테이블 신설:
  - `payment_schedule_events`: `event_id`, `schedule_id`, `issue_id`, `user_id`, `type(CREATE|DELETE|EXECUTE|MISS)`, `status`, `error_code`, `error_message`, `occurred_at`, `payload`(json).
  - `webhook_events`: `event_id`, `source`, `idempotency_key`, `status`, `processed_at`, `payload`(json).
- 예약 생성/삭제/실행, 웹훅 처리 시 이벤트 로그 기록을 필수화.
- Kibana/Grafana 등으로 집계 지표(예약 실패율, 웹훅 처리 지연, 중복 이벤트 차단) 대시보드 구성.

## 작업 항목
- [x] JPA 엔티티/리포지토리 추가(`payment_schedule_events`, `webhook_events`) 및 로깅 훅 적용.
- [x] `RecurringPaymentScheduleService`에서 예약 성공/실패/삭제 로깅 추가.
- [x] 웹훅 핸들러에 원본 payload 저장 + 멱등 키/처리 결과 기록 추가.
- [x] 배치 싱크 작업 시 예약 미존재/실패 발견 시 `MISS` 이벤트 생성.
- [x] 로그 데이터 보관/마스킹 정책 정의(민감정보 제거).

## 리스크 / 대응
- 로그 저장으로 인한 DB 부하: TTL/파티셔닝, 필요 시 S3 아카이빙.
- 개인정보 포함 위험: 카드/이슈ID 등 민감 필드 마스킹, payload 화이트리스트.

## 마스킹/보관 정책 (제안)
- 카드정보: 카드번호는 앞 6/뒤 4만 남기고 나머지 `*`, CVC/만료월·만료년/전체 번호는 저장하지 않음. 빌링키(issueId)만 저장.
- 연락처: 전화번호 중간 4자리 마스킹(`010-****-1234`), 이메일은 아이디 일부 마스킹(`te****@example.com`) 또는 저장하지 않음.
- 이름/주소: 알림·추적에 불필요하면 저장하지 않음. 필요한 경우 일부만 남기거나 해시.
- 웹훅/예약 payload: 화이트리스트 필드만 저장(결제ID, scheduleId, status, 금액 등). 원본 전체를 저장해야 하면 저장 전 마스킹 필터 적용.
- 로그 보관: 이벤트 로그는 90일 보관 후 삭제 또는 암호화 아카이브(S3 등). DB에는 최근 데이터만 유지.
- 접근 통제: 운영자/개발자 중 필요한 역할만 조회, 조회 이력 감사 로그 남김.
