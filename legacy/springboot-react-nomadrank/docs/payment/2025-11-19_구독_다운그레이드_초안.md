# 구독 다운그레이드 설계 (초안)

본 문서는 **간단한 다운그레이드 예약 기능**을 구현하기 위한 백엔드 설계 초안입니다. 추가 서비스/테이블을 최소화하고, 기존 `SubscriptionEntity`와 스케줄러를 활용해 “다음 결제일부터 하위 요금제 적용”을 제공하는 것을 목표로 합니다.

---

## 1. 목표 및 범위

- 사용자 요청 시, “다음 결제일부터 하위 플랜 적용”을 간단히 제공.
- 새로운 테이블이나 별도 큐 없이 `SubscriptionEntity`를 확장.
- 선불 결제 모델 유지(부분 환불 없음).
- 현재 스케줄러(`SubscriptionServiceApiV1.processDueSubscriptions`)와 자연스럽게 연동.
- PortOne 결제는 다음 결제 차례에 하위 플랜 가격으로 청구.

---

## 2. 요구사항 요약

| 항목 | 내용 |
| --- | --- |
| 사용자 시나리오 | “다음 결제일부터 하위 플랜으로 바꿔 주세요” |
| 즉시 적용 | 지원하지 않음 (선불 유지) |
| 환불/차액 | 없음 (현재 결제 기간 동안 기존 플랜 유지) |
| 저장 방식 | `SubscriptionEntity`에 `pending*` 필드로 예약 |
| 적용 시점 | 정기 결제 스케줄러가 `nextBillingDate` 도달 시 처리 |
| 취소 | `pending*` 필드 null 처리 |
| 기록 | `MembershipUserLogEntity 등 기존 로그 활용 |

---

## 3. SubscriptionEntity 확장

| 필드 | 타입 | 설명 |
| --- | --- | --- |
| pendingMembershipLevel | VARCHAR/INT (nullable) | 다음 결제에 적용할 멤버십 레벨 |
| pendingBillingCycle | ENUM(월간/연간, nullable) | 다음 결제에 적용할 결제 주기 |
| pendingReason | VARCHAR (optional) | 요청 사유 로그용 |
| pendingAppliedAt | DATETIME (nullable) | 적용 완료 시각 (스케줄러가 세팅) |

---

## 4. 서비스 계층 설계

### 4.1 다운그레이드 요청 API (예: `POST /v1/payments/subscriptions/{id}/downgrade`)
1. 활성 구독 조회 (`subscriptionRepository.findActiveByUserId`).
2. 요청한 멤버십/주기가 현재보다 낮은지 검증.
3. 이미 `pending*` 값이 들어있다면 중복 요청 → `BadRequestException` or overwrite 허용 여부 결정.
4. `pendingMembershipLevel`, `pendingBillingCycle`, `pendingReason`만 업데이트 후 저장.
5. `MembershipUserLogEntity`에 `"DOWNGRADE_SCHEDULED"` 로그 남김.
6. 응답에는 `effectiveDate = subscription.getNextBillingDate()` 등을 안내.

### 4.2 취소 API (optional)
```java
@PostMapping("/downgrade/cancel")
```
- `pending*` 값을 null 처리하고 로그에 `"DOWNGRADE_CANCELLED"` 기록.

### 4.3 즉시 다운그레이드 지원 여부
- 선불 모델 유지 가정: 즉시 하향/환불은 지원하지 않음.
- 향후 필요 시 별도의 정책/환불 로직을 추가.

---

## 5. 스케줄러 연동 (`SubscriptionServiceApiV1.processDueSubscriptions`)

기존 흐름에 아래 로직을 추가:
1. `pendingMembershipLevel`이 존재하면
   - `subscription.setMembershipLevel(pendingMembershipLevel)`
   - `subscription.setBillingCycle(pendingBillingCycle)`
   - `subscription.setAmount(resolvePlanPrice(...))`
   - `subscription.setPendingMembershipLevel(null)` (필드 초기화)
   - `subscription.setPendingBillingCycle(null)`
   - `subscription.setPendingAppliedAt(LocalDateTime.now())`
   - `save`
2. PortOne 결제 호출 시, 이미 `subscription.getAmount()`가 하위 플랜 금액으로 세팅되므로 추가 작업 없음.
3. 로그: `MembershipUserLogEntity`에 `"DOWNGRADE_APPLIED"` 저장.

---

## 6. 포트원 연동 고려사항

- **다음 결제 금액**
  - PortOne에는 “다음 결제 금액/주기” 예약 정보를 우리가 직접 전달해야 한다.
  - 정기 결제 스케줄러가 실행되기 전에 `subscription.amount`, `billingCycle` 등 하위 플랜 금액으로 갱신된 상태여야 PortOne 청구 금액이 올바르게 반영된다.
- **예약 수정**
  - PortOne 결제 예약(`payment-schedules`)이 이미 등록되어 있다면, 다운그레이드 요청 시점에 예약 금액을 조정하거나 삭제 후 재생성해야 한다.
  - 해당 작업은 우리 서버에서 PortOne API를 호출하여 수행한다.
- **환불**
  - 선불 모델이므로 부분 환불은 제공하지 않는다는 정책을 유지한다.
  - 환불을 지원하려면 PortOne 결제 취소/부분 취소 API 연동이 추가로 필요하다.

---

## 7. 예외/검증 규칙

- `targetLevel >= currentLevel` → `BadRequestException("하위 요금제로만 변경할 수 있습니다.")`
- `pendingMembershipLevel` 이미 존재 시:
  - 정책상 중복 요청 금지 → 예외
  - 혹은 최신 요청으로 덮어쓰기 → 적용
- 사용량/정책상 제한 조건(예: 특정 플랜은 다운그레이드 불가)을 체크할 수 있음.

## 8. API 계약 초안

### 8.1 Request
`POST /v1/payments/subscriptions/{subscriptionId}/downgrade`

```json
{
  "targetMembershipLevel": 1,
  "targetBillingCycle": "MONTHLY",
  "reason": "사용량이 많지 않아요"
}
```

### 8.2 Response

```json
{
  "status": "SCHEDULED",
  "effectiveDate": "2025-03-01",
  "message": "다음 결제일부터 하위 요금제가 적용됩니다."
}
```

### 8.3 취소 Request
`POST /v1/payments/subscriptions/{subscriptionId}/downgrade/cancel`

응답: `{"status":"CANCELLED","message":"다운그레이드 예약이 취소되었습니다."}`

## 10. 이후 작업 체크리스트

1. `SubscriptionEntity` 필드 추가 (`pendingMembershipLevel`, `pendingBillingCycle`, etc.).
2. 다운그레이드 요청/취소 API 구현 (검증 로직 포함).
3. 스케줄러 로직 보강 (pending 적용).
4. 단위/통합 테스트 작성 (`SubscriptionServiceApiV1` + 스케줄러).
5. 로그/알림/운영 도구 업데이트.
6. 프론트/CS 문서 업데이트.
