# 포트원 예약 API 전환 계획 (2025-11-20)

## 배경
- 기존 구현은 빌링키(`billingKey`)로 `/payment-schedules?billingKey=...` API를 호출해 예약 목록을 가져오는 방식을 사용했다.
- 포트원 V2 공식 REST는 `GET /payment-schedules/{paymentScheduleId}` 단건 조회와 `DELETE /payment-schedules/{paymentScheduleId}` 삭제를 제공하므로, scheduleId 기반으로 전환해야 커버리지와 호환성이 맞는다.

## 요구 사항
1. 정기 결제 예약 생성 시 반환되는 `scheduleId`를 모든 구독 레코드에 저장한다.(기존 `subscriptionId` 컬럼과 동일)
2. 예약 조회/삭제는 빌링키가 아닌 `scheduleId`를 직접 사용한다.
3. 해지 API(`/v1/payments/subscriptions/{subscriptionId}/cancel`)도 scheduleId 기반 PortOne 삭제를 수행하고, 삭제 실패/404 처리 정책을 정의한다.
4. 로그/모니터링에서 scheduleId를 함께 남겨 추적성을 높인다.

## 구현 단계
1. **클라이언트 레이어**
   - [x] `PortOneV2Client`에 `getPaymentSchedule(String scheduleId)` 추가 (`GET /payment-schedules/{scheduleId}`).
   - [x] `deletePaymentSchedules(billingKey)` 추가 (`DELETE /payment-schedules` with JSON body) — scheduleId 배열 대신 빌링키 단일 값으로 호출.
   - [x] 기존 `getPaymentSchedules(billingKey)` 사용부를 제거하고 BillingKey 기반 삭제로 단순화.

2. **서비스 레이어**
   - [x] `RecurringPaymentScheduleService`에 scheduleId 기반 API 호출 메서드로 교체.
   - [x] `cancelSubscription`에서 PortOne 예약 삭제를 빌링키 기반으로 수행(PortOne API 제약으로 scheduleId 배열 대신 billingKey 사용).
   - [x] 예약이 존재하지 않을 때 404를 허용하도록 예외 처리.

3. **데이터/도메인 변경**

   - [x] 필요 시 기존 빌링키 기반 로직 제거 후 scheduleId 저장 로직을 검증.

4. **테스트**
