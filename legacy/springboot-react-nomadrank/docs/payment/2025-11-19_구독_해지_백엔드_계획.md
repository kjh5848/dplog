# 구독 해지 백엔드 업데이트 계획

환불 및 구독 변경 정책(2025-01-24)과 현재 구현 사이의 차이를 해소하기 위해 `cancelSubscription` 플로우에 필요한 개편 사항을 정리한다. 정책 핵심은 **해지 신청 후 결제된 기간 동안 서비스 이용 유지**, **부분 환불 없음**, **다음 결제부터 자동 청구 중단**이다.

---

## 1. 현재 구현 요약

- `PaymentServiceApiV1.cancelSubscription`
  - PortOne 예약(`payment-schedules`) 즉시 삭제
  - PortOne 빌링키 즉시 삭제
  - `SubscriptionEntity.status = CANCELLED`, `endDate = today`
  - 활성 `MembershipUser`를 EXPIRED로 전환 (즉시 혜택 제거)
- 결과: 해지 요청 즉시 이용 종료 → 정책 상의 “결제된 기간 동안 이용 가능”과 불일치.

---

## 2. 요구 정책 vs 구현 차이

| 항목 | 정책 요구 | 현재 구현 | 변경 필요 |
| --- | --- | --- | --- |
| 서비스 이용 기간 | 결제된 기간 종료까지 유지 | 즉시 종료 | ✅ |
| 환불/차액 | 없음 | 없음 | - |
| 다음 청구 | 자동 중단 | 예약/빌링키 즉시 삭제 | 예약은 즉시 삭제 OK<br>빌링키는 종료 시점까지 유지 고려 |
| 멤버십 상태 | 기 결제 기간동안 ACTIVE 유지 | 즉시 EXPIRED | ✅ |
| 해지 사유 로깅 | 필요 | 이미 reason 로그 존재 | - |

## 3. 개편 방향

### 3.1 상태 전환
- 해지 요청 시 구독/멤버십을 즉시 비활성화하지 않고 “취소 예약” 상태로 전환.
- 제안:
  - `SubscriptionStatus`에 `PENDING_CANCEL` 사용 또는 신규 플래그 추가.
  - `SubscriptionEntity.cancelScheduledAt` 같은 필드로 종료 예정일(`nextBillingDate`) 저장.
  - `MembershipUser`는 endDate를 기존 주기 종료일로 유지, `MembershipState.ACTIVATE` 유지.
- 정기 결제 스케줄러가 다음 결제일에 도달하면:
  - `SubscriptionStatus`를 `CANCELLED`로 변경.
  - `MembershipUser`를 EXPIRED로 전환.
  - PortOne 빌링키/예약 삭제 처리.

### 3.2 PortOne 연동
- 예약(`payment-schedules`)은 즉시 삭제해도 정책과 일치(다음 결제 차단).
- 빌링키 삭제는 다음 결제일 이후로 미뤄도 되지만 의무는 아님. 단, 즉시 삭제 시 재사용 승인(다운그레이드/재가입) 절차를 명확히 해야 함.
- 구현 선택:
  1. **즉시 삭제 유지**: 재구독 시 신규 빌링키 요구. 정책과 충돌 없음.
  2. **지연 삭제**: 해지 완료 시점(다음 결제일)까지 유지 후 삭제.

### 3.3 스케줄러/배치
- `SubscriptionServiceApiV1.processDueSubscriptions` 혹은 별도 배치에서 `PENDING_CANCEL` + `nextBillingDate == today` 조건을 확인해 최종 해지 처리.
- 처리 시 단계:
  1. PortOne 빌링키/예약 삭제 (선택한 플로우에 맞게).
  2. `SubscriptionEntity` 상태 `CANCELLED`, `endDate = today`.
  3. `MembershipUser` 상태 EXPIRED, 로그 기록.
  4. Sentry/알림 등 운영 로그 추가.

### 3.4 DTO/응답
- 해지 요청 API 응답에 다음 결제일까지 이용 가능하다는 메시지와 예정 종료일을 포함.
- 예: `{ status: "PENDING_CANCEL", effectiveDate: subscription.getNextBillingDate(), message: "다음 결제일부터 자동 해지됩니다." }`

### 3.5 UI/CS 연동 고려
- 사용자 화면에서 해지 상태를 “다음 결제일까지 이용 가능”으로 표시.
- CS 도구에 종료 예정일, 해지 사유 로그 표시.

## 4. 구현 체크리스트

1. `SubscriptionEntity`에 해지 예약을 나타낼 필드 추가 (`cancelScheduledAt` 등) 또는 `status`/`nextBillingDate` 활용.
2. `cancelSubscription` 로직 변경:
   - 예약 삭제는 유지.
   - 구독 상태는 `PENDING_CANCEL`로 두고 `endDate`/`membership endDate`는 건드리지 않음.
   - 해지 사유 로그/analytics 기록.
3. 스케줄러/배치:
   - 다음 결제일 도달 시 최종 해지 처리(batch or existing scheduler).
   - PortOne 빌링키 삭제 시점을 정책에 맞춰 결정.
4. 응답 DTO/테스트 업데이트:
   - 해지 응답 메시지/상태 변경.
   - 단위/통합 테스트에서 해지 이후에도 멤버십이 유지되는지 검증.
5. 문서/운영:
   - 개발 문서(본 문서)와 정책 문서를 동기화.
   - 프런트/CS 안내 문구 업데이트.
