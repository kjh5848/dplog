# PortOne 빌링키 구독 해지 백엔드 구현 계획

프런트엔드 UX/요건 문서(“PortOne 구독 해지 프런트엔드 구현 계획”)를 기반으로 백엔드 측에서 준비해야 할 작업을 정리한다. App Route → Spring 서비스 → PortOne API 호출까지 일관된 계약을 제공하고, 해지 사유 필수 수집 요구사항과 예약 결제 삭제 의무를 지원하는 것이 목표다.

## 1. 주요 계약 정리
- **App Route HTTP 계약**
  - `DELETE /api/subscriptions/{subscriptionId}/cancel`
  - Request Body: `{ reasonCode: string, reasonDetail?: string }`
    - `reasonCode`는 프런트 UX 킥오프에서 확정된 코드 값(`PRICE`, `USAGE_LOW`, `ETC` 등) 중 하나로 필수.
    - `reasonDetail`은 `reasonCode === "ETC"`일 때 200자 이내 자유 입력.
  - Response Body: `{ status: "canceled", canceledAt: ISO8601, scheduleRemoved: boolean }`
  - 상태 코드 매핑: 200(성공), 404(이미 해지/미존재), 409(예약 결제 미삭제), 422(사유 누락/검증 실패), 5xx(PortOne 장애/내부 오류).

- **Spring REST 계약**
  - `DELETE /v1/payments/subscriptions/{subscriptionId}/cancel`
  - Request Body: `{ reasonCode: string, reasonDetail?: string }`
  - Response Body: `{ status: "CANCELED", canceledAt: ISO8601, scheduleRemoved: boolean }`
  - 세션/토큰을 통해 사용자 식별 후 `PaymentServiceApiV1.cancelSubscription(userId, subscriptionId, payload)` 호출.

## 2. 도메인/데이터 요구사항
- `SubscriptionEntity`
  - `status`를 `CANCELED`로 전환하고 `endDate`, `updatedAt`을 해지 시점으로 저장.
  - 예약 스케줄 삭제 여부(`scheduleRemoved`)를 저장할 필드 추가 검토(예: `lastScheduleId`, `scheduleClearedAt`), 혹은 별도 로그 테이블.
- `BillingKeyEntity`
  - `status`를 `DELETED`로 업데이트.
  - 감사 로그(`MembershipUserLogEntity`)에 해지 사유 코드/자유 입력을 남긴다.
- 사유 수집
  - 신규 테이블 `subscription_cancel_reason`(가칭) 또는 기존 로그에 JSON payload로 저장. 필드: `subscriptionId`, `reasonCode`, `reasonDetail`, `recordedAt`.
- 예약 데이터
  - `SubscriptionEntity.subscriptionId`가 PortOne 예약 ID와 동일하므로 이를 사용해 PortOne 스케줄 조회/삭제 연동.

## 3. 세부 작업 항목

### 3.1 App Route & BFF 계층
- [ ] `src/app/api/subscriptions/[subscriptionId]/cancel/route.ts` 생성: 세션 검증 → 백엔드 REST 호출.
- [ ] `src/lib/services/portoneService.ts` 또는 신규 서버 전용 유틸에 `cancelSubscription({ subscriptionId, reasonCode, reasonDetail })` 추가.
- [ ] 입력 검증: `reasonCode` 필수, `ETC`일 때 `reasonDetail` 길이(≤200) 확인.
- [ ] 응답 매핑: 백엔드 200/404/409/422/5xx를 프런트 문서의 토스트/안내 메시지와 맞추어 반환.

### 3.2 Spring Controller & DTO
- [ ] `PaymentControllerApiV1`에 `@DeleteMapping("/subscriptions/{subscriptionId}/cancel")` 추가.
- [ ] Request DTO: `SubscriptionCancelRequest`(`reasonCode`, `reasonDetail`), Validation Annotation으로 필수/길이/패턴 검증.
- [ ] Response DTO: `SubscriptionCancelResponse`(`status`, `canceledAt`, `scheduleRemoved`).
- [ ] 세션 기반 사용자 식별 후 `PaymentServiceApiV1.cancelSubscription(...)` 호출.

### 3.3 PaymentServiceApiV1 확장
- 신규 메서드 `cancelSubscription(Long userId, String subscriptionId, SubscriptionCancelRequest request)` 구현:
  1. 구독 존재 및 소유자 검증 (`SubscriptionRepository.findById`).
  2. 이미 `CANCELED` 또는 `EXPIRED` 상태면 404 반환.
  3. 빌링키 조회 (`BillingKeyRepository.findByIssueIdAndUserId`), PortOne 호출용 실제 `billingKey` 복호화.
  4. PortOne 스케줄 단건 조회/삭제 (`PortOneV2Client.getPaymentSchedule`, `.deletePaymentSchedule`).
  5. PortOne 빌링키 삭제 (`PortOneV2Client.deleteBillingKey`), PG 별 필요 파라미터(`storeId`, `reason`) 포함.
  6. 내부 상태 갱신: `SubscriptionEntity`/`BillingKeyEntity` 업데이트, 사유 로그 저장, `MembershipUser` 상태 조정.
  7. 응답 DTO 조립해 반환.
- 에러 매핑:
  - 스케줄 삭제 실패 → `PaymentScheduleConflictException`(→ HTTP 409)
  - 빌링키 미존재 → 404
  - PortOne 5xx → `ExternalSystemException`(→ HTTP 502)
  - 검증 실패 → `BadRequestException`(→ HTTP 422)
- 트랜잭션: 스케줄 삭제/빌링키 삭제는 외부 호출이므로 내부 DB 업데이트 전후 트랜잭션 경계(`@Transactional`) 재검토. 외부 호출 실패 시 롤백 보장.

## 5. 리스크 및 대응
- **사유 필수 정책 변경 가능성** → Request DTO에서 Validation 완화/강화가 필요할 수 있으므로 Feature Flag 또는 Config 기반 검증 적용.
- **포트원 API 제한** → 스케줄 삭제/빌링키 삭제 API 권한 확인, QA 환경에서 선 검증.
- **다중 상점 지원** → `storeId`를 요청마다 정확히 전달하도록 PortOne Config 재확인, 채널 매핑 테이블 유지.
- **레거시 구독 데이터** → 기존 PENDING/ACTIVE 구독 중 예약 ID가 없는 케이스에 대한 수동 정리 가이드 마련.
