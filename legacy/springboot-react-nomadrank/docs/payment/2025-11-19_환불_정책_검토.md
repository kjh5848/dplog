# 환불 및 구독 변경 정책 검토 메모

본 문서는 2025-01-24 버전의 “환불 및 구독 변경 정책” 문서를 백엔드/프런트 구현 상태와 비교하여 확인한 차이점과 조정이 필요한 항목을 단계별로 정리한 자료입니다.

---

## 1. 검토 단계별 요약

### 1.1 업그레이드 항목 점검
1. **조건**: 상위 플랜 또는 월→연간 전환만 허용 → 현재 `SubscriptionServiceApiV1.upgradeSubscription` 구현과 일치.
2. **차액 계산**: “잔여 가치와 상위 플랜 가격 비교 후 차액 결제” → `UpgradeCalculator`의 Hybrid 모델과 일치.
3. **즉시 적용/결제 실패 처리**: 코드에서 결제 후 즉시 혜택 적용, 실패 시 예외 처리로 기존 플랜 유지 → 정책과 일치.

➡️ 업그레이드 문구는 현 구현과 충돌 없음.

### 1.2 다운그레이드 항목 점검
1. **정의**: 하위 요금제 변경 시 “선불 결제, 부분 환불 없음” 명시.
2. **절차**: 문서에는 UI상 “멤버십 → 결제” 메뉴에서 버튼으로 신청 시 다음 결제일부터 자동 적용이라고 기재.
3. **구현 상태**:
   - 백엔드에는 다운그레이드 API/예약/스케줄러가 존재하지 않음 (`SubscriptionServiceApiV1`는 업그레이드·해지·정기결제만 구현).
   - UI에도 다운그레이드 버튼 및 플로우 미구현.
4. **결론**: 현재 정책 문구는 **미구현 기능을 안내**하고 있음.

➡️ 조치 필요: “다운그레이드는 해지 후 재가입 또는 고객센터 문의” 등 기존 정책(자동 미지원)으로 되돌리거나, 문서에 “기능 준비 중” 문구를 명시해야 함.

### 1.3 환불(구독 해지) 항목 점검
1. **정책**: “해지 후에도 결제된 기간 동안 이용 가능, 부분 환불 제공하지 않음”.
2. **코드**: `PaymentServiceApiV1.cancelSubscription`은 해지 즉시
   - `SubscriptionEntity.endDate = LocalDate.now()`
   - `MembershipUser` 상태를 `EXPIRED`로 변경
   - 빌링키/예약 삭제
3. **문제**: 위 구현은 **해지 즉시 사용 권한을 종료**시키므로, “기결제 기간 동안 이용 가능” 정책과 충돌.
4. **예외 환불**: 서비스 장애/결제 오류에 대한 자동 환불 로직 미구현 → CS 수동 처리만 가능.

➡️ 조치 필요: 
   - 정책을 현재 구현(즉시 이용 종료) 기준으로 수정하거나,
   - 구현을 변경하여 종료일·권한을 다음 결제일까지 유지하도록 보완.

### 1.4 기술 장애·서비스 종료 시 환불
- 문서는 “최근 3개월 내 결제 건은 전액 환불 가능”이라고 안내.
- 자동 환불/정산 로직 없음 → 운영/CS 수동 프로세스 마련 필요.

## 2. 수정/보완 권장 사항

| 우선순위 | 영역 | 권장 조치 |
| --- | --- | --- |
| 높음 | 환불 정책(3장) | 해지 시 이용 기간 유지 여부를 구현과 맞추어 재정의하거나, 코드 수정으로 `endDate`를 기존 `nextBillingDate`로 설정하는 등 정책과 구현 정합성 확보 |
| 높음 | 다운그레이드 정책(2장) | 현재 기능 미구현 사실 명시 및 “해지 후 재가입/고객센터 문의”로 안내 (향후 기능 출시 시 업데이트) |
| 중간 | 예외 환불(3.3, 4장) | CS/재무 절차 문서화 및 PortOne 결제 취소/부분 환불 자동화를 로드맵에 포함 |
| 낮음 | 문서 메타 | 최종 업데이트일은 실제 변경 시점에 맞춰 갱신, UI 경로는 구현 확정 후 반영 |

## 3. 다음 단계 제안
1. **정책 담당자와 정합성 점검 미팅**: 환불/다운그레이드 정책을 실제 제공 가능한 수준으로 확정.
2. **문서 업데이트**: 정책 수정 후 `docs/payment/subscription-plan-and-refund-policy.md` 및 사용자 공지(FAQ/앱 내 안내) 동기화.
3. **개발 범위 정의**: 다운그레이드 자동화·환불 자동화 개발 여부를 로드맵에 반영.
4. **운영 가이드 정리**: 예외 환불, 기술 장애 대응 절차를 CS·재무팀과 공유.
