## 1. 목표 (Phase 1 우선)
1. **표준 사용량 응답 스키마**  
   - 한도 초과 시 모든 API가 동일한 JSON 구조(`limitMeta`)와 코드(-9)를 반환하도록 한다.  
   - 프런트가 업그레이드 CTA, 캐시 만료 시점 등 메타 정보를 재사용할 수 있게 한다.

2. **(Phase 2) 서버 캐싱/차단 로직**  
   - 동일 사용자가 같은 기간(일간/월간) 동안 한도 초과 상태면, 서버 레이어에서 즉시 차단하고 캐시된 정보를 반환한다.  
   - Redis(or 다른 KV) TTL = 일간 → 다음 0시(KST), 월간 → 다음 월초.

3. **(Phase 2) 업그레이드/리셋 훅**  
   - 업그레이드/추가 구매/관리자 증설, 일간 0시, 월간 UsageReset 시 cache invalidate.  
   - invalidate API 또는 이벤트를 노출하여 프런트/백오피스에서 사용량 상태를 재초기화한다.

## 2. LimitMeta 표준화
### 2.1 응답 구조
```json
{
  "code": -9,
  "message": "5회 사용횟수를 모두 사용하였습니다.",
  "meta": {
    "usageLimit": {
      "usageType": "NPLACE_RANK_REALTIME",
      "used": 5,
      "limit": 5,
      "periodType": "DAILY",
      "periodKey": "2025-11-21",
      "resetAt": "2025-11-22T00:00:00+09:00",
      "recommendedPlanId": 11,
      "blocked": true
    }
  }
}
```

- [ ] `NplaceServiceApiV1`, `NplaceKeywordServiceApiV1`, 상점등록/트랙 자동화 등 모든 ServiceSort에 적용.

## 3. UsageLimitCache (Phase 2 백로그)
| 키 | 예시 값 | TTL | 용도 |
| --- | --- | --- | --- |
| `usage-limit:{memberId}:{usageType}:{periodKey}` | `{"used":5,"limit":5,"periodType":"DAILY","resetAt":"2025-11-22T00:00:00+09:00"}` | 일간: 다음 0시, 월간: 다음 월초 | 한도 초과 상태 캐시 |

### 3.1 구현 단계 (Phase 2 시점)
1. Redis(or in-memory) 추상화 `UsageLimitCache`.  
2. 한도 초과 시 cache SET + TTL 설정.  
3. 요청 진입 시 cache HIT → 즉시 LimitMeta 반환(HTTP 200, `blocked:true`).  
4. invalidate 메서드: `delete(memberId, usageType, periodKey)` / `deleteAll(memberId)`.

---

## 4. Invalidate & Reset 훅 (Phase 2 백로그)
| 이벤트 | 동작 |
| --- | --- |
| 업그레이드/추가 구매 | 결제 성공 후 `UsageLimitCache.deleteAll(memberId)` 호출. |
| 관리자 증설 | 동일. |
| 일간 0시 cron | 일간 캐시 키 삭제. |
| 월간 UsageResetService | 월간 서비스 sort만 삭제 (이미 구현됨). |

- [ ] `/v1/membership/usage/invalidate` API (관리자/시스템용).  
- [ ] PortOne 웹훅 후 `UsageLimitCache.deleteAll` 연동.

## 5. 프런트 통신 요구사항 (서버 관점)
- API 응답에는 항상 `limitMeta.periodType`, `periodKey`, `resetAt`, `recommendedPlanId`, `usageType` 포함.  
- `blocked:true` → 프런트는 재요청 없이 캐시 메시지를 보여준다.  
- 업그레이드 후 `/v1/membership/current` 응답에 `usageLimitResetToken` 같은 식별자를 넣어 프런트 store 초기화를 유도할 수도 있음.
