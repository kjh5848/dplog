# 멤버십 프로필 요약 API 백엔드 계획

## 1. 목표 및 개요
- 단일 엔드포인트 `GET /v1/membership/profile-summary` 에서 현재 멤버십, 최근 이력, 멤버십 카탈로그 비교 정보, 사용량을 한 번에 반환한다.
- 커서 기반 페이징으로 멤버십 이력을 무한 스크롤 UX에 맞춰 전달한다.
- 본인 사용자와 관리자(백오피스) 요청을 모두 지원하며, 동일 응답을 다른 도메인에서도 재사용할 수 있도록 DTO를 모듈화한다.
- **Usage 정의 합의 필요**: 포함할 지표(예: 트랙 슬롯, 보고서 생성, 실시간 조회, 알림 발송 등), 단위(건/분/회 등)와 소수점 처리 규칙(정수 반올림/올림/내림)을 정의 단계에서 명시하여 이후 혼선을 방지한다.
- **ServiceSort ↔ usage 키 매핑 및 한도/사용량 소스 확정**
  - enum에 `usageKey` 내장(권장) 또는 설정 기반 매핑을 사용해, 문자열 키와 ServiceSort의 단일 출처를 확보한다.
  - `MembershipDetailEntity`에서 서비스별 `limit_count`를 읽어와 limit 맵 구성 (`null` → 무제한).
  - `UseLogRepository` 등 사용 이력을 기준으로 사용자·ServiceSort·기간별 `used` 값을 집계한다(기본은 당월 범위).

### Usage / Metrics 명세 합의안
| 지표 key | 설명 | 단위 | 타입 | 소수 처리 |
| --- | --- | --- | --- | --- |
| `trackKeywords` | 트랙 슬롯 사용 건수 | 건 | `int` | 소수 없음 (정수 저장) |
| `reportExports` | 보고서 내보내기 횟수 | 회 | `int` | 소수 없음 (정수 저장) |
| `realtimeQueries` | 실시간 조회 호출 수 | 건 | `int` | 소수 없음 (정수 저장) |
| *확장 시 예: `alertMessages`* | 알림 발송 건수 | 건 | `int` | 소수 없음 |

- 각 항목은 `{ used: int, limit: int }` 구조 유지.
- 추후 새로운 지표 추가 시 key 이름과 단위, 타입을 표 형태로 문서에 추가 후 동기화한다.
- **전략**: `ServiceSort` 열거형과 요금제별 한도 정보를 매핑하고, `MembershipUsageService`가 각 `ServiceSort`의 사용량 집계를 `usage` 블록으로 변환할 수 있도록 확장 계획을 세운다.
  1. `ServiceSort` → usage key 매핑 표 작성 (`NPLACE_RANK_TRACK` → `trackKeywords` 등).
  2. 요금제별 한도 소스를 확인하고 `MembershipEntity` 또는 별도 설정에서 limit 값을 주입.
  3. 사용량 집계 데이터 소스(로그/카운트 테이블) 연결 후 `MembershipUsageService`에서 {used, limit} 계산.
  4. 신규 지표 추가 시, 위 표와 프론트 타입 정의 모두 갱신.

## 3. Controller 설계
```java
@GetMapping("/profile-summary")
public HttpEntity<?> getMembershipProfileSummary(
        HttpSession session,
        @RequestParam(required = false, name = "userId") Long userId,
        @RequestParam(required = false, name = "historyLimit", defaultValue = "10") int historyLimit,
        @RequestParam(required = false, name = "historyCursor") String historyCursor)
```
- 세션 기반 인증 후 권한 검증 (`requesterAuth.contains(ADMIN)` 또는 본인 여부).
- `historyLimit` 상한 (예: 50) 적용.
- Service 호출 결과를 `ResDTO` wrapper로 반환.

## 4. Service 로직
`MembershipServiceApiV1#getMembershipProfileSummary(...)`
1. **대상 사용자 조회**: `userRepository.findByIdAndDeleteDateIsNull`.
2. **현재 멤버십**: `membershipUserRepository.findActiveMembershipByUserAndDate`.
3. **히스토리 커서 파싱**: `"2025-10-23T00:00:00Z#11"` → `LocalDateTime`, `membershipId` 분리.
4. **히스토리 로드**: `MembershipUserRepository` 커서 쿼리 → `Page<MembershipUserEntity>` 변환 → DTO 매핑.
5. **카탈로그**: `membershipRepository.findAllActiveMembershipsSortedByOrder()` → 필요한 필드만 요약.
6. **Usage**: `membershipUsageService.getUsageSnapshots(userId)` 호출.
7. **Response 구성**: 위 요소를 `ResMembershipProfileSummaryDTOApiV1`로 조합.

## 5. Repository 커서 쿼리 규칙
- 정렬: `endDate DESC, id DESC`.
- 커서 포맷: `{endDate ISO8601}#{membershipUserId}`.
- SQL 조건 예:
  ```sql
  WHERE user_id = :userId
    AND (end_date < :cursorEndDate
         OR (end_date = :cursorEndDate AND id < :cursorId))
  ORDER BY end_date DESC, id DESC
  LIMIT :historyLimit + 1
  ```
- `LIMIT + 1` 방식으로 `hasNextPage` 판단, 초과분으로 `nextCursor` 생성.

## 7. 권한 및 예외
- **401**: 세션/토큰 없음.
- **403**: 타인 정보 요청 시 권한 부족.
- **404**: 사용자 또는 멤버십 데이터 없음.
- **Cursor 오류**: 잘못된 포맷 → `BadRequestException`.

## 8. QA 및 문서화
- Postman 컬렉션 업데이트: 새로운 엔드포인트 추가.
- 통합테스트: 본인/관리자/페이지 끝 커서/빈 이력 케이스.
- 경계 테스트 시나리오:
  - `historyLimit=1` 또는 최대치(예: 50) 요청 시 정상 작동 여부.
  - 커서 미지정 첫 요청 → `nextCursor` 생성 확인.
  - 마지막 페이지에서 `hasNextPage=false`, `nextCursor=null`.
  - 잘못된 커서 포맷 전달 시 400 반환.
  - 사용량이 빈/null일 때 기본값(`used=0`, `limit=0` 또는 null 허용 여부) 검증.
- API 문서 (`docs/membership/...`) 업데이트, 프론트와 공유.

## 9. 후속 작업 체크리스트
- `/v1/membership/current`, `/v1/user/{id}/membership` 호출 레퍼런스 조사 및 단계적 대체.
- 사용량 통계 수집 로직 재사용 범위 검토 (추후 다른 도메인 제공 여부).
- 모니터링(응답 시간, 에러율) 대시보드 항목 추가.
