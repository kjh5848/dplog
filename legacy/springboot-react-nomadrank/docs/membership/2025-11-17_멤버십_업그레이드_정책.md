# 멤버십 업그레이드 정책 (2025-11-17)

본 문서는 `/v1/subscription/upgrade` 플로우(구현: `SubscriptionServiceApiV1.upgradeSubscription`)와 `UpgradeCalculator`/`AbstractSubscriptionCalculator`가 만들어 내는 과금 규칙을 기준으로 **플랜 업그레이드**와 **기간 업그레이드(월→연)** 시의 정책, 결제 흐름, 사용량 초기화를 명확히 정의한다.

---

## 1. 업그레이드 유형

| 유형 | 설명 | 코드 진입점 |
| --- | --- | --- |
| 플랜 업그레이드 | FREE → 실속, 실속 → 성장처럼 등급이 바뀜 | `SubscriptionServiceApiV1.applyUpgradeWith/WithoutPayment` |
| 기간 업그레이드 | 동일 플랜에서 월간 → 연간 변경 | `UpgradeCalculator.calculateMonthlyToYearly` 결과를 사용하는 `/upgrade` |

`UpgradeCalculator`는 `AbstractSubscriptionCalculator`를 상속해 잔여 일수, 차액 산출 등을 담당하며, 월/연 변환 시에도 동일한 프라메틱 계산(Proration)을 활용한다.

---

## 2. 플랜 업그레이드 (등급 상승)

### 2-1. 정책

1. **사용량 즉시 초기화**  
   - FREE 포함 하위 플랜에서 상위 플랜으로 이동하면 `UseLog` 기반 월간 사용량을 0으로 초기화한다.  
   - 구현 포인트: 업그레이드 성공 직후 `MembershipUsageService` 혹은 별도 리셋 컴포넌트에서 `ServiceSort` 별 누적치를 삭제/리셋하고 `MembershipUserLog`에 `UPGRADE_RESET` 이벤트 기록.

2. **상위 플랜 한도 즉시 적용**  
   - `MembershipDetail` 한도값을 새 멤버십 기준으로 다시 로딩해 프런트에 반영.  
   - 월 초/말과 상관없이 즉시 새 한도가 노출되어야 CS 이슈가 없다.

3. **상위 기능·권한 즉시 활성화**  
   - `SubscriptionEntity` 의 `membershipLevel`, `membershipName`, `amount`, `billingCycle` 업데이트가 끝나면 프런트/백엔드 가드 로직에서도 상위 권한을 사용한다.

4. **제한 로직 해제**  
   - FREE 팝업/쿼터 제한을 프런트에서 즉시 제거. 백엔드는 한도 초기화가 이뤄졌으므로 동일 `ServiceSort`에서 AuthenticationException이 더는 발생하지 않는다.

5. **로그 남김**  
   - `MembershipUserLogEntity` 에 기존/신규 플랜, 초기화 전 사용량, 초기화 시각을 `UPGRADE_RESET` 으로 남긴다. (향후 감사/CS 대응용)

### 2-2. 예시

| 시나리오 | 업그레이드 전 | 업그레이드 후 |
| --- | --- | --- |
| FREE → 실속 | 월 30회 모두 사용 | 사용량 0으로 초기화, 실속 150회 한도 즉시 부여 |
| 실속 → 성장 | 150회 중 120회 사용 | 사용량 0으로 초기화, 성장 무제한 사용 및 고급 기능 활성화 |

## 3. 기간 업그레이드 (월간 → 연간)

`UpgradeCalculator.calculateMonthlyToYearly` 로 계산된 잔여 가치/차액을 기반으로 결제가 진행되며, `AbstractSubscriptionCalculator` 의 `resolveCycleDays` 로 월/연 일수를 통일한다.

### 3-1. 핵심 규칙

1. **연간 금액 즉시 결제** (예: 189,000원)  
2. **연간 기간 시작 시점**: 기존 월간 `nextBillingDate` 의 다음 날부터 1년.  
3. **혜택 적용**: 동일 플랜이므로 기능 차이는 없지만 연간 전용 혜택이 있다면 즉시 부여.  
4. **사용량 초기화 없음**: 월간 플랜의 리셋 규칙(예: 매월 1일)을 유지.  
5. **기존 월간 잔여 기간 보존**.

### 3-2. 상세

- 결제: `/upgrade` 진입 시 `UpgradeCalculator`가 잔여 가치를 산출, 즉시 청구.
- 기간: `SubscriptionEntity.nextBillingDate` 를 기준으로 `+1년`.
- 혜택: 연간 전용 프로모션/쿠폰이 있다면 즉시 활성화.
- 사용량: FREE→실속과 달리 **초기화하지 않음**. `UseLog`는 기존 주기에 따라 다음 월초에 리셋.

### 3-3. 예시

| 항목 | 값 |
| --- | --- |
| 월간 주기 | 11/01 → 12/01 |
| 업그레이드일 | 11/20 |
| 연간 가격 | 189,000원 즉시 결제 |
| 사용량 | 70/150 유지 (12/01 리셋) |
| 연간 기간 | 12/01 → 다음 해 12/01 |

### 3-4. 계산 로직 요약 (UpgradeCalculator)

- `/v1/subscription/upgrade` 는 항상 `UpgradeCalculator.calculate(...)` 를 통해 차액을 산출한다. 이 계산기는 세 가지 흐름만 허용한다.
  1. **동일 주기 업그레이드** (월→월, 연→연): `calculateSameCycle` 로 잔여 일수(`remainingDays`)와 주기 일수(`cycleDays`)를 기반으로 현재/목표 플랜 가격을 프레이션 계산한다. `differenceAmount = max(prorate(target) - prorate(current), 0)` 으로 계산되며 반올림 후 즉시 청구된다.
  2. **월간 → 연간 업그레이드**: `calculateMonthlyToYearly` 경로를 타며, 월간 잔여 가치를 차감한 뒤 연간 가격 전체를 청구한다. `nextBillingDate` 는 `today + 1년` 으로 재산출한다.
  3. **그 외 주기 전환** (예: 연간 → 월간)은 명시적으로 `BadRequestException("지원하지 않는 결제 주기 전환입니다.")` 를 던지므로 **불가**하다. 프런트/CS 가 유저 안내 시 “연→월 전환 업그레이드는 제공되지 않음”을 고지해야 한다.
- 위 로직 때문에 “연간 플랜 사용자가 월간 플랜으로 업그레이드” 같은 조합은 서버에서 차단되며, 정책상 허용하려면 `UpgradeCalculator` 와 `SubscriptionServiceApiV1` 전제 조건을 함께 변경해야 한다.

### 3-5. 연간 → 월간 상위 플랜 전환 (부분 환불 + 즉시 재결제)

> **정책 업데이트**: 연간 플랜 이용자가 상위 월간 플랜으로 즉시 전환하려는 경우, 잔여 가치를 환불한 뒤 월간 금액을 결제하는 경로를 제공한다. 프런트는 환불/결제 동의 문구를 노출해야 한다.

#### 3-5-1. 예시 (2개월 사용 후 월간 업그레이드)

| 항목 | 값 |
| --- | --- |
| 기존 플랜 | 실속 연간 (189,000원, 12개월) |
| 사용 기간 | 2개월 사용, 10개월 잔여 (60일 가정) |
| 잔여 가치 | `189,000 × (10/12) ≈ 157,500원` → PortOne 즉시 환불 |
| 신규 플랜 | 성장 월간 39,000원 |
| 결제 흐름 | (1) 환불 성공 → (2) 월간 39,000원 즉시 결제 |
| 결과 | 성장 월간 권한 즉시 활성화, 다음 결제일 = 결제일 + 1개월 |

#### 3-5-2. 구현 가이드

1. **계산기 확장**: `UpgradeCalculator` 에 `calculateYearlyToMonthly` 를 추가하고 `Result` 에 `refundableAmount` 등을 포함한다.
2. **업그레이드 서비스 흐름**: `SubscriptionServiceApiV1.upgradeSubscription` 이 연간→월간 상향을 허용하도록 조건을 완화하고, `applyUpgradeWithPayment` 내에서 환불 → 월간 결제 순으로 진행한다. 환불 실패 시 전체 업그레이드를 중단하고 기존 연간 플랜을 유지한다.
3. **사용량 및 로그**: 상향이므로 `UsageResetService.resetUsage` 실행, `MembershipUserLog` 에 `UPGRADE_RESET` + `FROM_YEARLY_TO_MONTHLY` 이벤트 기록.
4. **스케줄/구독 엔티티**: 기존 연간 예약을 삭제하고 월간 예약을 재생성, `nextBillingDate`/`startDate` 를 월간 기준으로 업데이트한다.
5. **프런트/CS 안내**: “잔여 기간 약 XX원이 환불되고, 성장 월간 39,000원이 즉시 결제됩니다” 안내를 표준화한다.

## 5. 구현 체크리스트

1. `/v1/subscription/upgrade` 성공 후 `UseLogRepository`를 통해 사용자 월간 사용량 삭제 또는 별도 테이블 리셋.  
2. `MembershipUserLog`에 `UPGRADE_RESET` 기록 (기존/신규 플랜, 초기화 전 사용량, 처리자).  
3. 프런트는 업그레이드 성공 응답 시 새 한도/기능을 즉시 반영.  
4. 기간 업그레이드는 **초기화 로직을 건너뛰도록 조건 분기** (호출 시 `BillingCycle` 비교). 다만 연→월 상향 전환은 플랜 업그레이드로 간주해 초기화를 수행한다.  
5. `UpgradeCalculator`/`AbstractSubscriptionCalculator` 기반 로직을 유지하여 차액 산출 일관성 확보. 연→월 케이스도 동일 계산기를 통해 환불 금액을 산출할 수 있도록 확장한다.  
6. `/v1/membership/profile-summary` 응답(`MembershipControllerApiV1` → `MembershipServiceApiV1.getMembershipProfileSummary`)은 `MembershipUsageService.buildUsage` 결과를 그대로 사용하므로, 업그레이드 직후 리셋된 사용량이 즉시 노출되도록 해당 서비스가 최신 `UseLog` 상태를 조회할 수 있어야 한다.

## 6. 데이터 & 스케줄 정책

- `nextBillingDate`: 월→연 업그레이드 시 `current nextBillingDate + 1년`.  
- 연→월 상향 전환 시 기존 연간 `nextBillingDate` 를 폐기하고 월간 결제 성공 시점을 새로운 `startDate`/`nextBillingDate` 로 저장한다.  
- 사용량 리셋 & 카운터 설계:  
  - **주기 기준일**은 사용자 단위로 관리한다. (`SubscriptionEntity.startDate` 또는 `nextBillingDate` 기반)  
  - **플랜 업그레이드** → 업그레이드 성공 시점을 새 주기 시작일로 저장하고 사용량 카운터를 0으로 초기화한다.  
  - **정기 리셋(월/연)** → 결제일을 기준으로 다음 리셋일을 계산해 카운터를 초기화한다. (예: 11/17 결제 → 12/17에 월간 리셋)  
  - `UseLog`는 감사/세부 이력 용도로 유지하되, 실시간 한도 체크는 `MembershipUser` 내 카운터 필드를 참조한다.  
  - 리셋/업그레이드 이벤트는 `MembershipUserLog`에 `RESET_BY_BILLING_CYCLE`, `UPGRADE_RESET` 등으로 기록.  
- `MembershipUsageService.buildUsage` 동작:  
  - `MembershipDetailRepository`로 활성 멤버십 한도를 조회한 뒤, `UseLogRepository.countByUserEntityAndServiceSortAndCreateDateBetween`으로 서비스별 사용량을 누적한다.  
  - 기준 기간은 시스템 타임존 기준 `YearMonth.now(zoneId)`의 1일 00시부터 다음 달 1일 00시까지다.  
  - 따라서 `getMembershipProfileSummary` API의 `usage` 블록은 **현재 달** 사용량을 반환하며, 업그레이드 리셋이 발생하면 해당 월 구간의 `UseLog` 삭제/리셋 결과가 그대로 반영된다.
- Webhook: PortOne 성공 웹훅(`PaymentWebhookService`)에서 연간 플랜 스케줄 재계산, `SubscriptionEntity.status`를 ACTIVE로 유지.

## 7. 후속 과제

- `SubscriptionServiceApiV1.applyUpgradeWith/WithoutPayment` 내에서 사용량 초기화를 호출하는 헬퍼(`UsageResetService`) 추가.  
- FREE 사용자가 업그레이드할 때 프런트에서 남은 호출 수 대신 “업그레이드 시 즉시 초기화” 메시지 제공.  
- `MembershipUsageService`/`UseLog` 리셋 작업에 대한 트랜잭션 경계 정의(업그레이드 성공 시에만 커밋).  
- PortOne 결제/환불 트랜잭션 보강: 결제 성공 후 후속 처리 실패 시 PG 취소 재시도 및 관리자 알림 경로 구성, 환불 실패 시 재시도 정책 수립.
- 연→월 환불 후 월간 결제가 실패했을 때 롤백 또는 재결제 유도 플로우 정의(예: 환불 취소 시도, 사용자 알림 및 재시도 엔드포인트).
- `UpgradeCalculator` 결과를 이용한 preview API(`/v1/subscription/upgrade?preview=true` or `/quote`) 구현으로 프런트에 차액/환불 정보를 사전 제공. 
- `processYearlyToMonthlyRefund` 로그 확장: 환불-월간 결제 연계 식별자 기록, 월간 금액 0원 시에도 업그레이드가 정상 완료되는지 확인.
- PortOne API 호출 실패에 대한 리트라이/서킷브레이커 전략 수립 및 모니터링 이벤트 정의.
- 통합 테스트 추가: 연→월 환불 후 월간 결제 성공/실패 케이스, 결제 성공 후 DB 롤백 케이스, preview 모드 응답 검증 등.

### 7-1. 구현 난이도/우선순위 표

| 우선순위 | 난이도 | 항목 | 완료 여부 | 필요한 경우의 수/비고 |
| --- | --- | --- | --- | --- |
| 1 | 중간 | 사용량 초기화 헬퍼 통합 (`UsageResetService`) | ✅ (기본 로직 완료) | 플랜 업그레이드, 연→월 전환 모두 적용 확인 |
| 2 | 하 | FREE 업그레이드 안내 문구 | ⏳ (UI 반영 필요) | 프런트 팝업/모달 메시지 |
| 3 | 중간 | 트랜잭션 경계 개선 (`MembershipUsageService`) | ⏳ | 업그레이드 성공 시에만 커밋되는지 테스트 |
| 4 | 높음 | PortOne 결제/환불 재시도 & 알림 | ❌ | 결제 성공 후 후속 작업 실패, 환불 실패, 네트워크 오류 |
