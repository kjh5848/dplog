# Nplace Reply – Encryption & Key Rotation Guide

아주 초보자도 “왜 암호화가 필요한지 + 왜 키 회전이 들어가는지”를 한 번에 이해할 수 있도록, 실제 운영 환경에서 벌어지는 상황을 기준으로 설명합니다. 보안 실무에서 왜 이런 설계를 했는지도 함께 정리했습니다.

---

## 1. 암호화(Encryption)가 왜 필요한가?
- 암호화는 “PW를 숨긴다”가 아니라 **데이터가 유출돼도 원문이 드러나지 않게 하는 마지막 방어선**입니다.
- 네이버 ID/PW는 외부 서비스에 로그인하기 위한 고위험 자격증명입니다. 평문 저장은 법·정책상 금지이며, 사고 시 바로 개인정보 유출 사고로 분류됩니다.
- 따라서 DB에 저장하기 전 자동 암호화, DB에서 읽을 때 자동 복호화 흐름이 반드시 필요합니다.

---

## 2. 왜 AES 기반 “양방향 암호화”인가?
✔ **이유: Pumpingstore가 진짜 PW를 필요로 하기 때문입니다.**

댓글을 ON/OFF하려면 다음과 같은 요청이 Pumpingstore로 전달됩니다:
```json
POST /seller-nvid
{
  "naverId": "user123",
  "naverPw": "실제 비밀번호"
}
```
Pumpingstore는 네이버에 로그인해야 하므로 **복호화된 실제 PW**를 써야 합니다.  
→ 따라서 **복호화 가능한 AES(TextEncryptor)**가 필요하고, `BCrypt` 같은 단방향 해시는 사용할 수 없습니다.

---

## 3. 왜 전용 암호화 로직 + AttributeConverter인가?
일반적인 민감 정보 암호화 방법은 두 가지가 있습니다.

| 방법 | 단점/장점 |
| --- | --- |
| A) 서비스/컨트롤러에서 직접 암호화 | 코드 중복, 실수 가능성 높음, 복호화 로직이 여기저기 흩어짐 |
| B) **JPA AttributeConverter에서 자동 암·복호화** | 엔티티에 값을 넣으면 자동 암호화, 가져오면 자동 복호화, 비즈니스 로직과 보안 로직 분리 |

현재 프로젝트는 B 방식을 선택했습니다.  
→ 개발자는 평소처럼 `entity.setNaverPw()`만 호출하면 내부적으로 암호화가 끝납니다.

---

## 4. 왜 “키 회전(Key Rotation)”이 필수인가?
### 키는 영원히 안전하지 않다
- 운영팀 내부 유출, GitHub 노출, 퇴사, 서버 침해 등 다양한 시나리오가 존재합니다.
- 한 번 노출된 키로는 DB의 모든 암호문을 복호화할 수 있습니다.

### 키 회전이 없으면?
- 예: `key_v1`로 암호화된 PW가 수천 건인데 키가 유출되면?
  - 모든 PW 복호화 가능
  - 네이버 계정 탈취 → 서비스 중단 → 법적 처벌 리스크
- 해결책: 키를 즉시 바꾸고, 새 키로 다시 암호화해야 합니다.

---

## 5. 이 프로젝트의 키 회전 설계
핵심 포인트:
1. **암호문 형식**: `v2:암호문`  
   → 앞의 `v2`가 “어떤 키로 암호화했는지”를 나타냅니다.
2. **`EncryptionProperties`에서 다중 키 관리**  
   - `active-key`: 지금 새 데이터에 사용할 키  
   - `keys.v1`, `keys.v2`, `keys.v3`: 과거부터 현재까지 모든 키
3. **새 키 배포 흐름**  
   - 새 키(v3)를 등록하고 `active-key`로 지정  
   - 새로 저장되는 데이터는 자동으로 `v3`로 암호화  
   - 기존 데이터는 읽을 때 `v1/v2`로 복호화 가능  
   - 다시 저장되면 자연스럽게 `v3`로 갱신

→ **과거 데이터가 어떤 키로 암호화됐든 그대로 읽어 들이고, 점진적으로 새 키로 전환**되는 구조입니다.

---

## 6. 복잡해 보이지만 이렇게 정리할 수 있습니다

| 이유 | 설명 |
| --- | --- |
| 네이버 PW는 고위험 민감 정보 | 평문 저장 금지, 반드시 암호화 필요 |
| 복호화 가능한 암호화가 필요 | Pumpingstore 전달 시 실제 PW를 써야 하므로 단방향 해시 불가 |
| 키는 언젠가 유출될 수 있음 | 키 회전 없으면 전체 PW 유출 위험 |
| 암호문에 키 버전을 넣은 이유 | `vX` 프리픽스로 어떤 키인지 추적, 복호화 가능 |
| AttributeConverter 사용 이유 | 비즈니스 로직에서 암호화 코드를 직접 다루지 않아도 됨 |

---

## 7. 한 줄 요약
> **암호화**는 네이버 계정이 유출돼도 원문이 보이지 않게 막는 최후의 방어선이고,  
> **키 회전**은 키가 위험해졌을 때 시스템을 멈추지 않고 새 키로 교체할 수 있게 해주는 안전장치입니다.

---

## 더 필요한 자료가 있다면
- 키 회전 전체 시나리오(운영 절차)  
- v1 → v2로 전환될 때 내부적으로 어떤 일이 일어나는지  
- Swimlane ASCII 다이어그램  
- “암호화가 없으면 어떤 사고가 생기는가” 사례  
- AES / PBKDF2 / Salt / IV 초보자용 안내

필요한 형식을 알려주시면 계속 도와드릴 수 있습니다.
