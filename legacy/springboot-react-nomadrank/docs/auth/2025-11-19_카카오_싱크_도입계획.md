# 카카오 싱크 도입 단계 계획

## 목표
- 카카오 로그인 시 `legal_needs`(약관 동의 내역)를 수집·검증·저장하여 컴플라이언스 요구사항을 충족하고, 필수 약관 미동의 사용자의 흐름을 제어합니다.

## 선행조건
- 카카오 앱 설정(redirect URI, 동의 항목) 확인. 기본적으로 `v2/user/me` 응답에 약관 동의 내역이 포함됩니다.
- 백엔드: Spring Boot 3.x, WebClient 사용(`KakaoOAuthService`).
- 저장 방식 결정: `USER` 테이블 확장 vs. 별도 `USER_TERMS_AGREEMENT` 테이블. 권장: 별도 테이블.

## 3단계: DTO 수정 (백엔드)
- 파일: `src/main/java/.../domain/auth/dto/external/KakaoUserResponseDTO.java`
- 목적: `kakao_account.legal_needs` 필드를 매핑합니다.
- 구현 포인트:
  - Jackson에 `@JsonIgnoreProperties(ignoreUnknown = true)` 적용(하위 호환성).
  - `kakao_account` 내부에 `legal_needs` 리스트 추가.

```java
@JsonIgnoreProperties(ignoreUnknown = true)
public class KakaoUserResponseDTO {
    private Long id;
    @JsonProperty("kakao_account")
    private KakaoAccount kakaoAccount;

    @JsonIgnoreProperties(ignoreUnknown = true)
    public static class KakaoAccount {
        @JsonProperty("legal_needs")
        private List<LegalNeed> legalNeeds;
        // ... 기존 필드(이메일, 프로필 등)
    }

    @JsonIgnoreProperties(ignoreUnknown = true)
    public static class LegalNeed {
        private String code;         // 약관 코드
        private Boolean required;    // 필수 여부
        private Boolean agreed;      // 동의 여부
        @JsonProperty("revision_time")
        private String revisionTime; // 약관 개정 시각(버전)
        @JsonProperty("agreed_at")
        private String agreedAt;     // 동의 시각
    }
}
```

## 4단계: 서비스 로직 수정 (`KakaoOAuthService.java` 등)
- `getUserInfo`:
  - 보통 추가 파라미터 없이 `legal_needs`가 포함됩니다(필요 시 scope/props 확인).
  - DTO만 확장해도 수신 가능합니다.
- `AuthServiceApiV1#kakaoCallback`:
  - `KakaoUserResponseDTO`에서 `kakaoAccount.legalNeeds` 추출.
  - 검증: `required == true` 항목은 모두 `agreed == true` 여야 함.
  - 미동의 시 응답 예:
    - `code: CONSENT_REQUIRED`, `data.missingConsents: ["TERM_CODE", ...]` 반환.
  - 저장: 별도 테이블에 upsert.

### 권장 스키마 및 마이그레이션
- 테이블: `USER_TERMS_AGREEMENT`

```sql
CREATE TABLE USER_TERMS_AGREEMENT (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  user_id BIGINT NOT NULL,
  provider VARCHAR(20) NOT NULL,         -- e.g., KAKAO
  term_code VARCHAR(100) NOT NULL,
  required BOOLEAN NOT NULL,
  agreed BOOLEAN NOT NULL,
  version VARCHAR(64),                   -- revision_time 스냅샷
  agreed_at TIMESTAMP NULL,
  updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY uq_user_term (user_id, term_code)
);
```

- 동작: 로그인 시 `legal_needs`를 순회하며 upsert(최신 동의 시각/버전 반영).
- 백필: 배포 후 첫 로그인 시점에 저장. 이전 사용자는 다음 카카오 로그인 때 반영.

## 프론트엔드 연동 계약(요약)
- 성공 시: 로그인 응답에 `consents` 배열(code, version, agreedAt) 포함 가능.
- 미동의 시: `code=CONSENT_REQUIRED`, `data.missingConsents`로 유도 UI 처리.

## 테스트
- DTO 바인딩: `v2/user/me` 샘플 JSON으로 `legal_needs` 매핑 검증.
- 서비스: 필수 약관 미동의 → `CONSENT_REQUIRED`; 전부 동의 → 저장 및 통과.
- 통합: H2로 upsert 동작과 idempotency 확인.

## 롤아웃
- 1단계: DTO/DB/서비스 구현, 기능 플래그로 비활성.
- 2단계: 스테이징 검증(테스트 계정), 로그 모니터링.
- 3단계: 프로덕션 활성화, 미동의 사용자 플로우를 프론트와 협의해 안내.
