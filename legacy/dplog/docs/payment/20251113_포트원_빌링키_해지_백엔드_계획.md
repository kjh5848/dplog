# PortOne 빌링키 구독 해지 개발 계획

## 1. 배경 및 목표
- 프런트 문서와 백엔드 가이드에 정의된 구독 해지 플로우를 실제 서비스에 구현한다.
- 예약 결제 삭제 → 빌링키 해지 → 내부 상태 동기화까지 자동화하여 사용자에게 안정적인 해지 경험을 제공한다.
- 포트원 V2 API 계약과 내부 DB 구조를 일관성 있게 정리해 향후 기능 확장을 용이하게 한다.

## 2. 범위
- 프런트엔드: `DELETE /api/subscriptions/{subscriptionId}/cancel` App Route 구현 및 UI 연동.
- 백엔드(Next App Route → Spring API): 해지 전용 App Route 작성, 백엔드 Spring 서비스/클라이언트 확장.
- PortOne 연동: 예약 결제 조회·삭제, 빌링키 삭제 REST 호출 추가.
- 데이터 저장소: `SubscriptionEntity`, `BillingKeyEntity` 상태 전환 로직 보강 및 감사 로그 남기기.
- 테스트/운영: Playwright 시나리오, Spring 단위/통합 테스트, Sentry 로깅 정비.

## 3. 요구사항 정리
- 프런트 App Route
  - HTTP: `DELETE /api/subscriptions/{subscriptionId}/cancel`
  - Request Body: `{ "reason": string | undefined }`
  - Response Body: `{ "status": "canceled", "canceledAt": ISO8601 }`
  - 상태 코드: 200(성공), 404(이미 해지/미존재), 409(예약 결제 미삭제), 5xx(포트원 장애/내부 오류)
- 백엔드 Spring
  - 신규 엔드포인트: `/v1/payments/subscriptions/{subscriptionId}/cancel` (가칭)
  - 서비스 흐름: 예약 스케줄 조회/삭제 → 빌링키 검증 → 포트원 해지 → 내부 DB 갱신 → 감사 로그/알림
- PortOne API
  - `GET /v2/payment-schedules?billingKey=...`
  - `DELETE /v2/payment-schedules/{scheduleId}`
  - `GET /v2/billing-keys/{billingKey}`
  - `DELETE /v2/billing-keys/{billingKey}` (필요 시 `storeId`, `reason`)
- 데이터
  - `SubscriptionEntity.status`: `ACTIVE` → `CANCELED`
  - `SubscriptionEntity.endDate`: 해지일로 업데이트
  - `BillingKeyEntity.status`: `ACTIVE` → `DELETED`
  - 예약 내역/알림 로그 남기기 (`membershipUserLog`, 별도 이벤트 테이블 검토)

## 4. 작업 항목

### 4.1 프런트엔드
- [ ] `src/app/api/subscriptions/[subscriptionId]/cancel/route.ts` 생성, 백엔드 REST 호출 서비스(`src/lib/services/portoneService.ts`) 구현.
- [ ] `useCancelSubscription` 훅 작성 및 `CancelModal`/`CancelButton`에 연동.
- [ ] 응답 상태 코드별 사용자 메시지/재시도 처리, `requiresReason` 등 확장 플래그 대응.
- [ ] Zustand 스토어(또는 현재 상태 관리)에서 해지 후 상태 반영 및 Optimistic Update 검토.

### 4.2 백엔드(App Route → Spring)
- [ ] Next App Route에서 Spring API 호출을 래핑하는 서버 전용 함수 작성 (JWT/세션 인증 포함).
- [ ] Spring `PaymentControllerApiV1`에 `@DeleteMapping("/subscriptions/{subscriptionId}/cancel")` 추가.
- [ ] `PaymentServiceApiV1`에 해지 Orchestration 메서드 추가: 예약 조회/삭제 → 빌링키 검증 → 포트원 해지 → DB 업데이트.
- [ ] `PortOneV2Client`에 `getPaymentSchedules`, `deletePaymentSchedule`, `deleteBillingKey` 메서드 추가.
- [ ] `BillingKeyRepository`, `SubscriptionRepository`에 필요한 조회/업데이트 메서드 확장(예: reservationId 기반 조회).
- [ ] 예외 처리 정책 수립: PortOne 응답 코드 → 도메인 예외 → App Route 매핑.

### 4.3 데이터 & 동기화
- [ ] 예약 스케줄과 `SubscriptionEntity.subscriptionId`(PortOne 예약 ID) 매핑 규칙 문서화.
- [ ] 해지 성공 시 `SubscriptionEntity`의 `nextBillingDate`, `failureCount`, `membershipState` 정리.
- [ ] 사용자 통지(이메일/푸시) 트리거 포인트 정의, 임시로 SQS/이벤트 발행 여부 검토.
- [ ] 감사 로그 및 Sentry 태그 정의(`subscriptionId`, `billingKeyHash`, `storeId` 등).

### 4.4 테스트 & QA
- [ ] Spring 단위 테스트: PortOne 클라이언트 Mock → 예약 존재/미존재, 해지 성공/실패 분기 검증.
- [ ] 통합 테스트: `PaymentServiceApiV1.cancelSubscription()`(가칭) DB 상태 변화 확인.
- [ ] 프런트 Playwright: 예약 삭제 성공/실패, 빌링키 미존재, 5xx 재시도 시나리오.
- [ ] QA 체크리스트 업데이트: 실제 해지 후 결제 이력, 재구독 동선, Sentry 경고 확인.

### 4.5 운영 & 배포
- [ ] `npm run lint`, `npm test`, `./gradlew test` 자동화 파이프라인에 포함.
- [ ] 포트원 API 인증키/상점 ID 환경 변수 점검 및 배포 전 QA 계정으로 해지 실 검증.
- [ ] 운영팀/재무팀과 PG별 해지 사유 요구 여부 재확인, 고객센터 FAQ 업데이트.
- [ ] 배포 후 1주간 모니터링 지표(해지 요청 수, 실패율, 장애 응답)를 대시보드화.

*실제 일정은 백엔드/프런트 리소스 및 QA 창구와 협의 후 확정한다.*

## 6. 테스트 전략 요약
- Spring: JUnit5 + Mockito(`@MockBean`), 트랜잭션 롤백 여부 확인.
- Next App Route: API mock(예: MSW)으로 HTTP 계약 보증.
- E2E: Playwright + 테스트 PortOne 채널 사용, 예약/해지 API Mock.
- 회귀: 결제/구독 기존 시나리오(`subscribe`, `charge`)가 깨지지 않는지 확인.

## 7. 리스크 및 대응
- **포트원 API 권한/상점 매핑 오류** → 사전 QA 환경에서 storeId/channelKey 검증, 실패 시 Sentry 경고.
- **예약 스케줄 중복/유실** → 예약 ID와 내부 구독 ID 동기화 체크, 삭제 실패 시 재시도 로직/운영 알림.
- **프런트-백엔드 계약 불일치** → Swagger 또는 명세서 기반 상호 검증, 통합 테스트에 계약 테스트 추가.
- **사용자 통지 누락** → 해지 완료 시 webhook/이벤트 발행 여부 사전 정의, QA 체크리스트에 포함.
- **롤백 전략 미흡** → 해지 실패 시 상태 복원 로직 문서화, 수동 대응 프로세스 정비.
