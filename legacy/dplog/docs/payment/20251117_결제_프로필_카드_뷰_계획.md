# 결제 프로필 카드 뷰 계획 (2025-11-17)

## 1. 배경
- `/app/(home)/payment` 화면은 요금제 선택 및 결제 진행만 제공하고, 현재 결제 상태(최근 결제일, 다음 결제일, 결제 수단)를 한눈에 볼 수 있는 **내 결제 카드**가 없다.
- 고객센터 문의: “지금 내 결제가 유효한지?”, “다음 결제일이 언제인지?” 등의 기본 정보를 확인하기 위해 프로필 페이지나 CS에 문의해야 한다.
- 따라서 결제 페이지 상단 혹은 요금제 리스트 옆에 **결제 프로필 카드**를 추가해 현재 상태를 즉시 확인시키고, 업그레이드/다운그레이드 문구와도 연동해야 한다.

## 2. 목적
1. 현재 구독 상태, 다음 결제일, 최근 결제 여부, 결제 수단 요약을 한 곳에서 보여준다.
2. Pending cancel/downgrade 상태, 사용량 초기화 여부, 유예 기간 표시 등 CS 문의 감소.
3. 업그레이드/다운그레이드 CTA와 동일한 데이터를 공유하여 UX 일관성 확보.

## 3. 요구 사항
| 구분 | 내용 |
| --- | --- |
| 데이터 출처 | `/api/membership/current`, `/api/membership/profile-summary`, `/api/payments/status`(신규) |
| 표시 항목 | 플랜명/레벨, 루비/상태 뱃지, 다음 결제일, 마지막 결제일/금액, 결제 수단 끝 4자리/발급사, pendingCancel/pendingDowngrade, 사용량 요약 |
| 인터랙션 | “영수증 보기”, “결제 수단 변경”, “구독 해지/다운그레이드 관리” 버튼 |
| 빈 상태 | 유료 구독 없음 → “현재 결제 내역이 없습니다” + 구독 CTA |
| 오류 처리 | 각 API 실패 시 카드 내에서 skeleton → 에러 메시지 → 재시도 버튼 |

## 4. 데이터/엔드포인트 계획
1. **GET `/api/membership/current`**  
   - 기존 응답(`MembershipCurrentResponse`)에서 `plan`, `period`, `amount`, `status`, `nextPaymentDate`, `pendingCancel`, `pendingDowngrade` 활용.
2. **GET `/api/membership/profile-summary`**  
   - 사용량 및 리셋 정보를 카드에 요약 (ex: “이번 달 70/150회 사용”).
3. **신규 GET `/api/payments/status`** (Route Handler + `PaymentRepository.getPaymentStatus`)  
   - 최근 결제일, 최근 결제 금액, 결제 수단 정보(`brand`, `lastFour`, `billingKeyStatus`)를 내려줌.  
   - 서버는 PortOne/내부 BillingKey 테이블에서 최근 성공 결제 기록을 반환.

## 5. UI/컴포넌트 설계
```
┌──────────────────────────────────────────────┐
│  [플랜 아이콘]   성장 플랜 · 연간 (ACTIVE)       │
│  다음 결제일: 2025.12.01   |   최근 결제 2024.11.01 · ₩189,000 │
│  결제 수단: 우리카드 ****-1234 (정상)                │
│  이번 달 사용량: 70 / 150회 (12/01 리셋)              │
│  [영수증 보기]   [결제 수단 변경]   [해지/다운그레이드 관리] │
└──────────────────────────────────────────────┘
```
- `PaymentProfileCard.tsx` (신규):  
  - props: `currentSubscription`, `usageSummary`, `paymentStatus`.
  - 상태별 뱃지: ACTIVE(초록), PENDING_CANCEL(주황), PENDING_DOWNGRADE(파랑), EXPIRED/미결제(빨강).
- Skeleton/에러 컴포넌트 포함.
- `PaymentClientPage.tsx` 에서 상단 Hero 아래 혹은 좌측 column에 배치.

## 6. 상태 관리/쿼리
1. React Query로 `useQuery(['membershipCurrent'], fetchCurrentMembership)` 재활용.
2. `useQuery(['paymentStatus'], fetchPaymentStatus)` 추가, staleTime=60s.
3. `PaymentClientPage` 진입 시 `Promise.all`로 사전 fetch → Suspense 대응.
4. 업그레이드/다운그레이드/해지 성공 시 `queryClient.invalidateQueries` 3종(`membershipCurrent`, `membershipProfileSummary`, `paymentStatus`) 호출.

## 7. 작업 순서
1. **API 추가**
   - `/api/payments/status` Route Handler (백엔드 `PaymentServiceApiV1.getPaymentStatus` 호출).
   - `PaymentRepository.getPaymentStatus()` + `PaymentStatusResponse` 타입 정의.
2. **컴포넌트 작성**
   - `src/components/payment/PaymentProfileCard.tsx` (UI+상태 뱃지+버튼 슬랏).
   - Skeleton/에러 헬퍼 (`PaymentProfileCardSkeleton`, `PaymentProfileCardError`).
3. **PaymentClientPage 통합**
   - 서버 컴포넌트에서 `await getPaymentStatusAction()` 호출 or 클라이언트 fetch.
   - 상단에 카드 추가, 버튼 클릭 시 기존 모달/라우트 연결 (`/payment/receipt`, `/payment/method` 등).
4. **스타일/반응형**
   - 모바일에서는 한 열로, 데스크탑에서는 요금제 선택 패널과 나란히.
5. **QA**
   - ACTIVE, PENDING_CANCEL, PENDING_DOWNGRADE, FREE, 오류 상황 각각 확인.
   - 결제 실패/미결제 상태 시 빨간 경고 문구 + “결제 수단 변경” 버튼 강조.

## 8. 후속 이슈
- 영수증 보기/결제 수단 변경 경로가 아직 없다면 추가 계획 필요.
- PortOne에서 카드 발급사가 비어 있는 경우 fallback 문구 정의.
- Next.js 캐시 정책: Route Handler는 서버 캐시 무효화가 필요하므로 `cache: 'no-store'`.

## 9. 결제수단(빌링키) 관리 추가 플로우
- 첫 유료 구독 시 사용자가 직접 카드 정보를 입력해 PortOne 빌링키를 발급한다. 이후 자동결제·업그레이드·다운그레이드는 모두 이 **기존 빌링키**를 재사용한다.
- 현재 결제 페이지 어디서도 “내 카드” 정보를 확인하거나 변경할 수 없으므로, 프로필 카드와 연결된 결제수단 관리 플로우를 추가한다.

### 요구사항
1. **결제수단 정보 노출**
   - 프로필 카드에 현재 빌링키에 연결된 카드 브랜드/끝 4자리/유효 상태 표시 (예: “우리카드 ****-1234 · 유효”).
   - 카드가 만료되었거나 PortOne에서 비활성 상태면 경고 배지 + “결제수단 변경” 버튼 강조.
2. **결제수단 변경 UI**
   - `/payment/method` 또는 모달형 UI로 진입.
   - 기존과 동일한 카드 입력 폼(`PaymentClientPage`에서 쓰는 `cardForm`)을 재사용하되, `issueBillingKeyManually` → `completeBillingKey` 플로우를 “수정” 모드로 호출.
3. **API/서비스**
   - `POST /api/payments/billing-key/issue` (기존) 재사용 + `billingKey`가 업데이트되면 멤버십 구독 레코드에 최신 값 저장.
   - `PaymentRepository.updateBillingKey` (신규) 혹은 기존 `issueBillingKeyManually`를 래핑하여 “카드 정보 업데이트” 용도로 명시.
   - 성공 시 `/api/membership/current`와 `/api/payments/status`를 invalidate하여 카드 정보를 즉시 갱신.
4. **UX**
   - 변경 완료 후 “새 카드로 결제가 진행됩니다” 토스트 제공.
   - 업그레이드/다운그레이드 진행 시에도 최신 빌링키를 자동으로 사용하므로 추가 입력 불필요.
5. **보안/검증**
   - 카드 입력 시 masking 및 필수 검증(생년월일/비밀번호 2자리) 그대로 유지.
   - 실패 시 사용자 친화적 메시지 및 PortOne error code 로깅.

이 기능이 포함되면 고객은 결제 페이지에서 자신의 자동결제 카드 정보를 확인하고, 만료 전에 직접 갱신할 수 있어 CS 문의를 줄일 수 있다.

## 10. 백엔드 요청 사항
1. **`GET /v1/payments/status` (신규)**
   - 응답: 최근 결제 `history`(결제일, 금액, 상태, paymentId), `nextPaymentDate`, `pendingCancel`, `pendingDowngrade`, `billingKey` 정보(`brand`, `issuer`, `lastFour`, `status`, `updatedAt`).
   - PortOne/내부 billing 테이블에서 최신 성공 거래를 조회하고, 없으면 `history: null`.
   - 캐시 금지(`Cache-Control: no-store`), 세션 쿠키 기반 인증.
2. **`POST /v1/payments/billing-key/update` 혹은 기존 `/billing-key/issue` 확장**
   - 요청: 카드 정보 + `mode: 'update'`.
   - 처리: PortOne에서 기존 billingKey를 대체하고, `SubscriptionEntity.billingKey` 갱신, 변경 이력 로깅.
   - 응답: 새 `billingKey`, `cardBrand`, `lastFour`, `updatedAt`.
3. **멤버십 API 보강**
   - `/v1/membership/current` 응답에 `billingKeyStatus`, `currentPlanPrice`, `periodLabel` 추가.
   - `/v1/membership/profile-summary` 는 사용량/리셋 정보 그대로 유지하되, `resetDate` 필드 명시.
4. **웹훅/스케줄러**
   - BillingKey 업데이트 후 다음 정기 결제 예약(`PaymentScheduler`)에 새 키가 반영되도록 invalidate.
5. **에러/로그**
   - 카드 업데이트 실패 시 명확한 코드 (`BILLING_KEY_UPDATE_FAILED`, `INVALID_CARD_INFO` 등).
   - `/v1/payments/status` 호출 로그에 userId, subscriptionId, billingKeyId 기록.

위 계획대로 구현하면 결제 페이지에서 내 결제 상태를 즉시 확인할 수 있는 프로필 카드를 제공해 고객 문의를 감소시키고 업그레이드 UX와 연동할 수 있다.
