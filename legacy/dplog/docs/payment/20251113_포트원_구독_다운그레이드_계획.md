# PortOne 구독 다운그레이드 프런트엔드 구현 계획

## 1. 배경 및 목표
- 백엔드 초안(`Subscription Downgrade Design (Draft)`)에 맞춰 프런트에서 “다음 결제일부터 하위 요금제 적용” 플로우를 제공한다.
- 기존 구독 해지/업그레이드 UX를 확장해 다운그레이드 예약과 취소를 지원하며, 선불 모델(부분 환불 없음)을 유지한다.
- SubscriptionEntity의 `pending*` 필드를 활용해 예약 상태를 UI에 반영하고, PortOne 정기 결제 금액이 다음 청구 타이밍에 자동으로 변경되도록 안내한다.

## 2. 사용자 여정 및 UX 요구사항
- **엔트리 포인트**: 마이페이지 `구독 관리` 섹션의 요금제 카드 또는 가격표 화면.
- **주요 시나리오**
  1. 사용자가 하위 요금제를 선택 → “다음 결제일부터 적용됩니다” 안내 → 예약 생성.
  2. 이미 예약된 다운그레이드가 있을 때 → 상태 표시 및 취소 버튼 제공.
  3. 스케줄러 적용 후 → UI가 새 요금제로 자연스럽게 전환되며 “다운그레이드 완료” 메시지 노출.
- **필수 안내 문구**
  - “즉시 요금제 변경은 지원하지 않습니다. 다음 결제일부터 적용됩니다.”
  - “현재 결제된 기간 동안은 기존 요금제를 그대로 이용할 수 있습니다.”
  - “부분 환불은 제공되지 않습니다.”
- **상태 구조**
  - `ACTIVE`: 기본 상태, 다운그레이드 예약 없음.
  - `PENDING_DOWNGRADE`: `pendingMembershipLevel` 존재, `effectiveDate = nextBillingDate`.
  - `DOWNGRADE_APPLIED`: 스케줄러 처리 후 새로운 플랜 적용 (백엔드 상태 조회 시 반영).

## 3. API 연동 설계
### 3.1 App Route / 백엔드 계약
- **예약 생성**: `POST /api/subscriptions/{subscriptionId}/downgrade`
  - 바디: `{ targetMembershipLevel, targetBillingCycle, reason? }`
  - 성공 시: `{ status: "SCHEDULED", effectiveDate: ISO_STRING, message }`
- **예약 취소**: `POST /api/subscriptions/{subscriptionId}/downgrade/cancel`
  - 성공 시: `{ status: "CANCELLED", message }`
- 내부에서 `SubscriptionService` → 백엔드 `/v1/payments/subscriptions/{id}/downgrade` / `.../downgrade/cancel` 호출.
- 백엔드 응답(예: pending 필드)과 상태 코드를 그대로 매핑해 사용자 메시지 구성.

### 3.2 상태 조회
- `GET /api/payments/subscriptions/status`에 `pendingMembershipLevel`, `pendingBillingCycle`, `pendingAppliedAt` 등을 노출하도록 백엔드와 협업.
- 프런트는 `currentSubscription` 외에 `pendingDowngrade`(effectiveDate, targetPlan, targetCycle) 정보를 별도 모델로 관리.

## 4. 프런트 구조 변경 사항
### 4.1 상태 관리
- `usePriceViewModel`에 다운그레이드 관련 상태 및 액션 추가
  - `scheduleDowngrade(options)`
  - `cancelScheduledDowngrade()`
  - `pendingDowngrade` (effectiveDate, targetPlan, targetCycle, reason)
- API 호출 실패 시 사용자 친화적 메시지(`getUserFriendlyErrorMessage`) 활용, 재시도 옵션 제공.

### 4.2 UI 컴포넌트
- **다운그레이드 예약 모달**
  - 요금제 선택 시 “다음 결제일부터 적용” 안내, 확인 시 `scheduleDowngrade` 호출.
  - 간단한 사유 입력 필드(선택) 제공.
- **구독 카드 상태 표시 (`ProfileView`)**
  - `PENDING_DOWNGRADE` 상태라면:
    - 현재 플랜 정보 + “다음 결제일 {effectiveDate}부터 {targetPlanName}으로 전환 예정” 메시지.
    - “다운그레이드 예약 취소” 버튼 노출.
  - 예약 취소 시 `cancelScheduledDowngrade` 호출 후 상태 갱신.
  - 다운그레이드 적용 완료 후 `pendingAppliedAt`를 활용해 “{적용일 기준 메시지}” 토스트/배너 노출.
- **요금제 선택 화면 (`payment/page.tsx`)**
  - 업그레이드/다운그레이드 조건 검사 시 새 로직 추가:
    - 현재 플랜보다 하위인지 비교 → 아니면 경고 토스트.
    - `pendingDowngrade`가 이미 존재한다면 예약 덮어쓰기 여부 확인 후 안내.
  - CTA 레이블: `다운그레이드 예약`, `예약 취소`.

### 4.3 토스트 및 알림
- 예약 성공: “다운그레이드가 예약되었습니다. {effectiveDate}부터 적용됩니다.”
- 예약 취소: “다운그레이드 예약이 취소되었습니다.”
- 덮어쓰기 시 정책 결정(예: “기존 예약이 업데이트되었습니다”)에 따른 메시지 조정.

## 5. 검증 규칙 및 에러 처리
- **클라이언트 측 선검증**
  - targetLevel >= currentLevel → 즉시 토스트로 차단.
  - 현재 예약 상태에서 동일한 요청 시 사용자에게 안내 (중복 요청)
- **백엔드 에러 매핑**
  - 400: “하위 요금제로만 변경할 수 있습니다” / “이미 예약된 다운그레이드가 있습니다” 등.
  - 404: “활성 구독을 찾을 수 없습니다.”
  - 409: 포트원 예약 금액 갱신 실패 등 → “잠시 후 다시 시도해주세요.”
- **UI 반영**
  - 에러 발생 시 모달 유지, 입력값 보존, 재시도 안내.

## 6. 테스트 전략
- **단위 테스트**
  - `usePriceViewModel.scheduleDowngrade` / `cancelScheduledDowngrade`의 성공/에러 흐름(MSW mock).
  - 상태 해석 로직(`pendingDowngrade` → UI) 검증.
- **통합 테스트**
  - Next App Route → 백엔드 mock (`proxyBackendJson`) 호출 성공/실패 시나리오.
- **Playwright**
  - 다운그레이드 예약 → 상태 표시 → 취소 플로우.
  - 이미 예약된 상태에서 덮어쓰기 / 취소 버튼 비활성화 등의 엣지 케이스.

## 7. QA & 운영 체크리스트
- 예약 후 `nextBillingDate`까지 기존 요금제가 유지되는지, 스케줄러 실행 후 자동 전환이 이뤄지는지 확인.
- 예약 생성/취소 시 알림(토스트, 이메일/푸시 등)이 정상 발송되는지 검증.
- CS 매뉴얼 업데이트: “예약 확인/취소 방법”, “환불 없음” 명시.
- 모니터링: 예약 생성/취소 건수, 스케줄러 적용 로그를 Sentry/로그 대시보드에 연동.

## 8. 향후 확장 고려
- 백엔드가 즉시 다운그레이드(부분 환불) 정책을 지원할 경우, 프런트에서 별도 플로우(해지/환불과 유사)를 추가할 수 있도록 컴포넌트와 훅을 모듈화.
- 구독 관리 화면에서 업그레이드/다운그레이드/해지 상태를 한눈에 볼 수 있도록 리팩토링 고려.
