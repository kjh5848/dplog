# Payment Server Actions Integration Plan

## 1. Route Handlers 현황
| 엔드포인트 | 역할 | 상태 | 서버 액션 대상 여부 |
| --- | --- | --- | --- |
| `POST /api/payments/pre-register` | 결제 사전등록 | 클라이언트 fetch → Route | ✅ 서버 액션으로 이전 예정 |
| `POST /api/payments/billing-key/complete` | 빌링키 완료 | 클라이언트 fetch → Route | ✅ 서버 액션 대상 |
| `POST /api/payments/billing-key/fail` | 실패 기록 | 클라이언트 fetch → Route | ✅ 서버 액션 대상 |
| `POST /api/payments/upgrade` | 구독 업그레이드 | Route → 백엔드 | ◻︎ (중요도 중) |
| `/api/subscriptions/...` | 해지/다운그레이드 등 | Route → 백엔드 | ◻︎ (추후 단계) |

## 2. 서버 액션 도입 이유 (상세)
- 브라우저 network 탭에 `/api/payments/**` 요청이 찍히는 것을 제거
- 쿠키/세션 처리를 서버에서만 수행 → 민감 정보 보안 강화
- 클라이언트는 단순 함수 호출/폼 제출만 처리, 서버가 백엔드 API 호출 담당

## 3. 난이도별 계획
### 하
- (완료) 멤버십 카탈로그 ISR/SSR 적용
- `validateSelectedPlan` → 서버 액션 `getMembershipDetailAction` 도입 (완료)

### 중
- `scheduleCancel`, `scheduleDowngrade`, `cancelScheduledCancel` 등 Route 호출을 서버 액션으로 변경
- Route Handler 로직을 `paymentService`/`subscriptionService` 모듈로 재사용하고, 서버 액션은 해당 모듈을 직접 호출
- 클라이언트 측에서는 `await cancelSubscriptionAction(...)` 형태로 호출 (네트워크 노출 제거)

### 상
- `handleSubscriptionPayment` 전체를 서버 액션에 위임 (`pre-register`, `billing-key/complete`, 실패 기록)
- 업그레이드/다운그레이드 플로우도 동일 방식으로 전환
- 서버 액션 내부에서 `cookies()` 기반 사용자 검증 및 오류 처리 (로그인 필요 → 즉시 redirect)

## 5. 다음 단계
1. 서버 액션 파일(`actions.ts`)에 취소/다운그레이드 액션 추가 → 클라이언트 fetch 제거
2. Route Handler에서 사용 중인 `fetchMembershipDetail`/`PaymentService` 등을 서버 액션에서도 사용할 수 있게 refactor
3. 결제 플로우 서버 액션화 완료 후, 브라우저 네트워크에 `/api/payments/**`가 노출되지 않는지 확인

## 6. `/v1/membership/current` SSR & Cache 계획
- `/payment/page.tsx`에서 로그인 사용자의 현재 구독 정보를 SSR 단계에서 직접 조회하고 `PaymentClientPage`에 `initialCurrentSubscription`으로 전달한다.
- SSR fetch 시 `next: { revalidate: 300, tags: ["membership-current"] }`를 부여해 사용자 단위 캐시를 확보하고, 동일 요청이 연속해서 발생하지 않도록 한다.
- 클라이언트 진입 시에는 전달된 초기 데이터를 `usePriceViewModel`의 `hydrateCurrentSubscription`으로 수화(hydrate)해 별도의 `useEffect` fetch를 제거한다.
- 구독 상태가 변경되는 Route (`/api/payments/billing-key/complete`, `/api/payments/upgrade`, `/api/subscriptions/**`)에서 `revalidateTag("membership-current")`를 호출하여 캐시를 즉시 무효화한다.
- 결제 성공 직후에는 기존대로 `loadCurrentSubscription()`을 호출해 클라이언트 상태를 최신화하되, 이후 `/payment` 페이지 방문 시에는 SSR 캐시가 자동으로 최신화를 반영한다.
