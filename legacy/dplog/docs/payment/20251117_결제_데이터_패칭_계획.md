# 2025.11.17 Payment Page Data Fetching Plan

## 1. 현황 요약
| 구간 | 위치 | 방식 | 비고 |
| --- | --- | --- | --- |
| 카드 입력/결제 흐름 | `src/app/(home)/payment/page.tsx` | 클라이언트 컴포넌트 | `fetch('/api/...')` 요청이 browser DevTools에 노출됨 |
| Route Handler | `src/app/api/...` | 서버 측 → 백엔드 호출 | 쿠키/세션 처리, API 키 보호 |
| 멤버십 카탈로그 | 클라이언트 `fetch('/api/membership/...')` | 매 진입 시 fetch | 캐시 없음 |

## 2. 목표
- 민감 데이터를 다루는 결제 요청을 서버에서 전담해 브라우저 네트워크 노출 최소화
- 정적 데이터는 ISR/SSG로 캐시해 초기 렌더와 UX 개선

## 3. 변경 전략 (난이도별)
### 하 (정적 데이터 → SSG/ISR)
1. 멤버십 카탈로그/FAQ를 Server Component에서 `fetch`하고 `export const revalidate = 3600;` 등 ISR 적용
2. 클라이언트 `fetch` 제거, 초기 props로 전달

### 중 (Route Handler + 서버 액션 통합)
- ✅ `validateSelectedPlan` 클라이언트 fetch를 서버 액션(`getMembershipDetailAction`) 호출로 전환
- 추가 계획: `scheduleCancel`, `scheduleDowngrade` 등 나머지 결제 관련 fetch도 서버 액션으로 이관

### 상 (결제 플로우 서버화)
1. `handleSubscriptionPayment`의 `pre-register`, `billing-key/complete`, `billing-key/fail` 등을 서버 액션으로 이관
2. 서버 액션 내부에서 `cookies()` 기반 유저 체크 및 `redirect('/login?redirect=...')` 처리
3. 클라이언트에는 서버 액션 호출 결과만 전달 → 브라우저 네트워크에 민감 API 노출 방지

## 4. 적용 순서
1. (하) 멤버십 카탈로그/FAQ ISR 적용
2. (중) 서버 액션 도입 + Route Handler 로직 재사용
3. (상) 결제/해지/다운그레이드 플로우 전체를 서버 액션/Server Component로 재구성

이 계획에 따라 순차적으로 전환하면, 민감한 결제 요청은 서버에서만 처리되고, 프론트엔드에서는 정적 데이터만 비동기로 가져오게 됩니다.
