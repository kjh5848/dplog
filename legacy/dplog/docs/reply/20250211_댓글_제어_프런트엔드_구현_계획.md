# 엔플레이스 댓글 제어 프런트엔드 구현 계획

## 1. 배경 및 목표
- 백엔드에서 전달된 "Nplace Reply – 프론트 요청 순서 & API 스펙"을 기반으로 네이버 계정 연동부터 댓글 ON/OFF 토글, 전체 삭제까지의 플로우를 프런트에 이식한다.
- 백엔드 스펙상 **댓글 본문은 내려주지 않으며**, 상점 단위로 AI 댓글 자동응답 활성/비활성만 제어한다. 화면에서도 댓글 텍스트 리스트를 보여주지 않는다.
- 모든 호출은 Next.js App Route(`/src/app/api`)를 통해 백엔드(`/v1/...`)로 프록시해 `HttpSession` + `authInfo`를 그대로 유지하고, 공통 `ResDTO`(`code`, `message`, `data`) 계약을 지킨다.
- 신규 화면(`(nplace)/(rank)/reply`)은 계정 미등록/등록/토글/삭제까지 사용자 여정을 안내하며, 기존 `ReplyContent`가 가진 상점 그리드/리스트 UX를 유지한다.

## 2. 기능 요구사항 요약
| 구분 | 엔드포인트 | 목적 | 특이사항 |
| --- | --- | --- | --- |
| 네이버 계정 여부 | `GET /v1/user/naver/status` | 계정 존재 확인 | `exists=false`면 나머지 API 호출 차단 |
| 계정 조회 | `GET /v1/user/naver` | 저장된 `naverId`, 복호화된 `naverPw` 수신 | PW는 화면에 노출하지 않고 단순 재전송용으로만 사용 |
| 계정 등록/수정 | `POST/PUT /v1/user/naver` | 아이디/비밀번호 등록 또는 변경 | 성공 시 곧바로 상태 재조회 |
| AI 댓글 제어 상태 조회 | `GET /v1/reply` | placeId별 자동응답 ON/OFF 상태 리스트 | 댓글 본문/컨텐츠는 내려오지 않음 |
| AI 댓글 토글 | `POST /v1/reply` | 단일 placeId에 대해 ON/OFF | 서버에서 네이버 인증 및 Pumpingstore 호출까지 처리 |
| 댓글 제어 정보 삭제 | `DELETE /v1/reply` | Pumpingstore & 로컬 DB 모두 삭제 | 실행 전 사용자 재확인 필수 |

예외 응답은 `code != 0` + 메시지 조합이므로, 프런트는 공통 알림/재로그인 UX를 사용한다.

## 3. 권장 사용자 흐름
1. `useAuthGuard`로 로그인 여부를 확인해 비로그인 시 `/login` 이동.
2. Reply 페이지 진입 시 `GET /api/nplace/naver/status` 호출 → 미등록이면 계정 등록 폼만 노출.
3. 등록되지 않은 경우 `POST /api/nplace/naver`로 네이버 계정을 저장하고 상태를 다시 조회.
4. 등록된 경우에는 `GET /api/nplace/naver`로 계정 식별자만 확보(폼 프리필 또는 정보 배너에 사용).
5. 동시에 `GET /api/nplace/reply`와 기존 `TrackRepository.getShopList()` 결과를 조합해 플레이스 카드에 AI 댓글 자동응답 ON/OFF 상태를 반영(댓글 본문 노출 없음).
6. 사용자가 특정 플레이스를 선택하고 토글 버튼을 누르면 `POST /api/nplace/reply`를 호출하고, 성공 시 목록/상태를 invalidation.
7. 전체 삭제 요청 시 확인 모달에서 사용자가 승인해야 `DELETE /api/nplace/reply` 호출 → 응답 성공 후 리스트/상태 초기화.

## 4. API 연동 설계

### 4.1 Next.js App Routes (서버)<br>
모든 엔드포인트는 Next 서버에서 `proxyBackendJson`을 이용해 Spring 백엔드를 호출한다. 경로 제안은 아래와 같다.

| App Route | HTTP | 백엔드 경로 | 설명 |
| --- | --- | --- | --- |
| `/api/nplace/naver/status` | GET | `/v1/user/naver/status` | 계정 존재 여부 조회 |
| `/api/nplace/naver` | GET | `/v1/user/naver` | 저장된 계정 정보 수신 |
| `/api/nplace/naver` | POST | `/v1/user/naver` | 신규 등록 |
| `/api/nplace/naver` | PUT | `/v1/user/naver` | 수정 |
| `/api/nplace/reply` | GET | `/v1/reply` | placeId + active 리스트 |
| `/api/nplace/reply` | POST | `/v1/reply` | 단건 토글 |
| `/api/nplace/reply` | DELETE | `/v1/reply` | 전체 삭제 |

App Route 공통 처리
- `src/lib/services/nplaceReplyService.ts`를 신설해 JSON 파싱, 필드 검증(예: placeId 빈값 여부), 에러 메시지 매핑을 담당한다.
- `proxyBackendJson` 호출 시 `label`을 지정해 Sentry/로그에서 추적 가능하게 한다 (예: `label: "naver-status"`).
- 백엔드 `code != 0`인 경우 `NextResponse.json({ success: false, code, message })` 형태로 내려 UI에서 일관 처리.
- `DELETE`는 본문이 없으므로 `request.json()` 파싱을 건너뛰고 바로 프록시한다.

### 4.2 타입 & 유틸 정의
- `src/types/nplaceReply.ts`
  ```ts
  export interface NaverStatusResponse { exists: boolean; }
  export interface UserNaverCredentials { userId: number; naverId: string; naverPw: string; }
  export interface NplaceReplyInfo { id: number; placeId: string; active: boolean; }
  ```
- `ClientApiResponse<T>` 패턴을 재사용해 `success`, `code`, `error`, `data` 구조를 맞춘다.
- `src/utils/errors/apiErrorMapper.ts`(가칭)에 네이버 계정/댓글 제어 전용 메시지 테이블을 추가해 서버 메시지를 그대로 노출할지, 사용자 친화 문구로 바꿀지 결정한다.
- React Query 키는 `['nplace','reply','list']`, `['nplace','naver','status']` 등 도메인 단위로 나눈다.

## 5. 클라이언트 설계

### 5.1 상태 관리 전략 (ViewModel 미사용)
- 별도 ViewModel 파일 없이 `ReplyClientPage`와 `ReplyContent`에서 React Query + 로컬 상태만으로 상태를 관리한다.
  - `useQuery` 세트: `['nplace','naver','status']`, `['nplace','naver','account']`, `['nplace','reply','list']`를 선언하고, 네이버 계정이 존재할 때만 계정/댓글 리스트 API를 호출한다.
  - `useMutation` 세트: 계정 저장(등록/수정), 댓글 토글, 전체 삭제를 컴포넌트 내에서 선언하고 성공 시 각 쿼리를 `invalidateQueries`로 갱신한다.
  - 선택된 플레이스 관리용 Set(`selectedPlaceIds`)을 유지해 대량 활성화/비활성화 시 순차적으로 mutation을 호출한다.
  - 실패 시 서버 `message`를 그대로 토스트로 보여주고, 인증 오류는 공통 로직으로 `/login` 리디렉션한다.
  - 네비게이션은 App Router의 `useRouter`만 사용하며, 댓글 기능은 페이지 내부 요청으로 처리하고 서버 액션은 사용하지 않는다.

### 5.2 UI/UX 변경점
- `ReplyClientPage`
  - 페이지 진입 시 네이버 계정 상태 쿼리만 먼저 호출하고 `exists` 값으로 계정 폼 또는 댓글 리스트를 조건부 렌더링한다.
  - 계정 미등록 시 등록 폼만 노출하고, 성공하면 `['nplace','naver','status']` 쿼리를 무효화해 즉시 UI를 갱신한다.
- `ReplyContent`
  - 상단에 네이버 계정 안내 카드와 등록/수정 버튼, 전체 삭제 버튼을 배치한다.
  - 각 상점 카드에는 AI 댓글 자동응답 상태 배지 + ON/OFF 토글 버튼을 추가하고, 선택 상태는 `selectedPlaceIds`로 기록한다. 댓글 텍스트/최근 댓글 리스트는 보여주지 않는다.
  - 다중 선택 시 "선택 댓글 활성화/비활성화" 버튼으로 순차 Mutation을 실행하고, 성공/실패 개수를 토스트로 안내한다.
- 신규 컴포넌트 제안
  - `NaverAccountForm`: `react-hook-form` + `zod` 검증으로 아이디/비밀번호 입력을 처리하고, PW는 state에서만 관리한다.
  - `ReplyStatusBadge`: placeId가 활성화됐는지 시각화 (예: `Active`, `Paused`).
- `ReplyShopList`
  - `replyStateMap`을 받아 `shop.shopId`와 댓글 상태를 매칭해 카드마다 상태/토글을 보여준다.
  - `isReplyLoading` 플래그로 댓글 데이터 로딩 상태를 표시한다.

### 5.3 Error & Loading UX
- `code != 0` 시 서버 메시지를 기본으로 띄우되, `네이버 인증 정보가 없습니다`는 계정 등록 카드로 포커스 이동.
- `AuthenticationException` 감지 시 `toast.error('재로그인이 필요합니다')` 후 `/login`으로 이동.
- Pumpingstore 장애(`NomadscrapException`)는 재시도 버튼 + 고객센터 안내 문구를 추가한다.
- `DELETE` 성공 시 되돌릴 수 없음을 다시 한번 토스트로 안내하고, `selectedPlaceIds` 초기화.

### 5.4 라우터 및 서버 코드 구성 전략
- 백엔드 호출은 모두 Next App Route(`/src/app/api/nplace/...`)를 경유해 `HttpSession`을 유지한다. 클라이언트 컴포넌트에서는 해당 App Route를 REST로 호출하고, 서버 액션을 사용하지 않는다.
- App Route 폴더 구조 예시
  - `/src/app/api/nplace/naver/status/route.ts` → `GET /api/nplace/naver/status`
  - `/src/app/api/nplace/naver/route.ts` → `GET/POST/PUT /api/nplace/naver`
  - `/src/app/api/nplace/reply/route.ts` → `GET/POST/DELETE /api/nplace/reply`
- 각 라우트에서는 `src/lib/services/nplaceReplyService.ts` 같은 서버 전용 서비스만 호출하도록 구성한다. 서비스 내부에서 요청 본문 파싱, 필수 필드 검증, `proxyBackendJson` 호출, 에러 매핑을 담당한다.
- 공용 타입은 `src/types/nplaceReply.ts`, 에러 매핑은 `src/utils/errors/nplaceReply.ts`처럼 기능별 파일로 분리해 IDE 검색성을 높인다.
- 서버 로직을 하나의 `server.ts`에 몰아넣기보다는, 기능 단위 서비스(`nplaceReplyService`, `naverAccountService`)로 세분화하면 테스트와 유지보수가 용이하다.

### 5.5 상점 등록 모달 & 링크 추출 계획 (/track 재사용)
- **모달 재사용**: `/track`에서 쓰는 `TrackNplaceSearch`(경로 `src/components/nplrace/rank/track/TrackNplaceSearch.tsx`)를 Reply 페이지에서도 그대로 사용한다. 트리거 버튼만 Reply 전용 UI에 배치하고, 모달 열림 상태/콜백을 props로 연결한다.
- **데이터 흐름**
  1. 사용자가 네이버 지도 링크 또는 placeId를 입력 → 모달에서 `useNplaceRankTrackViewModel.fetchTrackable` 호출.
  2. 백엔드 `/v1/nplace/rank/trackable`이 링크 내 placeId를 파싱·검증 후 `nplaceRankShop`을 반환.
  3. 모달의 “등록” 버튼에서 `nplaceRankShop.shopId`를 **Reply placeId**로 사용해 `POST /api/nplace/reply`(active 기본값 true) 호출.
  4. 성공 시 React Query에서 `['nplace','reply','list']`와 `['nplaceRankShopList']`를 모두 invalidation해 카드 목록을 최신화.
- **링크 → placeId 추출 규칙(프런트 보조용)**
  - 지원 URL 예시: `https://map.naver.com/p/search/.../place/1203311506`, `https://map.naver.com/p/entry/place/1203311506`, `https://m.place.naver.com/restaurant/1203311506/home`, 또는 숫자 `1203311506` 직접 입력.
  - 보조 util 초안: `extractNaverPlaceId(urlOrId: string)` → `const match = urlOrId.match(/place\\/(\\d+)|entry\\/place\\/(\\d+)|(\\d{6,})/); return match?.[1] || match?.[2] || match?.[3] || null;`
  - 파싱 성공 시 입력창 아래에 “추출된 placeId: {id}”를 미리보기로 띄워 사용자 확인 UX 제공. 추출 실패 시 백엔드 파싱에 맡기되 에러 토스트.
- **UI 동작 요약**
  - Reply 리스트 상단에 “플레이스 등록” CTA → 클릭 시 `/track` 모달 오픈.
  - 검색 결과 카드에서 상점명/주소 미리보기 → “등록” 클릭 시 바로 활성화 요청까지 처리(POST /v1/reply), 이후 토스트 및 리스트 반영.
  - 이미 등록된 placeId일 때는 서버 `code != 0` 메시지를 그대로 노출하고, 입력값을 유지해 재시도/수정 가능하게 한다.

## 7. 리스크 & 오픈 이슈
- **암호 노출 위험**: `GET /v1/user/naver` 응답에 복호화된 PW가 포함되므로, 프런트 state에 오래 저장하지 않고 폼 제출 후 즉시 초기화한다.
- **데이터 불일치**: `nplaceReplyList`에 있지만 `TrackRepository`의 `shopList`에 없는 placeId 처리를 위한 fallback UI가 필요하다.
- **동시 요청 충돌**: 다중 플레이스를 동시에 토글할 경우 서버는 단건 처리만 제공하므로, 순차 Promise 혹은 `Promise.allSettled`로 응답을 관리하고 부분 성공을 사용자에게 알린다.
- **세션 만료**: App Route에서 401/419 응답 시 재로그인 플로우가 반복되지 않도록 전역 에러 핸들러와 연결한다.
- **Pumpingstore 장애**: 서버 메시지를 그대로 보여주는 것 외에, 재시도/고객센터 CTA를 UI에 명시해 사용자 혼란을 줄인다.
