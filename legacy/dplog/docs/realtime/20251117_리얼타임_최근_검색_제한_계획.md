# Realtime 최근 검색 한도 대응 계획 (with /track Toast)

## 1. 목표
- `/realtime` 화면이 `/track`과 동일한 한도 초과 경험(CTA 버튼 포함)을 즉시 노출한다.
- 한도 도달 이후에는 추가 API 요청 없이 프론트에서만 토스트/배너를 재생하며, 최근 검색 결과로 대체 UX를 제공한다.
- 한도 정보(usageType, 사용량/총량)를 `/track`과 동일하게 `notifyUsageLimit`을 통해 표기한다.

## 2. 참고 대상
- `/track` ViewModel (`src/viewModel/nplace/track/nplaceRankTrackViewModel.ts`)  
  - `notifyUsageLimit(undefined, { usageType: "NPLACE_RANK_SHOP", force: true })` 형태로 공통 토스트를 호출.
- 공통 토스트 유틸: `src/utils/usageLimitNotifier.tsx` (CTA, 쿨다운 등 이미 구현됨).
- Realtime ViewModel: `src/viewModel/nplace/realtime/nplaceRankReailTimeViewModel.ts`  
  - 현재 `fetchSearchResults`/`useExecuteSearch`가 한도 초과 이후에도 매번 API를 다시 호출.

## 3. 구현 단계 (/track 참조)
1. **한도 초과 감지 & 토스트**
   - `/track`에서는 `TrackViewModel`의 `addShop`/`update` 등에 `notifyUsageLimit(undefined, { usageType: "NPLACE_RANK_SHOP", force: true })`를 호출해 CTA 토스트를 노출한다.
   - `/realtime`도 `fetchSearchResults`에서 `payload.code === -9`일 때 동일한 패턴으로 즉시 `notifyUsageLimit`을 호출해 CTA 버튼(멤버십 업그레이드, 닫기)을 표시한다.

2. **한도 상태 플래그 & 멱등 처리**
   - `useExecuteSearch` 내부에서 `usageLimitReached`/`usageLimitMeta` 상태를 유지한다.
   - 한도 초과 응답이 들어오면 상태를 `true`로 바꾸고, 이후 `handleSearch`/`handleRecentSearchClick`에서 서버 요청 대신 토스트만 재실행한다.

3. **최근 검색 UX**
   - 한도 도달 시 최근 검색 결과 패널을 자동 확장해 사용자는 저장된 결과를 확인할 수 있다(서버 호출 없음).
   - CTA 버튼과 함께 `/membership` 링크 버튼을 제공해 `/track`과 동일한 업셀 경험을 유지한다.

4. **UI 배너**
   - 폼 상단에 “한도 초과” 배너 + 업그레이드 CTA를 렌더링해 브라우저 새로고침 후에도 즉시 상태를 알 수 있도록 한다.

5. **테스트**
   - `/track`과 동일한 플로우로 QA: 29회까지 정상 → 30회차 응답에서 CTA 토스트, 이후 요청 차단 여부 확인.
   - 최근 검색 버튼/엔터 입력 등 모든 진입점을 반복 테스트.

## 4. 작업 항목 요약
| 구분 | 작업 | 담당 파일 |
| --- | --- | --- |
| 토스트 재사용 | `notifyUsageLimit` 호출 추가 (`usageType: "NPLACE_RANK_REALTIME"`) | `src/viewModel/nplace/realtime/nplaceRankReailTimeViewModel.ts` |
| 상태 관리 | `usageLimitReached` 도입 및 reset 인터페이스 | 동일 ViewModel + `RealtimeForm.tsx` |
| UX 보완 | 최근 검색 버튼 비활성화/배지, 토스트 트리거 | `src/components/nplrace/rank/realtime/RealtimeForm.tsx` |
| QA | `/track` 플로우 대비 `/realtime` 한도 도달/해제 E2E 확인 | Playwright or 수동 |

위 순서대로 진행하면 `/track`과 동일한 한도 초과 경험을 `/realtime`에도 적용하면서, 불필요한 API 호출도 차단할 수 있다.
