# N-PLACE Realtime MVVM 구조 분석과 Next(App Router) 개선 제안

## 1) 현재 구조 요약
- 계층
  - `components`: 클라이언트 페이지(UI, 헤더/가드/레이아웃 조합)
    - 예: `src/components/nplrace/rank/realtime/RealtimeClientPage.tsx`
  - `viewModel`: React Query 기반 데이터 로딩/뮤테이션, 일부 로컬스토리지 캐시
    - 예: `src/viewModel/nplace/realtime/nplaceRankReailTimeViewModel.ts`
  - `model`: 백엔드 REST 호출 캡슐화(Repository)
    - 예: `src/model/TrackRepository.ts`
- 특징
  - 클라이언트 중심 데이터 페칭(CSR). 초기 렌더 전 로딩 UI 노출, SEO/초기속도 제약.
  - 인증은 클라이언트 훅(`useAuthGuard`)으로 처리(게스트 시 토스트 + push).
  - VM 내부에 직접 fetch(검색)와 Repository 호출(상점/그룹)이 혼재.
  - 최근 검색/결과는 로컬스토리지, 서버 데이터는 React Query 캐시로 이중 관리.
  - 일부 네이밍/타입 일관성 이슈: `nplaceRankReailTimeViewModel`(오탈자).

## 2) Next(App Router) 권장 구조 대비 차이
- 데이터 페칭/SSR
  - 현행: 클라이언트에서만 fetch → 초기 공백/스피너, 번들 증가.
  - 권장: 서버 컴포넌트에서 초기 데이터 fetch 또는 React Query SSR(prefetch + Hydrate)로 첫 화면을 채워 LCP/SEO 개선.
- 인증/리다이렉트
  - 현행: 클라이언트 가드로 토스트 후 `router.push`.
  - 권장: 레이아웃/페이지(server)에서 쿠키 검사 후 `redirect('/login')`, 또는 `middleware.ts`로 보호 → 깜빡임 제거, 더 안전.
- 라우트 단위 로딩/에러
  - 현행: 페이지 내부 `Suspense`와 커스텀 로딩/에러 UI 혼용.
  - 권장: 세그먼트 `loading.tsx`/`error.tsx`로 표준화, 컴포넌트 내부는 필요한 곳만 `Suspense`.
- 계층 일관성
  - 현행: VM에 직접 fetch + Repository 혼재, 쿼리 키/옵션도 분산.
  - 권장: 모든 API는 Repository에서 일원화, VM은 React Query 훅만 조립. 쿼리 키/옵션은 중앙화.
- 네트워크/보안
  - 현행: 브라우저에서 외부 API(`NEXT_PUBLIC_API_URL`) 직접 호출 + `credentials: 'include'`.
  - 권장: App Router API(`route.ts`) 또는 Server Actions로 서버에서 프록시/쿠키 제어(도메인/멱등/보안 관리 용이).

## 3) 개선 방향 제안
- 레이어 정리
  - VM 내부 fetch(예: 검색)를 `RealtimeRepository`로 이전(모든 API 호출은 model 계층으로 일원화).
  - 쿼리 키 팩토리: `src/query/keys.ts`로 키 생성 함수를 중앙화.
- 서버 주도 초기화(SSR/Hydration)
  - `(dplog)/(nplace)/(rank)` 세그먼트 `layout.tsx` 추가.
  - 서버에서 상점/그룹 데이터 prefetch 후 `<Hydrate>`로 주입 → 클라이언트 VM은 동일 쿼리 키로 즉시 사용.
- 인증 개선
  - `middleware.ts` 또는 세그먼트 `layout.tsx`에서 쿠키 검사 후 `redirect('/login')`로 보호(클라이언트 가드 최소화).
- API 라우트 도입(선택)
  - `/app/api/auth/logout/route.ts` 등 추가하여 멱등/쿠키/리다이렉트 서버 단일 처리. 클라이언트는 내부 API만 호출.
- 상태/캐시 일관성
  - 로컬스토리지(최근 검색/결과)는 유지 혹은 React Query persist로 일원화. 이중 관리 최소화.
  - React Query 전역 옵션(기본 `staleTime`, `retry`, onError)을 Provider에서 통일.
- 네이밍/타입/토폴로지
  - 오탈자 수정: `nplaceRankReailTimeViewModel` → `nplaceRankRealTimeViewModel`.
  - 모델 응답 검증(Zod 등)으로 런타임 안정성 강화.
- UI 경계선
  - 라우트 수준 로딩/에러는 `loading.tsx`/`error.tsx`로 표준화. 페이지 내부 `Suspense`는 국소 비동기에만 사용.
- 멱등/사용성
  - 뮤테이션 후 invalidate 규칙을 유틸로 묶어 재사용(상점 추가/삭제/그룹 변경 등 공통 처리).

## 4) 점진 적용 플랜(우선순위)
1. 모델 일원화
   - `fetchSearchResults` → `RealtimeRepository.fetchSearchResults`로 이동.
   - VM은 `useMutation({ mutationFn: RealtimeRepository.fetchSearchResults })`로 교체.
   - 쿼리 키 팩토리 도입.
2. SSR/Hydrate 도입
   - `rank` 세그먼트 `layout.tsx`에 React Query Provider + Hydrate 래핑.
   - 서버에서 `getShopList`/`getGroupList` prefetch → 클라이언트 초기 로딩 감소.
3. 인증 서버 전환
   - `middleware.ts` 또는 세그먼트 `layout.tsx`에서 쿠키 검사 후 `redirect('/login')`.
4. API 라우트 추가(선택)
   - `/api/auth/logout` 작성(백엔드 프록시, 멱등, 쿠키 관리, 리다이렉트).
5. 리팩토링/정리
   - 네이밍/타입/쿼리 옵션 표준화, 로컬스토리지/캐시 전략 결정.

## 5) 기대 효과
- 초기 페인트/SEO 개선: 서버 주도 데이터로 첫 화면 완성도 향상.
- 깔끔한 인증 UX: 깜빡임/토스트 의존도↓, 서버 리다이렉트로 보안/일관성↑.
- 유지보수성 향상: API 호출 책임(model) 일원화, 쿼리 키/옵션 중앙화.
- 멱등/안정성 강화: 서버 API 라우트로 쿠키/리다이렉트/에러 처리 표준화.
