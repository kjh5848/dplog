# 사용량 한도 메시지 & 캐싱 전략 (2025-12-05)

> 참고: `docs/membership/20251120_멤버십_일간사용량_프론트_체크리스트.md`, `docs/realtime/20251117_리얼타임_최근_검색_제한_계획.md`, `docs/membership/20251017_멤버십_SSR_Next_계획_국문.md`

## 1. 목표 & 범위
- 일간/월간 혼합 한도 모델(참고: 2025-11-20 문서)을 사용하는 모든 기능(Nplace 실시간·키워드·블로그·리뷰, 상점 등록, 키워드 추적 등)에 동일한 한도 경험을 제공한다.
- 한도 도달 시 토스트/배너/CTA 메시지로 업그레이드를 즉시 유도하고, 백엔드 재호출을 차단해 낭비를 줄인다(참고: 2025-11-17 실시간 계획).
- SSR 멤버십 페이지(참고: 2025-10-17 계획)와 연결되는 업셀 경로(`/membership/[planId]`)를 공통 CTA로 사용한다.
- 캐시된 한도 상태는 업그레이드 성공 또는 주기별 초기화(일간 0시, 월간 월초)에 자동 만료된다.
- **Phase 1 범위**: 서버 캐시 없이 API 응답 기준으로만 한도 상태를 판별하고, `5/5` 도달 및 초과 요청 시 일관된 토스트/CTA를 노출하는 데 집중한다. 캐시/백엔드 확장은 Phase 2(V2) 백로그로 이관한다.

## 2. 메시지/토스트 UX 규칙
1. **트리거**
   - 백엔드 응답 `code === -9` 혹은 UsageLimitMeta.used ≥ limit일 때 즉시 `notifyUsageLimit` 호출한다.
   - “5/5” 시점, 즉 사용량이 정확히 한도에 도달하는 요청에서도 토스트를 노출해 사용자가 초과 시나리오를 명확히 인지하도록 한다.
   - ViewModel 레벨(예: `nplaceRankTrackViewModel`, `nplaceRankRealtimeViewModel`)에서는 최근 `limitMeta`만 저장해 토스트 카피를 재사용하며, API 호출 자체는 막지 않는다.
2. **토스트 카피(react-hot-toast 기반)**
   - 헤드라인: `한도 초과 · {기능 레이블}` + `({used}/{limit})` 표시.
   - 본문: “등록 가능한 수량을 모두 사용했습니다. 업그레이드하면 더 이용할 수 있어요.” (기존 공통 카피 유지, CTA 강조).
   - CTA 버튼: `업그레이드 하기` → `/membership?from=usage-limit&usageType={usageType}`.
   - 15초 쿨다운(`TOAST_COOLDOWN_MS`)은 유지하되, 캐시 상태 true일 때 `notifyUsageLimit(..., { force: true })`로 재노출 허용.
3. **배너/빈상태**
   - 각 화면 상단에 `UsageLimitBanner`(신규) 컴포넌트를 추가해 토스트 이후에도 업셀 CTA가 남도록 한다.
   - 빈상태에는 한도 도달 사유 + 업그레이드 버튼 + FAQ(일간/월간 리셋 규칙) 링크를 포함한다.
4. **문구 현지화**
   - 일간 기능은 “오늘 0시 기준 초기화” 배지, 월간 기능은 “매월 1일 리셋” 배지.
   - `ServiceSortDisplayRule.limitUnit`에 `"회/일"`, `"건/월"` 등 단위 문자열을 정의해 재사용한다.

## 3. CTA & 경로 전략
- 기본 CTA URL: `/membership` (SSR). 필요 시 `UsageLimitMeta.recommendedPlanId`를 받아 `/membership/{id}`로 deep-link.
- CTA 버튼 속성
  - Label: “업그레이드 하기” (토스트/배너), “플랜 비교하기” (빈상태/모달).
  - Secondary CTA: “오늘 사용량 확인” → `/profile?tab=usage` (일간/월간 Gauge 확인).
- Tracking: `data-analytics="usage-limit-cta"` + `data-usage-type` 속성으로 이벤트 로깅.

## 4. Phase 1 동작 원칙 (캐시 미적용)
- 모든 요청은 백엔드에 전달하고, 응답에 포함된 `limitMeta` 값으로 토스트/CTA를 노출한다.
- `5/5`에 도달하는 첫 요청과 초과 요청 모두에서 동일한 API 경고(`code === -9`)를 활용해 `notifyUsageLimit(..., { force: true })`를 호출한다.
- 프론트는 in-memory `lastToastTimestamp`만 사용해 스팸을 방지하며, 추가적인 Zustand/localStorage 캐시는 Phase 2에서 도입한다.
- API가 연속으로 실패하더라도 UI에 남아있는 배너/토스트가 업셀 경로를 유지하도록, ViewModel 레벨에서 “마지막 실패 상태”만 기록한다.

### 4-1. Phase 2(V2) 캐싱 & 만료 백로그
| 구분 | 캐싱 위치 | 키 | 만료 조건 | 설명 |
| --- | --- | --- | --- | --- |
| 프론트 토스트 | in-memory (`lastToastTimestamp`) | `{usageType}-limit-reached` | 15초 | 같은 기능에서 토스트 스팸 방지. (현행) |
| 프론트 한도상태 | `useUsageLimitStore` (신규, Zustand) | `{usageType}` | 업그레이드 이벤트, TTL, 페이지 리셋 | true이면 API 호출 막고 CTA만 노출. |
| 백엔드 응답 캐시 | `UsageLimitCache` (서버; Redis or in-memory) | `{memberId}:{usageType}:{periodKey}` | `periodType === DAILY` → 다음 0시, `MONTHLY` → 다음 월초, 또는 `upgrade_at` | 첫 실패 이후 동일 요청을 short-circuit. |
| 클라이언트 Persist | localStorage `usageLimit:v1` | `{usageType}` payload (limit, used, expiresAt) | `expiresAt`, upgrade 성공, manual reset | 새로고침 후에도 바로 배너 노출. |

- (V2) 백엔드 캐시 hit 시 429-style 응답을 즉시 돌려주고, `limitMeta`를 cache와 함께 반환한다.
- (V2) 업그레이드/추가 구매 성공 시 서버가 `UsageLimitCache.invalidate(memberId)`를 호출하고, 프론트는 `/v1/membership/current` 재조회 시 store를 초기화한다.
- (V2) `UsageResetService` (월간)와 일간 자동 리셋(0시) 이벤트에서 동일 키를 삭제한다.

## 5. 시스템 연동 흐름 (Phase 1)
1. **한도 도달(5/5 포함)**
   - API가 `{ code: -9, data: limitMeta }`를 반환하면 `notifyUsageLimit(limitMeta)`로 토스트/CTA를 즉시 노출하고, 동일 뷰에서 배너/빈상태를 렌더링한다.
   - ViewModel은 `lastLimitMeta` 정도만 저장하고, API 호출 자체는 차단하지 않는다.
2. **초과 요청 처리**
   - 사용자가 다시 요청을 시도하면 API를 그대로 호출하고, 재차 `code === -9`가 올 때마다 `notifyUsageLimit(..., { force: true })`로 동일 메시지를 반복 노출한다.
   - 쿨다운은 in-memory timer로만 제어하며, 별도의 캐싱/로컬스토리지는 사용하지 않는다.
3. **만료/재사용**
   - 백엔드가 일간/월간 리셋 이후 정상 응답을 보내면 자동으로 다시 요청이 가능해진다.
   - 업그레이드 성공 시 `/v1/membership/current` 재호출을 통해 UI가 최신 한도 값을 다시 받도록 안내한다.

### 5-1. Phase 2(V2) 확장 목표
- ViewModel 레벨에서 `usageLimitStore`(Zustand)로 재요청 차단.
- 백엔드 Redis 캐시 hit 시 `blocked: true` 응답을 내려 API/인프라 비용을 절감.
- localStorage `usageLimit:v1`를 활용해 새로고침 이후에도 배너 상태를 유지.

## 8. 오픈 이슈
1. **일간/월간 혼합 화면 copy 정합성** – 다국어 필요 여부 검토.
2. **캐시 TTL 정확도** – 일간 기능은 로컬 타임존 기준 0시, 백엔드는 KST 기준인지 확인 필요.
3. **업그레이드 감지 신뢰도** – PortOne 완료 후 `currentMembership`가 즉시 최신화되지 않는다면 invalidate webhook 필요.

## 9. (Phase 2) 백엔드 조치 사항
Phase 1에서는 서버 캐시 작업을 수행하지 않으며, 아래 항목은 모두 후속 백로그로 남겨둔다.
1. **limitMeta 스키마 통일** – 모든 `/api/*` 라우트에서 한도 초과 시 `{ usageType, used, limit, periodType, periodKey, recommendedPlanId }`를 포함하도록 DTO를 표준화하고, `code === -9` 규약을 유지한다.
2. **UsageLimitCache 구축** – Redis(or KV)에 `{memberId}:{usageType}:{periodKey}` 키로 캐시 저장, TTL은 일간 → 다음 0시, 월간 → 다음 1일 00:00로 설정하고 `blocked: true` 응답 시 캐시 hit 여부를 반환한다.
3. **Invalidate 훅 제공** – 멤버십 업그레이드/추가 구매/관리자 증설 시 `UsageLimitCache.invalidate(memberId, usageType)`를 호출하는 Webhook 혹은 내부 이벤트를 구현한다.
4. **Reset 스케줄러 업데이트** – `UsageResetService`가 월간 한도만 초기화하는 현 구조를 유지하되, 일간 기능은 서버 cron(0시)에서 캐시 키를 삭제하고 프론트에 전달할 `resetAt` 값을 포함시킨다.
5. **App Route 응답 확장** – Next.js `src/app/api/*` 라우트에서 캐시 hit 시 HTTP 200 + `blocked: true`를 내려 프런트가 재호출 차단 여부를 판별할 수 있게 한다.
6. **모니터링/로깅** – limit 초과 이벤트를 Sentry/Datadog에 기록하고, CTA conversion을 위해 `/membership` 진입 referrer(`from=usage-limit`)를 서버 로그에 추가한다.

## 10. Phase 1 구현 현황 (2025-12-05)
- **`UsageLimitMeta` 확장** (`src/types/api.ts`): 백엔드 메타 규격에 맞춰 `periodType`, `periodKey`, `recommendedPlanId`를 수용해 `5/5` 복구와 딥링크 CTA 계산이 가능하다.
- **공통 토스트 개선** (`src/utils/usageLimitNotifier.tsx`): 기능 라벨·CTA URL·analytics 속성을 통일하고, `/membership?from=usage-limit&usageType=…&planId=…` 쿼리로 바로 업셀 페이지에 진입한다.
- **Realtime/Track ViewModel 업데이트** (`src/viewModel/nplace/realtime/*`, `.../track/*`, `.../keword/*`): `code === -9` 응답의 `meta`를 그대로 `notifyUsageLimit`에 전달해 5/5 및 초과 요청에서 동일한 카피/CTA를 보장한다.
