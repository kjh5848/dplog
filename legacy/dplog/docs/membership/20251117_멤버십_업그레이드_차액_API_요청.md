# 멤버십 업그레이드 차액 API 요청 (2025-11-17)

프런트에서 업그레이드 확인 모달에 “실제 청구될 차액/환불 금액”을 사전에 보여주기 위해, `/v1/subscription/upgrade` 백엔드 API에 **견적(preview)** 응답이 필요하다. 기존에는 선택한 플랜의 정가만 노출되므로 잔여 기간 공제 로직(UpgradeCalculator)과 불일치했다.

## 1. 요구 사항 요약
- 업그레이드 CTA 클릭 시 **결제 전**에 차액 정보를 가져와야 함.
- 프론트에서 임의 계산이 불가하므로 백엔드 계산값을 그대로 사용.
- 연간→월간 상향 전환의 환불/재결제 플로우도 동일 응답으로 해결.
- 최종 결제 API는 기존 `/v1/subscription/upgrade`를 유지하되, 미리보기 플래그를 추가하거나 별도 견적 API를 제공.

## 2. 제안 인터페이스

### 옵션 A: 기존 업그레이드 API 확장
```
POST /v1/subscription/upgrade
Content-Type: application/json

{
  "targetMembershipLevel": 3,
  "billingCycle": "MONTHLY" | "YEARLY",
  "preview": true   // true면 결제/환불을 실행하지 않고 견적만 반환
}
```

- `preview=true`일 경우
  - 결제/환불/UsageReset은 수행하지 않는다.
  - `paymentRequested`는 항상 `false`.
  - 견적 데이터(`differenceAmount`, `refundableAmount`, `newChargeAmount`, `message`, `upgradeType`, `nextBillingDate`, `currentPlanPrice`, `targetPlanPrice`, `remainingDays`)를 모두 채워 반환.
- `preview`가 없거나 `false`면 기존 로직대로 업그레이드를 실행.

### 옵션 B: 전용 견적 API 신설
```
POST /v1/subscription/upgrade/quote

{
  "targetMembershipLevel": 3,
  "billingCycle": "MONTHLY" | "YEARLY"
}
```

- 응답 스키마는 `SubscriptionUpgradeResponse`와 100% 동일.
- 요청 파라미터는 업그레이드와 동일하므로 프론트 재사용 용이.
- 실제 업그레이드 시에는 `/v1/subscription/upgrade`를 그대로 사용.

## 3. 응답 스키마 제안
`SubscriptionUpgradeResponse` 확장 항목 (preview/실행 공통 사용):

| 필드 | 타입 | 설명 |
| --- | --- | --- |
| `differenceAmount` | number | 플랜 업그레이드 시 즉시 결제될 금액(잔여 기간 차감 포함) |
| `refundableAmount` | number? | 연→월 상향 시 환불 예정액 |
| `newChargeAmount` | number? | 연→월 상향 시 새로 청구될 월간 금액 |
| `refundableCurrency` | string? | 환불 금액 화폐 단위(기본 KRW) |
| `refundPaymentId` | string? | 환불 트랜잭션 식별자(실행 시) |
| `upgradeType` | `'PLAN' | 'PERIOD' | 'YEARLY_TO_MONTHLY'` | 계산된 업그레이드 유형 |
| `nextBillingDate` | string? | 업그레이드 적용 후 다음 결제일(ISO 8601) |
| `remainingDays` | number? | 현재 주기 잔여 일수 |
| `currentPlanPrice` | number? | 현재 플랜 기준 가격 |
| `targetPlanPrice` | number? | 목표 플랜 기준 가격 |
| `paymentRequested` | boolean | 결제/환불 실행 여부 (`preview`면 false) |
| `message` | string? | 고객 안내용 메시지(예: “환불 157,500원 + 결제 39,000원”) |

## 4. 프런트 사용 플로우
1. 고객이 업그레이드를 선택하면 `preview=true`로 견적을 요청.
2. 응답의 `differenceAmount`/`refundableAmount`/`newChargeAmount`를 모달과 CTA 하단에 노출.
3. 사용자가 동의 후 진행하면 `preview` 없이 다시 업그레이드 API 호출.
4. 성공 응답에서는 동일 필드를 이용해 토스트/구독 카드 업데이트.

## 5. 기타 고려 사항
- Rate limit: 동일 사용자에 대해 1초 내 중복 preview 요청은 debounce 가능하도록 요청당 고유 ID 필요 없음.
- 오류 메시지: 백엔드에서는 현재처럼 `message` 필드를 채워 프론트가 그대로 노출할 수 있도록 한다.
- 보안: preview는 실제 상태를 변경하지 않으므로 CS·QA에서는 안전하게 테스트 가능.
- 로깅: preview 호출도 UpgradeCalculator 로그에 남겨 계산 이력을 추적할 수 있게 해달라.

위 명세대로 백엔드가 구현해주면, 프런트는 즉시 차액/환불 안내를 제공할 수 있으며 업그레이드 정책 문서(`20251117_프런트엔드_멤버십_업그레이드_조치.md`)의 요구사항을 충족할 수 있다.
