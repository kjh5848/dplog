# Membership Profile Summary API-front

## 배경
- `/profile` 화면에서 현재 구독 정보와 멤버십 이력을 보여주기 위해 `GET /v1/membership/current`, `GET /v1/user/{id}/membership` 두 엔드포인트를 각각 호출 중.
- 전체 이력 응답을 매번 내려받는 구조라 트래픽과 데이터 가공 비용이 높고, 다른 화면에서도 동일 조합을 반복하게 될 가능성이 있음.

## 목표
1. 현재 활성 멤버십, 결제 주기, 잔여 일수 등 프로필에 필요한 핵심 정보를 한 번에 받는다.
2. 멤버십 이력은 **무한 스크롤**을 지원하도록 커서 기반 페이징 정보를 포함해 단계적으로 내려준다.
3. 업그레이드·다운그레이드 로직에서 참조할 비교 정보(레벨, 주기, 가격 등)와 **주요 기능 사용량(예: 트랙 슬롯, 보고서 생성 수)**을 함께 제공한다.
4. 최신 설계 반영: `payments` 배열까지 포함한 단일 응답으로 프로필에 필요한 데이터(현재/이력/사용량/결제)를 완성한다.

## Usage / Metrics 규칙 (Backend Plan 연동)
- 참조 문서: `docs/membership/profile-summary-backend-plan.md`
- 모든 사용량 항목은 `{ used: number; limit: number }` 구조를 사용한다.
- 지표 정의 및 타입/단위는 아래 표를 기준으로 유지하며, 신규 지표 추가 시 동일 표에 업데이트 후 공유한다.

| 지표 key | 설명 | 단위 | 타입 | 소수 처리 |
| --- | --- | --- | --- | --- |
| `trackKeywords` | 트랙 슬롯 사용 건수 | 건 | 정수(`int`) | 소수 없음 |
| `reportExports` | 보고서 내보내기 횟수 | 회 | 정수(`int`) | 소수 없음 |
| `realtimeQueries` | 실시간 조회 호출 수 | 건 | 정수(`int`) | 소수 없음 |
| *(추가 시 예: `alertMessages`)* | 알림 발송 건수 | 건 | 정수(`int`) | 소수 없음 |

## 프론트 개발 계획
1. **타입/계약 정의**
   - `MembershipProfileSummaryResponse`와 하위 인터페이스(`MembershipUsage`, `MembershipHistoryPageInfo` 등)를 `src/types/membership.ts`에 초안으로 추가.
   - 응답 enum(`billingCycle`, `membershipState`)을 BE와 합의된 값으로 업데이트하고, 사용량 지표별 단위/타입을 명확히 지정.
2. **API 계층 정비**
   - `src/app/api/membership/profile-summary/route.ts`(Next.js App Route)에서 백엔드 `/v1/membership/profile-summary` 호출을 프록시하고, 커서·limit 파라미터를 안전하게 전달.
   - 클라이언트 재사용을 위해 `src/lib/services/membership.ts`에 `fetchMembershipProfileSummary({ cursor, limit })`를 추가하고, payments 필드를 포함한 새 타입을 사용한다.
3. **/profile 화면 리팩터링**
   - 기존 `Promise.all` 호출을 제거하고 `/api/membership/profile-summary`만 사용한다.
   - `current`, `history`, `usage`, `payments`를 모두 매핑해 초기 상태를 구성하며, 결제 기록 섹션은 `payments` 배열을 바로 활용한다.
   - 히스토리 섹션은 커서 기반 무한 스크롤(“더 보기”)을 유지하고, 결제 기록은 최근 3건을 표시하며 “새로고침” 시 `/profile-summary`를 재호출한다.
   - 로딩/에러/empty 상태 UI와 재시도 버튼을 구성해 페이징 도중 오류가 나도 기존 데이터를 유지한다.
4. **공유 데이터 재사용**
   - `/payment` 등 다른 화면에서 사용하는 멤버십 비교 정보(레벨, 월/연 요금 등)를 전역 상태(Zustand) 또는 React Query 캐시에 저장해 재조회 최소화.
   - PortOne 결제 플로우에서 필요한 `current` 정보가 있으면 신규 응답을 활용하는 방향을 검토.
5. **QA 준비**
   - 스토리북 또는 mocked provider를 사용해 `usage`, `history` 길이에 따른 다양한 케이스를 시각적으로 점검.
   - 통합 후 `npm run lint`, `npm test` 실행 및 Playwright 시나리오에 무한 스크롤/재시도 경로를 추가.
   - 네트워크 오류, `hasNextPage=false`, 빈 히스토리 등 엣지 케이스별 스냅샷을 QA 문서에 기록.
   - **사용량 UX 대비**: 백엔드 `ServiceSort` ↔ usage 키 매핑이 확정되면 `Record<ServiceSortKey, MembershipUsageMetric>` 형태로 타입을 재정비하고 UI copy를 동기화한다.
   - 사용량 데이터가 비거나 집계 실패 시 기본값(예: `0 / 제한` 또는 “데이터 없음”) 중 어떤 표현을 할지 시나리오를 마련해 둔다.
   - 요금제별 한도 노출이 필요해지는 경우에 대비해 툴팁·그래프 등 UI 옵션별 와이어프레임 초안을 준비해 반복 작업을 줄인다.
   - 결제 기록 UI는 `payments` 배열(최근 N건)을 기반으로 렌더링하고, 새로고침 동작은 `/profile-summary` 재호출로 일원화한다.

## 프론트 변경 사항 공유 (백엔드 usage 설계 반영)
- usage 키는 백엔드 `ServiceSort`와 1:1 매핑된 구조로 넘어옵니다. 새 지표가 추가될 때 enum과 타입 정의(`Record<ServiceSortKey, MembershipUsageMetric>` 등)를 동시에 갱신할 수 있도록 준비합니다.
- `limit` 값은 `MembershipDetailEntity.limit_count`를 그대로 전달하며, `null`은 “무제한”으로 표시됩니다. 무제한 표기 방식(레이블, 아이콘 등)은 프론트에서 결정합니다.
- `used` 값은 USE_LOG 기반 월간 집계가 기본입니다. 향후 기간을 바꿔야 할 경우 협의 후 API 파라미터 확장으로 대응할 수 있도록 고려합니다.
- 위 합의에 따라 백엔드 usage 로직 구현을 진행합니다. 프론트는 응답 구조 변화에 맞춰 타입과 UI 반영을 준비해 두세요.
- 추가 공유: `ServiceSort` 기본 키(`trackKeywords`, `realtimeQueries`, `reportExports`) 외에 `extras` 객체 하위로 `relationKeywords`, `keywordLookups`, `shopRegistrations`, `blogQualityChecks`, `reviewReplies` 등이 추가될 수 있습니다. 타입 정의와 UI 레이블을 선반영해 주세요.
- `limit = null`은 “무제한” 의미로 확정되었습니다. 동일 표기 규칙으로 백엔드도 응답을 유지하므로, 프론트 UI 문구/아이콘을 결정한 뒤 공유 바랍니다.
- `used`는 “당월 집계” 기준입니다. 다른 기간 단위가 필요하면 파라미터 확장을 통해 협의 예정입니다.
- **결제 이력 통합**: `/profile-summary` 응답에 `payments` 배열을 추가하여 최근 결제 내역을 함께 반환합니다. 기본 구조는 아래와 같습니다.
  ```json
  {
    "payments": [
      {
        "id": "pay_202501010001",
        "amount": 29000,
        "planName": "실속 점주",
        "paymentMethod": "CARD",
        "status": "DONE",
        "paidAt": "2025-01-01T12:34:56+09:00",
        "receiptUrl": "https://..."
      }
    ]
  }
  ```
  - 기본적으로 최근 N건(예: 3건)을 내려주고, 추가 페이지 지원이 필요하면 추후 파라미터 확장으로 협의합니다.
  - 프론트는 `payments` 배열을 사용해 프로필 내 결제 기록 UI를 렌더링합니다.

### QA 경계 케이스 (Backend Plan 94~101행 참조)
- `historyLimit = 1` 요청 시 정상 응답 및 `nextCursor` 생성 확인.
- `historyLimit` 최대값(예: 50) 요청 처리.
- 커서 미지정 첫 호출 → `hasNextPage` 및 `nextCursor` 응답 확인.
- 마지막 페이지에서 `hasNextPage = false`, `nextCursor = null`.
- 잘못된 커서 포맷(예: 임의 문자열) 전달 시 400 처리 및 UI 에러 노출.
- 사용량 데이터가 비거나 null일 때 기본값 전략(`{ used: 0, limit: 0 }` 등) 검증.

## 제안 API
- **Endpoint**: `GET /v1/membership/profile-summary`
- **Query parameters**
  - `userId` (optional, 관리자/백오피스용)
  - `historyLimit` (optional, 기본 10)
  - `historyCursor` (optional, 커서 기반 무한 스크롤 토큰)
- **Response (초안)**
  ```json
  {
    "code": 0,
    "message": "success",
    "data": {
      "current": {
        "membershipId": 11,
        "membershipName": "실속 점주",
        "billingCycle": "YEARLY",
        "startDate": "2025-10-23",
        "endDate": "2026-10-23",
        "remainingDays": 365,
        "compareLevel": 2,
        "comparePriceMonthly": 19000,
        "comparePriceYearly": 182400
      },
      "history": [
        {
          "membershipId": 11,
          "membershipName": "실속 점주",
          "billingCycle": "MONTHLY",
          "startDate": "2025-10-23",
          "endDate": "2025-10-23",
          "state": "EXPIRED"
        }
      ],
      "historyPageInfo": {
        "nextCursor": "2025-10-23T00:00:00Z#11",
        "hasNextPage": true
      },
      "catalogSummary": [
        {
          "membershipId": 11,
          "level": 2,
          "priceMonthly": 19000,
          "priceYearly": 182400,
          "isPopular": true
        }
      ],
      "usage": {
        "trackKeywords": {
          "used": 8,
          "limit": 10
        },
        "reportExports": {
          "used": 2,
          "limit": 5
        },
        "realtimeQueries": {

          "limit": 100
        }
      }
    }
  }
  ```
- **에러 코드**
  - `401`: 인증 필요
  - `404`: 해당 유저 데이터 없음
  - `5xx`: 내부 오류
