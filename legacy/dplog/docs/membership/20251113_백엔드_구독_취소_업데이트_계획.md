# Subscription Cancel Backend Update Plan

백엔드 해지 로직을 정책(해지 요청 후 결제 기간 동안 이용 유지, 부분 환불 없음, 다음 결제부터 자동 청구 중단)에 맞추기 위한 개편 사항을 정리합니다.

## 1. 현재 구현 요약
- `PaymentServiceApiV1.cancelSubscription`
  - PortOne 예약(`payment-schedules`) 즉시 삭제
  - PortOne 빌링키 즉시 삭제
  - `SubscriptionEntity.status = CANCELLED`, `endDate = 오늘`
  - 활성 `MembershipUser`를 `EXPIRED`로 전환 (즉시 혜택 제거)
- 결과적으로 해지 요청 즉시 이용이 중단되어 정책과 불일치.

## 2. 정책 대비 구현 차이
| 항목 | 정책 요구 | 현재 구현 | 조치 |
| --- | --- | --- | --- |
| 서비스 이용 기간 | 결제된 기간 종료까지 유지 | 즉시 종료 | 상태 전환 로직 개선 |
| 환불/차액 | 없음 | 없음 | - |
| 다음 청구 | 자동 중단 | 예약/빌링키 즉시 삭제 | 예약 삭제 OK, 빌링키 지연 삭제 검토 |
| 멤버십 상태 | 기 결제 기간동안 ACTIVE 유지 | 즉시 EXPIRED | `PENDING_CANCEL` 상태 도입 |
| 해지 사유 로깅 | 필요 | 이미 로그 존재 | - |

## 3. 개편 방향
### 3.1 상태 전환
- 해지 요청 시 구독/멤버십을 비활성화하지 않고 “취소 예약” 상태로 전환.
  - `SubscriptionStatus`에 `PENDING_CANCEL` 또는 유사 플래그 도입.
  - `SubscriptionEntity.cancelScheduledAt` 등 종료 예정일(`nextBillingDate`) 저장.
  - `MembershipUser`는 종료일까지 `ACTIVATE` 유지.
- 스케줄러에서 다음 결제일 도달 시 최종 해지 처리:
  - PortOne 빌링키/예약 삭제
  - `SubscriptionEntity.status = CANCELLED`, `endDate = today`
  - `MembershipUser` → `EXPIRED`, 로그 기록

### 3.2 PortOne 연동
- 예약은 즉시 삭제해도 정책과 일치하며 다음 결제 차단.
- 빌링키는 즉시 삭제 유지(재구독 시 신규 발급 필요) 또는 종료 시점까지 유지 중 선택.

### 3.3 스케줄러/배치
- `SubscriptionServiceApiV1.processDueSubscriptions` 혹은 별도 배치에서 `PENDING_CANCEL && nextBillingDate == today` 조건을 처리.
- 처리 단계:
  1. PortOne 빌링키/예약 삭제 (선택한 전략에 맞게)
  2. `SubscriptionEntity` 상태/종료일 업데이트
  3. `MembershipUser` 상태 `EXPIRED`, 로그 추가
  4. 운영 로그(Sentry 등) 기록

### 3.4 DTO/응답
- 해지 요청 API 응답에 예정 종료일과 안내 메시지 포함:
  ```json
  {
    "status": "PENDING_CANCEL",
    "effectiveDate": "YYYY-MM-DD",
    "message": "다음 결제일부터 자동 해지됩니다."
  }
  ```

### 3.5 UI/CS
- 사용자 화면: “다음 결제일까지 이용 가능” 상태 명확히 표기.
- CS/어드민: 종료 예정일, 해지 사유 로그 조회 지원.

## 4. 구현 체크리스트
- `SubscriptionEntity`에 해지 예약 필드 추가 (`cancelScheduledAt` 등) 또는 기존 필드 재활용.
- `cancelSubscription` 로직 수정:
  - PortOne 예약 삭제 유지.
  - 상태는 `PENDING_CANCEL`, 종료일/멤버십 endDate는 건드리지 않음.
  - 해지 사유 로그 기록 유지.
- 스케줄러/배치 보강:
  - 조건 충족 시 최종 해지 및 빌링키 삭제.
  - 스케줄러 테스트 (`processDueSubscriptions`) 업데이트.
- 응답 DTO/테스트:
  - 응답 메시지/상태 검증.
  - 해지 후에도 멤버십이 유지되는지 단위/통합 테스트 추가.
- 문서/운영:
  - 정책 문서 및 프런트 안내 문구 업데이트.
  - CS 매뉴얼에 종료 예정일/사유 확인 절차 명시.
