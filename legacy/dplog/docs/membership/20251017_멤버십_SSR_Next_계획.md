# Membership SSR Plan (Next App Router + Repository-Only Fetch)

## Goals
- Use Next(App Router) SSR for first paint and SEO while keeping all network calls inside Model Repositories.
- Replace “Pricing/Plan” flow with “Membership” flow using new backend APIs.
- Enforce server-side eligibility checks before payment (pre-register), and make the client only orchestrate.

## High-Level Flow
1) /membership (SSR): show current membership + membership list (upgrade candidates)
2) /membership/[id] (SSR): fetch detail + eligibility; render CTA depending on `canPurchase`
3) Pre-register payment (Server API route or direct backend): final server eligibility check + amount/orderName resolution
4) /payment (Client): PortOne/Toss UI; confirm; backend completes membership activation; redirect

## Route Structure (Next App Router)
- /app/(home)/membership/page.tsx → Server Component (SSR)
- /app/(home)/membership/[id]/page.tsx → Server Component (SSR)
- /app/(home)/payment/page.tsx → Client Component (PortOne SDK flow remains)
- /app/(home)/(membership)/layout.tsx → Shared layout + server guard (optional)
- /app/(home)/(membership)/loading.tsx, error.tsx → Route-level UX
- /middleware.ts → Optional global guard for /membership|/payment

## Repository Design (All fetch here)
- File: `src/model/MembershipRepository.ts`
  - `getMembershipList()` → GET /v1/membership/list (credentials/cookies included)
  - `getMembership(id: string)` → GET /v1/membership/{id}
  - `getCurrentMembership()` → GET /v1/membership/current
- Fetch rules
  - Server-side: same-origin fetch carries cookies automatically. If cross-domain, forward `Cookie` header with `next/headers` `cookies()`.
  - Client-side: use `credentials: 'include'`.
  - Normalize response `{ code: number, message: string, data: ... }`.

## Backend Contracts (per backend note)
- GET /v1/membership/list
  - 200: `{ code: 0, message: "success", data: { membershipList: [ { id, name, point, price, description, isPopular, colorScheme, sortOrder } ] } }`
  - 401: `{ code: -1, message: "로그인이 필요한 서비스입니다." }`
- GET /v1/membership/{id}
  - 200: `{ code: 0, message: "success", data: { id, name, point, price, description, ..., currentMembershipId, currentMembershipName, currentMembershipPrice, canPurchase, compareResult, message } }`
  - 404: `{ code: -3, message: "존재하지 않거나 비활성화된 멤버십입니다." }`
- GET /v1/membership/current
  - 200: `{ code: 0, message: "success", data: { ...currentMembership } } | { code: 0, data: null }`
- Payment pre-register (recommended)
  - POST /v1/payments/pre-register
  - Request: `{ membershipId: string, period: 'MONTHLY' | 'YEARLY' }`
  - Response: `{ code: 0, data: { paymentId: string, orderName: string, amount: number } }`
  - Server re-checks `canPurchase` and price; return 409/400 if invalid.

Notes
- If membership supports multiple periods, expose `priceMonthly`/`priceYearly` or `periodPrices: { MONTHLY, YEARLY }` in list/detail.
- Decide eligibility on period-aware price if applicable.

## Data Flow: SSR vs Client
- SSR pages
  - /membership: `getCurrentMembership()` + `getMembershipList()` → render cards with server data
  - /membership/[id]: `getMembership(id)` → interpret `canPurchase`, `compareResult`, `message` to render CTA
- Client pages
  - /payment: reads `membershipId` + `period` (query); gets detail if needed only for display; calls pre-register; orchestrates PortOne/Toss flows.

## Auth & Cookies
- Use route/layout server guard for /membership and /payment; `redirect('/login?redirect=...')` if unauthenticated.
- Ensure cookie/session continuity:
  - Same-site: SSR fetch carries cookies.
  - Cross-site: forward Cookie header on server fetch; client fetch uses `credentials: 'include'`.

## Error Handling
- HTTP to UX mapping
  - 401: redirect to login (server) or prompt and push (client fallback)
  - 404 (-3): show message and link back to /membership
  - 409/400 on pre-register: show server `message`, return to detail page
- Route-level error boundaries: `error.tsx` per segment; `loading.tsx` for skeleton.

## Migration Plan (Phased)
1. Add MembershipRepository and wire fetch rules
2. Replace pricing data source with membership list on SSR page `/membership`
3. Add `/membership/[id]` SSR detail with eligibility gating
4. Switch payment param from `planId` to `membershipId`; keep `period`
5. Add pre-register call before invoking PortOne; use server amount/orderName
6. Add server guard via layout or middleware; add loading/error files
7. QA pass with scenarios (guest, current, upgrade, equal/lower, network)

## End-to-End Request/Response Control (Step-by-Step)

Step 0. Pre-req
- Ensure frontend and backend base URLs and cookies are aligned (domain, SameSite, secure).
- Populate `.env` (NEXT_PUBLIC_API_URL, PortOne keys, etc.).

Step 1. Load current membership (SSR)
- Page: `/membership`
- Action (server): `MembershipRepository.getCurrentMembership()`
  - Server fetch: `fetch(`${API}/v1/membership/current`, { headers/cookies })`
  - Expect: `{ code: 0, data: current | null }` or 401
- Control:
  - If 401 → `redirect('/login?redirect=/membership')`
  - Else render current badge

Step 2. Load membership list (SSR)
- Action (server): `MembershipRepository.getMembershipList()`
  - Expect: `{ code: 0, data: { membershipList: [...] } }`
- Control:
  - Sort by `sortOrder` (if provided). Highlight `isPopular`.
  - Render cards with price/description/features/colors.

Step 3. Select membership → navigate to detail (SSR)
- Link: `/membership/[id]`
- Action (server): `MembershipRepository.getMembership(id)`
  - Expect: `{ code: 0, data: { canPurchase, compareResult, message, ... } }`
- Control:
  - Render server CTA:
    - canPurchase=true → show "결제 진행" button (enabled)
    - canPurchase=false → show message; disable CTA
  - If 404 (-3) → show message and link back

Step 4. Decide period and start pre-register (client or server action)
- On CTA click (client): choose period (default MONTHLY/YEARLY from detail)
- Call backend pre-register
  - POST `${API}/v1/payments/pre-register`
  - Body: `{ membershipId, period }` (no client-side amount)
  - Expect: `{ code: 0, data: { paymentId, orderName, amount } }` or 409/400 with message
- Control:
  - If 409/400 → show server message, navigate back to `/membership/[id]`
  - Else: proceed to PortOne/Toss with `paymentId`, `orderName`, `amount`

Step 5. Invoke PortOne/Toss (client)
- Leverage `usePriceViewModel` + repository helpers to perform SDK auth/charge
- Handle success/fail
  - Success → call backend complete endpoint (existing), then route to success/dashboard
  - Fail → route to fail page with params; guide user back to payment or membership

Step 6. Post-payment confirmation
- Backend activates membership (sets endDate/level/etc.)
- Front loads `/v1/membership/current` on profile/dashboard to reflect new state
