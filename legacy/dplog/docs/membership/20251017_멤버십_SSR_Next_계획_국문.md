# 멤버십 SSR 계획 (Next App Router + 레포지토리 전용 Fetch)

## 목표
- Next(App Router) 기반 SSR로 초기 렌더 품질/SEO를 개선하고, 모든 네트워크 호출은 Model 레포지토리에서만 수행.
- 기존 요금제(Pricing/Plan) 플로우를 멤버십(Membership) 플로우로 교체.
- 결제 전 백엔드 사전등록(pre-register) 단계에서 최종 검증을 수행하고, 프론트는 오케스트레이션만 담당.
- 비로그인 사용자도 멤버십 목록을 확인할 수 있도록 공개 목록 API(공개 엔드포인트)를 사용.

## 고수준 흐름
1) /membership (SSR): 현재 보유 멤버십 + 멤버십 목록 표시(업그레이드 후보)
2) /membership/[id] (SSR): 단건 상세 + 업그레이드 가능 여부(canPurchase) 확인, CTA 결정
3) 결제 사전등록(서버 API 라우트 또는 백엔드 직접): 최종 검증 + 결제 금액/주문명 확정
4) /payment (클라이언트): PortOne/Toss UI 결제 → 백엔드 완료 처리 → 리다이렉트

## 라우트 구조 (Next App Router)
- `/app/(home)/membership/page.tsx` → 서버 컴포넌트(SSR, 공개 접근 허용)
- `/app/(home)/membership/[id]/page.tsx` → 서버 컴포넌트(SSR, 상세는 로그인 정책에 따라 선택)
- `/app/(home)/payment/page.tsx` → 클라이언트 컴포넌트(PortOne SDK 유지, 보호 필요)
- `/app/(home)/(membership)/layout.tsx` → 세그먼트 공통 레이아웃(서버 가드는 `/payment`에만 적용 권장)
- `/app/(home)/(membership)/loading.tsx`, `error.tsx` → 라우트 단위 UX 표준화
- `/middleware.ts` → `/payment` 등 보호 경로만 전역 가드(선택)

## 레포지토리 설계 (모든 fetch는 여기에서만)
- 파일: `src/model/MembershipRepository.ts`
  - 목록은 현재 API Route가 담당하지만, 공통 설정은 `src/model/Api.ts`에서 일원화(`Api.baseUrl`, `Api.url(path)`, `Api.endpoints.*`).
  - `getMembership(id: string)` → GET `/v1/membership/{id}` (로그인 필요 정책일 수 있음)
  - `getCurrentMembership()` → GET `/v1/membership/current` (로그인 사용자 전용)
- Fetch 규칙
  - 서버 사이드: 동일 도메인이면 쿠키 자동 전달. 크로스 도메인이면 `next/headers`의 `cookies()`로 Cookie 헤더 전달.
  - 클라이언트: 
    - 공개 목록: `credentials` 불필요
    - 로그인 필요 API: `credentials: 'include'` 사용
  - 응답 표준화: `{ code: number, message: string, data: ... }`를 공통으로 처리.

## 백엔드 계약(요약)
- GET `/v1/membership/list`
  - 200: `{ code: 0, message: "success", data: { membershipList: [ { id, name, point, price, description, isPopular, colorScheme, sortOrder } ] } }`
  - 401: `{ code: -1, message: "로그인이 필요한 서비스입니다." }`
- GET `/v1/membership/{id}`
  - 200: `{ code: 0, message: "success", data: { id, name, point, price, description, …, currentMembershipId, currentMembershipName, currentMembershipPrice, canPurchase, compareResult, message } }`
  - 404: `{ code: -3, message: "존재하지 않거나 비활성화된 멤버십입니다." }`
- GET `/v1/membership/current`
  - 200: `{ code: 0, message: "success", data: { ...현재멤버십 } }` 또는 `{ code: 0, data: null }`
- POST `/v1/payments/pre-register` (권장)
  - 요청: `{ membershipId: string, period: 'MONTHLY' | 'YEARLY' }`
  - 응답: `{ code: 0, data: { paymentId: string, orderName: string, amount: number } }`
  - 서버에서 `canPurchase`와 가격을 최종 재검증. 실패 시 409/400.

참고
- 멤버십이 여러 기간을 지원한다면 `priceMonthly`/`priceYearly` 또는 `periodPrices: { MONTHLY, YEARLY }` 제공.
- 필요 시 기간별 가격 기준으로 canPurchase 판정.

- SSR 페이지
  - `/membership`: `getPublicMembershipList()`로 공개 목록 SSR 렌더 + (선택) `getCurrentMembership()` 시도(401이면 게스트로 처리)
  - `/membership/[id]`: `getMembership(id)`로 `canPurchase`/`compareResult`/`message` 기반 CTA SSR 결정(상세 공개 여부에 따라 로그인 유도 가능)
- 클라이언트 페이지
  - `/payment`: 쿼리 `membershipId` + `period` 사용. 표시용 상세는 필요 시 재조회. 사전등록 호출 후 PortOne/Toss 오케스트레이션.

## 인증/쿠키
- `/membership`는 공개 접근: 401이어도 리다이렉트하지 않고 게스트 모드로 SSR 렌더
- `/payment`는 보호 경로: 레이아웃/미들웨어에서 쿠키 검사 후 미인증 시 `redirect('/login?redirect=/payment')`
- 쿠키/세션 유지 보장:
  - 동일 사이트: SSR fetch가 쿠키 자동 포함.
  - 크로스 도메인: 서버 fetch에 Cookie 헤더 전달, 클라이언트 fetch는 `credentials: 'include'`.

## 오류 처리
- HTTP → UX 매핑
  - 401: 서버에서 로그인 리다이렉트(클라이언트는 보조 안내)
  - 404(-3): 상세에서 친화적 메시지 + 목록으로 돌아가기 링크
  - 409/400(사전등록): 서버 메시지를 그대로 표시하고 결제 차단
- 라우트 단위 에러 바운더리: 세그먼트 `error.tsx`, 로딩은 `loading.tsx`로 스켈레톤 처리.

## End-to-End 단계별 제어(요청/응답까지)

Step 0. 준비
- 프론트/백엔드 베이스 URL/쿠키 정책 확인(도메인, SameSite, Secure).
- `.env` 구성(NEXT_PUBLIC_API_URL, PortOne 키, Redirect URI 등). 개발 환경에서는 HTTPS 자체서명 인증서 이슈를 피하기 위해 `http://localhost:8080` 같이 HTTP를 권장.

- 페이지: `/membership`
- 서버 액션: `fetch(Api.endpoints.CLIENT.MEMBERSHIP.LIST)` (내부 API 라우트)
  - 내부 라우트: `/app/api/membership/list/route.ts`에서 백엔드 `${API}${Api.endpoints.MEMBERSHIP.LIST}` 호출
  - 기대: `{ code: 0, data: { membershipList: [...] } }` (없으면 빈 배열)
- 제어:
  - 응답 순서(sortOrder ASC, id ASC)를 그대로 렌더(클라 정렬 금지)
  - FREE 포함, ADMIN 제외는 백엔드 보장

Step 1-B. 현재 멤버십 로드(SSR, 선택)
- 서버 액션: `MembershipRepository.getCurrentMembership()`
  - 서버 요청: `GET ${API}/v1/membership/current` (쿠키 포함)
  - 기대: `{ code: 0, data: 현재멤버십|null }` 또는 401
- 제어:
  - 401이면 리다이렉트하지 않고 게스트 모드로 렌더(보유 배지는 "로그인 후 확인")
  - 인증된 경우에만 현재 멤버십 배지 SSR 렌더

Step 2. 멤버십 목록 로드(SSR)
- 서버 액션: `MembershipRepository.getMembershipList()`
  - 기대: `{ code: 0, data: { membershipList } }`
- 제어:
  - `sortOrder` 정렬, `isPopular` 하이라이트
  - 카드로 이름/가격/설명/색상 렌더

Step 3. 멤버십 선택 → 상세로 이동(SSR)
- 링크: `/membership/[id]`
- 서버 액션: `MembershipRepository.getMembership(id)`
  - 기대: `{ code: 0, data: { canPurchase, compareResult, message, ... } }`
- 제어:
  - SSR CTA 결정: canPurchase=true면 “결제 진행” 활성, false면 메시지 + 비활성
  - 404(-3)면 안내 메시지 + 목록으로 돌아가기 링크 SSR 렌더

Step 4. 기간 선택 및 사전등록 시작(클라이언트 또는 서버 액션)
- CTA 클릭(클라): 기간 선택(MONTHLY/YEARLY, 상세 응답 기준)
- 사전등록 호출
  - 요청: `POST ${API}/v1/payments/pre-register` Body `{ membershipId, period }`
  - 기대: `{ code: 0, data: { paymentId, orderName, amount } }` 또는 409/400+message
- 제어:
  - 409/400 → 서버 메시지 토스트/모달 표시, 상세 페이지로 복귀
  - 정상 → PortOne/Toss에 `paymentId/orderName/amount` 전달하여 결제 진행

Step 5. PortOne/Toss 결제(클라이언트)
- `usePriceViewModel`과 레포지토리 헬퍼를 활용해 SDK 호출/승인 처리
- 성공/실패 핸들링
  - 성공 → 백엔드 완료 API 호출(구독 활성화) → 대시보드/성공 페이지로 이동
  - 실패 → 실패 페이지로 이동(쿼리 유지), 재시도 경로 제공

Step 6. 결제 후 확인
- 백엔드에서 멤버십 활성화(endDate/level 등 갱신)
- 프론트에서 `/v1/membership/current` 재조회(프로필/대시보드)로 최신 상태 반영

## 로컬 HTTPS/쿠키 전략

문제: 개발 환경에서 `http://`로 전환하면 `Secure` 쿠키가 저장되지 않습니다(브라우저 정책). 반대로 `https://` 자체서명 인증서는 SSR에서 `DEPTH_ZERO_SELF_SIGNED_CERT`로 실패할 수 있습니다.

해결 옵션(권장 순서)
1) 프론트 HTTPS + 백엔드 HTTP(리라이트 프록시)
- Next 개발 서버는 HTTPS로 접근(예: 3443 포트 프록시), 백엔드는 HTTP 유지.
- `next.config.js` 리라이트(`/v1/:path*`)로 같은 오리진(https://localhost:3443)에서 API를 호출하므로, 백엔드 Set-Cookie가 프론트 오리진에서 `Secure`로 저장 가능.
- 구현 예: `local-ssl-proxy` 또는 Caddy/Nginx로 `https://localhost:3443` → `http://localhost:3000` 프록시.

2) 백엔드/프론트 모두 HTTPS(신뢰 가능한 인증서)
- mkcert 등으로 신뢰되는 로컬 인증서 발급.
- Spring Boot에 인증서 적용, Next도 HTTPS로 서비스(리버스 프록시 권장).
- SSR fetch는 신뢰된 인증서로 성공, 쿠키도 `Secure` 저장.

3) 개발 전용 쿠키 완화(백엔드)
- dev 프로파일에서만 Set-Cookie의 `Secure` 제거 및 `SameSite=Lax` 설정.
- HTTP에서도 저장 가능. 운영에서는 반드시 `Secure + SameSite=None` 유지.

4) Next Route Handler 프록시(고급)
- `/app/api/*`에 프록시 라우트를 두고 서버에서 백엔드를 호출한 뒤, Next `Response.cookies`로 프론트 도메인에 쿠키를 재설정(개발 시 `Secure` 제거).
- 운영/개발 분기 필요. 보안 정책에 유의.

권장: 1) 또는 2). 운영 정책을 해치지 않으면서 개발 불편을 최소화합니다.

## 문제 해결 가이드 (Troubleshooting)

증상: `TypeError: fetch failed` (SSR에서 공개 목록 호출 시)

원인 후보 및 조치 순서
- [ ] 백엔드 기동 확인: 로컬 개발이라면 Spring Boot가 실행 중인지 확인하고 `GET {API}/v1/membership/list`가 200을 반환하는지 curl로 확인
  - 예: `curl -i http://localhost:8080/v1/membership/list`
- [ ] 환경변수 설정 확인: `NEXT_PUBLIC_API_URL`이 백엔드 Origin으로 설정되어 있는지 확인(프로토콜 포함, 슬래시 미포함)
  - 예(로컬): `NEXT_PUBLIC_API_URL=http://localhost:8080`
  - 예(운영): `NEXT_PUBLIC_API_URL=https://api.example.com`
  - 변경 후 Next 개발 서버를 재시작 필요
- [ ] 프록시(리라이트) 사용 고려: 백엔드가 다른 Origin이면 `next.config.js`에 리라이트를 구성하여 `/v1/:path*`를 백엔드로 프록시
  - 예:
    ```js
    async rewrites() {
      return [
        { source: '/v1/:path*', destination: 'http://localhost:8080/v1/:path*' },
      ];
    }
    ```
  - 이 경우 레포지토리 베이스 URL을 비워둔 채 상대경로(`/v1/...`) 호출 가능
- [ ] 인증서 이슈: 로컬 HTTPS의 자체 서명 인증서 사용 시 Node fetch가 실패할 수 있음 → 로컬은 HTTP 사용 권장
- [ ] 방화벽/네트워크: 개발 환경에서 대상 Origin 접근 불가 여부 확인

참고
- 404는 fetch 실패(TypeError)가 아니라 응답 객체로 들어오므로, `TypeError: fetch failed`는 네트워크 레벨 실패일 가능성이 큼(Origin/포트/프로토콜/인증서/네트워크 문제)
