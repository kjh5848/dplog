# 🧩 **Structured Logging Rule**

## ✅ Purpose

Avoid using raw `console.log`, `console.error`, and `console.warn`. Instead, use the utility functions provided in `logger.ts` to maintain a **consistent and structured logging system** across the codebase.

---

## 1. 🔁 Basic Replacement Rules

| Old Logging Statement | Replace With                      | Example                                         |
| --- | --- | ----- |
| `console.log(...)`    | `logInfo()` or `logApiResponse()` | `logInfo("Message", { context })`               |
| `console.error(...)`  | `logError()`                      | `logError("Error message", error, { context })` |
| `console.warn(...)`   | `logWarn()`                       | `logWarn("Warning", { key: value })`            |
| `console.debug(...)`  | `logDebug()`                      | `logDebug("Debug data", { ... })`               |

---

## 2. 📦 Required Imports

```ts
// ✅ Recommended: Use utility functions
import {
  logInfo,
  logError,
  logWarn,
  logDebug,
  logApiResponse,
  logUserAction,
  logPerformance,
} from "@/src/utils/logger";

// ✅ Alternative: Use logger instance (for advanced cases)
import { logger } from "@/src/utils/logger";

// ❌ Avoid: Mixing both imports
import { logInfo, logger } from "@/src/utils/logger";
```

## 3. 🛠️ Common Logging Patterns

### ✅ API Call Flow

```ts
logInfo("API call started", { operation: "getUserList" });
logApiResponse("User list response", response, { operation: "getUserList" });
logError("API call failed", error, { operation: "getUserList" });
```

### ✅ Error Handling

```ts
// ✅ Correct: Error 객체를 직접 전달
try {
  // some operation
} catch (error) {
  logError("Operation failed", error, { operation: "someOperation" });
}

// ✅ Correct: Error 객체를 Record로 변환하여 전달
try {
  // some operation
} catch (error) {
  const errorRecord = error instanceof Error 
    ? { message: error.message, stack: error.stack, name: error.name }
    : { error: String(error) };
  logError("Operation failed", undefined, { ...errorRecord, operation: "someOperation" });
}

// ✅ Correct: 안전한 타입 가드 사용
try {
  // some operation
} catch (error) {
  const errorObj = error instanceof Error ? error : new Error('Unknown error occurred');
  logError("Operation failed", errorObj, { operation: "someOperation" });
}

// ❌ Incorrect: Error 객체를 context에 직접 전달
logError("Operation failed", undefined, { error }); // Type error!

// ❌ Incorrect: 안전하지 않은 타입 캐스팅
logError("Operation failed", error as Error, { operation: "someOperation" });
```

### ✅ Logger Instance Usage (Advanced)

```ts
// ✅ Direct logger instance usage for advanced cases
import { logger } from "@/src/utils/logger";

logger.info("Custom message", { customContext: true });
logger.error("Custom error", new Error("Custom error"), { customContext: true });

// ✅ With custom options
logger.info("Message with tags", { data: "value" }, {
  tags: { feature: "auth", operation: "login" },
  user: { id: "user123" }
});
```

### ✅ User Actions

```ts
logUserAction("Button clicked", userId, { button: "save" });
logUserAction("Login", userId, { method: "kakao" });
```

---

### ✅ Performance Logging

```ts
const startTime = performance.now();
// do some work...
logPerformance("Ranking calculation", performance.now() - startTime, { shopId });
```

## 4. 🧠 ViewModel Usage Patterns

| Context     | Example                                                                |
| ----- | ---- |
| On init     | `logInfo("ViewModel initialized", { component: "useMainViewModel" });` |
| API request | `logInfo("Keyword add requested", { keyword, province });`             |
| API success | `logApiResponse("Keyword add response", response);`                    |
| API error   | `logError("Keyword add failed", error);`                               |
| User action | `logUserAction("Keyword typed", userId, { keyword });`                 |

---

## 5. 🔒 Security Guidelines

* ❌ Do not log sensitive data such as passwords, tokens, or credit card numbers
* ✅ It is acceptable to log emails, user IDs, keywords, etc.
* ✅ Sanitize API responses before logging (especially user-related fields)

---

## 6. ⚙️ Environment-Based Logging Behavior

| Environment | Logging Output                                                                                                  |
| ----- | --- |
| Development | All logs (console + Sentry)                                                                                     |
| Production  | Only `logInfo`, `logError`, `logWarn`, `logUserAction`, `logPerformance` are allowed.<br>`logDebug` is ignored. |
