# 2025.11.17 Global Server Actions & Caching Plan

## 1. 요청 인벤토리 (요약)
| 영역 | 클라이언트 fetch 위치 | 대응 Route Handler | 현재 방식 | 개선 방향 |
| --- | --- | --- | --- | --- |
| 멤버십 선택 (`/membership`) | `src/app/(home)/membership/page.tsx` (`handleSubscribe`) | `/api/membership/*` | 클라이언트 fetch | - 정적 카탈로그 → ISR<br>- 로그인 체크 → 서버 액션/리다이렉트 |
| 결제 (`/payment`) | `PaymentClientPage.tsx` (pre-register, billing-key, cancel 등) | `/api/payments/**`, `/api/subscriptions/**` | 클라이언트 fetch + Route Handler | - 멤버십 상세: 서버 액션(완료)<br>- 나머지 결제 요청도 서버 액션으로 이전 |
| 결제 결과 (`/payment/*`) | Success/Fail/Billing pages | 없음(직접 fetch) | 클라이언트 fetch | - 서버 액션 or Server Component에서 데이터 조회 후 결과 렌더 |
| 프로필 (`/profile`) | `ProfileView.tsx` (취소/다운그레이드) | `/api/subscriptions/*` | 클라이언트 fetch | - 서버 액션 호출로 전환 (결제와 동일 로직 공유) |
| NPLACE Track/Realtime/Keyword | 각 ViewModel에서 `fetch(/api/...)` | `/api/nplace/**` | 클라이언트 fetch | - 서버 액션 or Server Component로 데이터 요청 단계 분리 |
| Reply/Session 등 | 관련 컴포넌트 | `/api/...` | 일부 클라이언트 fetch | - 기능별로 동일 패턴 적용 |

> **전체 인벤토리**: `rg -n "fetch\(" src` 로 클라이언트 fetch 위치를 계속 추적.

## 2. 전략 요약
- **정적/캐시 가능한 데이터**: SSG/ISR로 페이지 초기 props 전달 → 클라이언트 fetch 제거
- **민감/사용자별 요청**: Route Handler 대신 서버 액션으로 호출 (쿠키 처리, 로그인 확인 포함)
- **Route Handler 공통화**: `lib/services/**`에 백엔드 호출 로직을 모으고, Route Handler와 서버 액션이 동일 함수 사용

## 3. 난이도별 계획
### 하
1. (완료) 멤버십 카탈로그/FAQ → ISR 적용

3. 다른 정적 페이지(FAQ, Terms 등)도 필요 시 SSG/ISR로 확대

### 중
1. 결제/프로필의 구독 취소·다운그레이드·취소취소 API 호출을 서버 액션으로 이전
2. NPLACE 주요 페이지(ViewModel)에서 `/api/...` 대신 서버 액션 호출 구조 마련
3. Route Handler 로직을 `lib/services/*`로 추출해 서버 액션과 공유

### 상
1. 결제 플로우(`handleSubscriptionPayment`) 전체를 서버 액션으로 위임 (pre-register, billing-key complete/fail 등)
2. 업그레이드/다운그레이드/해지 등 모든 결제 관련 경로를 서버 액션 기반으로 통일
3. NPLACE 데이터 fetch도 가능한 범위 내에서 Server Component + 서버 액션으로 변환하여 브라우저 노출 최소화
