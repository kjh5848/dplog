# Global Loading Component Integration Plan

## 1. 목표와 범위
- `src/utils/loading.ts` 의 표준 메시지/사이즈 맵을 활용해 전역 로딩 UX를 일관되게 만든다.
- 결제/멤버십/프로필 등 주요 화면에서 각각 다른 방식으로 표시되던 스피너·토스트를 하나의 오버레이 컴포넌트로 통합한다.
- 페이지마다 사용 중인 로딩 관련 상태를 점검해, 어디에서 전역 오버레이를 띄우고 어디에서는 기존 인라인 스켈레톤을 유지할지 명확히 정의한다.

---

## 2. 로딩 메시지 관리 전략 (`src/utils/loading.ts`)
1. **기존 유틸 재사용**  
   - `createLoadingConfig`, `loadingUtils` 등 이미 정의된 헬퍼를 그대로 사용해 message/size를 가져온다.
   - 페이지별 특수 문구가 필요한 경우 `customLoading` 으로 override 하되, 기본 타입은 `LoadingType` 중 하나로 매핑한다.
2. **공통 인터페이스**  
   - 전역 컴포넌트가 받을 프롭을 아래처럼 통일한다.  
     ```ts
     type GlobalLoadingConfig = ReturnType<typeof createLoadingConfig> & {
       subMessage?: string;
     };
     ```
   - message/size는 `loading.ts` 에서 가져오고, 서브 카피(예: “결제 정보를 확인하고 있습니다”)만 각 페이지에서 주입한다.
3. **도메인별 프리셋**  
   - 인증/페이지 이동은 `loadingUtils.userAuth`, `loadingUtils.redirectLoading`.
   - 결제계열(구독하기∙업그레이드∙다운그레이드)은 `customLoading('결제를 처리하고 있습니다', 'lg')` 를 기본값으로 두고, 세부 진행 단계는 `subMessage` 로 분리한다.

## 3. 전역 로딩 컴포넌트 설계
| 항목 | 내용 |
| --- | --- |
| 파일 위치 | `src/components/common/loading/GlobalLoadingOverlay.tsx` |
| Props | `{ visible: boolean; config: GlobalLoadingConfig; }` |
| 렌더 구조 | `Portal` + `fixed inset-0` dim + 카드형 컨테이너 (size 따라 스피너 크기 조절) |
| 접근성 | `aria-live="polite"`, dim 영역 클릭 시 무시(사용자 입력 차단) |
| 상태 관리 | 각 페이지/상위 뷰모델에서 boolean + config 전달. 전역 store(Zustand) 도입은 필요 시 추후 고려. |

### 컴포넌트 의사 코드
```tsx
const sizeClass = { sm: 'h-6 w-6', md: 'h-8 w-8', lg: 'h-10 w-10' }[config.size];

return visible ? (
  <Portal>
    <div className="fixed inset-0 z-[9999] bg-black/40 backdrop-blur-sm flex items-center justify-center">
      <section role="status" aria-live="polite" className="rounded-2xl bg-white/95 p-6 shadow-2xl text-center space-y-2">
        <div className={`mx-auto animate-spin rounded-full border-b-2 border-blue-600 ${sizeClass}`} />
        <p className="text-base font-semibold text-gray-900">{config.message}</p>
        {config.subMessage && (
          <p className="text-sm text-gray-600">{config.subMessage}</p>
        )}
      </section>
    </div>
  </Portal>
) : null;
```

### 4.1 프로필 페이지 (`src/components/profile/ProfileView.tsx`) — 난이도 하
| Existing State | 역할 | 전역 로딩 적용 방안 |

### 4.2 멤버십 선택 페이지 (`src/app/(home)/membership/page.tsx`) — 난이도 중
| Existing State | 역할 | 전역 로딩 적용 방안 |
| --- | --- | --- |
| `isProcessing` | 구독/업그레이드 공통 처리 중 | `visible = isProcessing` 으로 연결. |
| `downgradeProcessing` | 다운그레이드 예약/취소 | `visible = isProcessing \|\| downgradeProcessing`. |
| `portoneStatusMessage` | 단계별 진행 문구 | `subMessage` 로 전달, 기본 message 는 `customLoading('결제를 처리하고 있습니다', 'lg')`. |
| 초기 fetch 로딩 (`membershipLoading`, `subscriptionLoading`, `subscriptionCheckLoading`) | 페이지 진입 스켈레톤 | **유지** (페이지 전체 뷰가 아직 생성 안 되었으므로 전역 오버레이 대신 기존 스피너 유지). |

실행 순서:
1. `const [globalLoading, setGlobalLoading] = useState<{visible:boolean; config:GlobalLoadingConfig}>({ visible:false, config:customLoading('로딩 중...') });`
2. 구독/다운그레이드 관련 핸들러에서 `setGlobalLoading({ visible:true, config:{ ...customLoading(...), subMessage: portoneStatusMessage ?? ... }})`.
3. `finally` 블록마다 `setGlobalLoading(v => ({ ...v, visible:false }))`.
4. JSX 최상단에 `<GlobalLoadingOverlay visible={globalLoading.visible} config={globalLoading.config} />`.

| Existing State | 역할 | 전역 로딩 적용 방안 |
| --- | --- | --- |
| `useAuthGuard`의 `isLoading/isGuest/isLogoutPending` | 인증 체크 | 현재 `LoadingFallback`만 게스트 분기에 사용 → 전역 오버레이로 일원화. |
| `toast.loading` 패턴 (`handleGridItemClick`) | 데이터 재조회 안내 | 토스트는 유지하되, 같은 타이밍에 `GlobalLoadingOverlay`를 켜서 페이지 전체 상태를 표시. |
| ViewModel 상태 (`isLoadingShopList`, `isRealtimeDataLoading`, `isGridItemLoading`) | 특정 섹션 로딩 | 중첩 로딩으로 인한 UX 혼선을 줄이기 위해 최상위 bool(`isGlobalLoading = isLoadingShopList || ... || isGridItemLoading`)을 계산 후 오버레이 조건으로 사용. |

실행 순서:
1. `isLoading` (auth)일 때 즉시 `<GlobalLoadingOverlay visible config={loadingUtils.userAuth()} />` 반환.
2. 데이터 조회 함수에서 `setGlobalLoading({ visible:true, config: customLoading('순위 데이터를 불러오는 중입니다', 'md'), subMessage: ... })`.
3. 섹션별 스피너는 보조 표시로 유지하거나 상황에 따라 제거.

### 4.4 결제 페이지 (`src/app/(home)/payment/page.tsx`) — 난이도 상
| Existing State | 역할 | 전역 로딩 적용 방안 |
| --- | --- | --- |
| `isProcessing` | 구독/업그레이드 공통 처리 | 전역 오버레이의 핵심 trigger. |
| `downgradeProcessing` | 다운그레이드 예약/취소 | 동일하게 전역 상태에 포함. |
| `portoneStatusMessage` | 단계별 진행 문구 | `config.subMessage` 로 사용. |
| 초기 fetch 로딩 (`membershipLoading`, ...) | 초기 스켈레톤 | 기존 전체 화면 스피너 유지. |
| 버튼 내부 스피너 | 시각적 피드백 | 전역 오버레이 도입 후 제거 또는 보조 텍스트로 축소. |

실행 순서:
1. `const [globalLoadingConfig, setGlobalLoadingConfig] = useState<GlobalLoadingConfig>(customLoading('결제를 준비하고 있습니다', 'lg'));`
2. `const isGlobalLoading = isProcessing || downgradeProcessing;`
3. `<GlobalLoadingOverlay visible={isGlobalLoading} config={{ ...globalLoadingConfig, subMessage: portoneStatusMessage ?? ... }} />` 삽입.
4. 각 결제 핸들러에서 message 업데이트 (`setGlobalLoadingConfig(customLoading('빌링키 발급 중...', 'lg'))` 등).
