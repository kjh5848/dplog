# Nplace 키워드 조회 프런트엔드 구현 계획

## 0. 설계 원칙 (프로젝트 자산 재사용)
- **디자인/컴포넌트**: 기존 헤더(`DplogHeader`), 로딩(`GlobalLoadingOverlay`/`LoadingFallback`), 알림(`react-hot-toast`), 카드/테이블 스타일(Tailwind 토큰) 재사용.
- **상태 관리**: 기존 패턴대로 React Query + 로컬 상태 조합. 서버액션은 사용하지 않고 App Route 경유 REST만 사용(프로젝트 정책).
- **라우트 구조**: App Router 하위 `(dplog)/(nplace)/(rank)/keyword` 폴더 패턴 유지. 페이지 클라이언트 컴포넌트 + App Route로 API 프록시.
- **캐싱 전략**: React Query `staleTime/gcTime`, `retry` 최소화. 동일 키워드 조합은 키 기반 캐싱 후 히스토리 리스트로 재사용.

## 1. 범위·목표
- 기능: 일반 키워드 조회 + 연관 키워드 조회(RELATION/ TRACK) UI를 신규 페이지로 제공.
- 데이터: 댓글/텍스트 노출 없음, 키워드별 SearchAd 지표(`relKeyword`, `monthlyPcQcCnt`, `compIdx` 등)만 표시.
- 인증: 세션 기반(로그인 필수). 모든 호출은 Next App Route를 경유해 백엔드 `/v1/...` API와 `ResDTO` 계약을 유지.

## 2. API 매핑 (App Route → Backend)
| App Route | HTTP | Backend | 용도 |
| --- | --- | --- | --- |
| `/api/nplace/keyword` | POST | `/v1/nplace/keyword/nsearchad/keywordstool` | 다중 키워드 조회 |
| `/api/nplace/keyword/relation` | GET | `/v1/nplace/keyword/nsearchad/keywordstool/relation` | 연관/트랙 모드 조회 |
| `/api/nplace/keyword/nblog` (옵션) | GET | `/v1/nplace/keyword/nblog/search/info` | 블로그 검색량 보조 |

공통 처리:
- `proxyBackendJson` + `label` 지정, `code != 0` 시 `{ success: false, code, error }`로 전달.
- 요청 검증: 빈 키워드/최대 5개 제한, query param encode 확인.

## 3. 페이지 흐름 (권장 UX)
1) 로그인 가드(`useAuthGuard`) → 게스트는 `/login`.
2) 키워드 입력 영역: 개행/쉼표 입력 → 전처리 후 배열화.
3) 액션 버튼 2개: “키워드 조회”(POST), “연관 키워드”(GET, requestType=RELATION).  
   - TRACK 모드 버튼이 필요하면 토글/Select로 노출.
4) 결과 영역: `keywordToolList` 테이블/카드 렌더, 조회 히스토리/최근 키워드 태그 제공.
5) 에러/사용량 초과: 서버 `message`를 토스트로 그대로 노출, 업그레이드 CTA(`/membership?from=keyword`).

## 4. 상태·쿼리 설계
- React Query 키:  
  - 일반 조회: `["nplace","keyword","list", keywordString]`  
  - 연관 조회: `["nplace","keyword","relation", requestType, keywordList]`
- 요청 파라미터:
  - POST `/keyword`: `keywordString` = 키워드들을 `\n`로 연결한 문자열.
  - GET `/keyword/relation`: `keywordList[]=...` (1~5개), `requestType=RELATION|TRACK`.
- 응답 매핑 타입:
  ```ts
  interface KeywordToolItem {
    relKeyword: string;
    monthlyPcQcCnt?: number;
    monthlyMobileQcCnt?: number;
    monthlyAvePcClkCnt?: number;
    monthlyAveMobileClkCnt?: number;
    monthlyAvePcCtr?: number;
    monthlyAveMobileCtr?: number;
    plAvgDepth?: number;
    compIdx?: string; // "높음"/"중간"/"낮음"
  }
  ```

## 5.1 컴포넌트/레이아웃 재사용
- 상단: `DplogHeader`(`title="N-PLACE"`, `message="키워드 조회"`).
- 로딩: `GlobalLoadingOverlay`(초기 인증/멤버십 확인) → 이후 API 호출 로딩은 섹션 스피너 또는 `LoadingFallback`.
- 컨테이너: 기존 `(rank)` 페이지와 동일한 `mx-auto max-w-6xl px-4` 구조, 카드/테이블 Tailwind 조합 재사용.
- CTA: `/membership` 링크 버튼(사용량 초과 시 `notifyUsageLimit` 흐름과 동일 토스트 스타일).
- 빈 상태/에러: 기존 `ReplyContent`/검색 페이지의 경고 박스 스타일 재사용(`bg-red-50`/`border-red-100`).

## 5. UI 스펙 (제안)
- 입력: 다중 키워드 textarea + “중복/빈값 제거” 안내, 최대 5개 제한 메시지.
- 모드 선택: Tabs 또는 Segmented control(기본: 키워드 조회, 보조: 연관/Track).
- 결과:  
  - 테이블: 키워드, PC/Mobile 검색량, 클릭수, CTR, 경쟁도(compIdx), 랭크/깊이(plAvgDepth).  
  - 카드 레이아웃(모바일) 대응.
- 빈 상태: “키워드를 입력해 조회하세요” 안내.
- 에러/한도: 토스트 + CTA 버튼(업그레이드).

## 6. 에러·사용량 처리
- `code != 0` → `message` 바로 노출.  
- 사용량 초과(`ServiceSort.NPLACE_KEYWORD`, `NPLACE_KEYWORD_RELATION`) 메시지 그대로 사용.  
- 401/AuthenticationException → 로그인 리다이렉트 안내.
- 빈 키워드/최대 개수 초과는 클라이언트 선검증.
- React Query 옵션:  
  - 일반 조회: `staleTime: 3 * 60 * 1000`, `gcTime: 30 * 60 * 1000`, `retry: 1`, `refetchOnWindowFocus: false`.  
  - 연관 조회: `staleTime: 2 * 60 * 1000`(키워드 변동이 잦으므로 짧게).  
  - `suspense` 미사용, 로컬 로딩 상태로 구분 표시.
- 캐시 키 설계로 동일 입력 재조회 시 캐시 활용, “최근 조회” 히스토리에서 선택 시 캐시된 데이터 우선 렌더 후 백그라운드 갱신.

## 6-1. 조회 결과 UX 고도화 요구사항 (신규)
- 결과 요약 표시: 조회 후 상단에 “결과 n건” 카운트 배지 노출.
- 필터링 여부: 필터 적용 시 배지/텍스트로 표시(예: “필터 적용됨”)하고 해제 버튼 제공.
- 페이징: 페이지 이동/페이지당 개수 설정을 지원(간단 페이저 + select). 상태는 쿼리스트링/로컬 상태로 유지.
- 결과 분리: 키워드 조회 결과와 연관 키워드 결과는 별도 상태·캐시 키로 관리해 서로 영향을 주지 않음.
- 히스토리 캐싱: 입력 키워드와 결과를 로컬스토리지/React Query 캐시에 저장해 “최근 조회” 목록으로 재호출 없이 즉시 복원, 필요 시 백그라운드 리프레시.

## 7. 테스트 체크리스트
- 세션 없음 → `/login` 리다이렉트.
- 빈 입력/6개 이상 입력 시 프런트 차단.
- 정상 조회: 다중 키워드 POST 결과 테이블 렌더 확인.
- 연관 조회: requestType=RELATION, TRACK 각각 호출/표시 확인.
- 사용량 초과/에러 메시지 토스트 노출, CTA 동작 확인.
- 모바일 뷰에서 입력/결과 레이아웃 깨짐 없는지 확인.

### 7-1. 상세 체크리스트 (신규)
- 입력/검증
  - 키워드 입력 시 개행·쉼표로 구분, 중복/빈값 제거 확인.
  - 연관 모드에서 5개 초과 입력 시 즉시 에러 메시지, 서버 호출 안 함.
- 호출/응답
  - 키워드 조회: `/api/nplace/keyword` POST payload/응답 정상 파싱.
  - 연관 조회: `/api/nplace/keyword/relation` GET 파라미터(`keywordList`, `requestType=RELATION`) 정상 전달.
  - 실패 응답(code != 0)에서 토스트 메시지 그대로 표출, lastError 표기.
- UI/상태
  - 결과 카운트 표시(“결과 n건”) 확인.
  - 필터 적용 상태 배지/해제 동작 확인(필요 시 자리 확보).
  - 페이징: 페이지 이동/페이지당 개수 변경 시 데이터와 카운트 반영.
  - 키워드 조회 결과와 연관 조회 결과가 서로 덮어쓰지 않고 별도로 유지.
  - 히스토리: 최근 조회 리스트에서 선택 시 캐시된 결과 즉시 렌더 후 백그라운드 리프레시.
- 반응형/스크롤
  - 모바일에서 결과 카드 내부 가로 스크롤 가능, 세로는 내용에 맞게 확장.
  - 데스크톱에서 테이블 폭/정렬 정상, 긴 리스트도 스크롤로 확인 가능.
- 사용량/인증
  - 사용량 초과 메시지 토스트/CTA(`/membership`) 동작.
  - 세션 만료 시 `/login` 리다이렉트 및 메시지 확인.

## 8. 고도화 요구사항 및 작업 계획
- 요구사항 요약
  - 결과 요약/필터/페이징 추가: 결과 건수, 필터 적용 여부 표시, 페이지 이동·페이지당 개수 설정.
  - 결과 분리: 키워드 조회 결과와 연관 키워드 결과의 상태/캐시를 분리 관리.
- 히스토리 캐싱: 입력·결과를 로컬스토리지/React Query 캐시에 저장해 “최근 조회”에서 즉시 복원.
- 반응형 스크롤: 결과 카드 내부에서 가로 스크롤 가능, 세로는 내용에 맞게 확장되도록 유지.
- 작업 계획
  1) 상태 분리: `keywordResults`, `relationResults`를 별도 state/React Query 캐시 키로 관리. 결과 카운트는 각 결과의 length로 표시.
  2) 페이징/표시 개수: 로컬 상태로 `page`, `pageSize` 추가 후 테이블 렌더 시 slice; UI에 간단 페이저 + 페이지당 개수 select 추가.
  3) 필터 배지: 필터 적용 시 배지/해제 버튼 노출, 필터링된 데이터 기반으로 카운트/페이징 반영.
  4) 히스토리: 최근 조회 리스트를 로컬스토리지에 저장하고, 클릭 시 캐시된 결과를 먼저 렌더 후 백그라운드 리프레시.
  5) 스크롤 레이아웃 점검: 결과 카드 `overflow-x-auto` 유지, 필요 시 `min-w` 조정; 세로는 내용에 따라 확장하도록 고정 높이 제거.

## 8. 작업 순서 제안
1) App Route 2종 작성(`/keyword`, `/keyword/relation`) + 타입 정의.  
2) 키워드 페이지 컴포넌트 생성(입력/모드/결과/히스토리).  
3) React Query 훅 및 응답 매핑, 클라이언트 검증 로직 추가.  
4) 에러/한도 토스트 + 업그레이드 CTA 연결.  
5) E2E/수동 QA (로그인/무제한/사용량 초과/빈값).  

## 9. 추가 설계: 필터 정렬·조건 검색·히스토리 TTL
- 정렬 기능
  - 상태: `sortKey`(`relKeyword`, `monthlyPcQcCnt`, `monthlyMobileQcCnt`, `compIdx`, `plAvgDepth` 등) + `sortDir`(`asc`/`desc`).
  - 처리: `activeResults`를 정렬 후 필터 → 페이징 순으로 적용. 헤더 클릭 토글 또는 드롭다운 UI 제공.
  - 경쟁도(`compIdx`) 정렬은 우선순위 맵핑(`높음 > 중간 > 낮음`)을 숫자화해 정렬.
- 조건부 필터
  - 상태: 수치 범위(예: `pcMin/pcMax`, `mobileMin/mobileMax`), 경쟁도 선택(`낮음/중간/높음`), 키워드 포함/제외 텍스트.
  - 처리: `filteredResults` 계산 시 조건식을 추가해 걸러낸 뒤 페이징. 입력 값은 로컬 상태로 유지, reset 버튼 제공.
- 히스토리 TTL·멤버십 정책
  - 스토리지: `localStorage`에 최대 10개 저장은 유지하되, `timestamp` 기반 TTL 추가.
  - TTL 정책: 무료 플랜 1일 유지, 유료/마스터 플랜은 TTL 없이 유지하되 최대 50개까지만 저장(초과 시 최신 50개만 보관). 멤버십 정보는 `membership/current` 결과로 판정.
  - 로딩 시: 무료는 24시간 초과 항목 제거 후 최신순 정렬; 유료는 개수 초과분만 제거. 동일 키 조합은 최신으로 교체.
  - UI: 히스토리 리스트에 “최근 조회” 타임스탬프 표시, 만료/삭제 시 안내(선택 사항).
