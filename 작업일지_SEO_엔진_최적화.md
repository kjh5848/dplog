# 🚀 D-PLOG 황금 키워드 발굴 엔진 최적화 및 딥스크랩 롤백 일지 (2026-04-01)

본 문서는 **"사용자(대표님)의 타스크 및 워크쓰루 기록을 프로젝트 루트에서 직접 파악하고 싶다"**는 지시에 따라 생성된 작업 일지입니다.

## 🎯 1. 이슈 개요 및 핵심 원인 파악
- **에러 증상**: 대시보드에서 `[발굴 시작]` 클릭 시 4개 상점에 대한 동시 접속 크롤링 봇이 기동되어, 네이버 IP 밴(429 Too Many Requests) 및 WAF 캡챠 차단이 발생했습니다. 이로 인해 100초가 넘어가는 대기열 병목 현상이 생겨 `504 Gateway Timeout`으로 프론트엔드가 강제 셧다운 되었습니다.
- **근본 원인 식별**:
  1. `순수 무작위 Jitter(0.1 ~ 10.0)` 적용 시, 우연히 동일한 초수(예: `1.23초`)에 브라우저가 몰려서 **군집 접속(Clustering) 충돌** 발생.
  2. `check_place_ranking`의 300위 무한 스크롤(딥 서치) 로직이 동기식(Sync)으로 동작하여 병목을 극대화.

## 🛠️ 2. 기술적 해결 (Staggered Jitter 재설계)
브라우저 충돌을 물리적으로 원천 차단하기 위해, "지연 딜레이 방식 구조 개편"을 진행했습니다.
```python
# full_list_extractor_8_threads.py 내 적용 코드
base_delay = ((search_num - 1) % 8) * 0.5   # 브라우저마다 0.5초의 간격 번호표 보장
jitter = random.uniform(0.1, 0.6)           # 난수 추가 합산
delay = base_delay + jitter
```
- 결과적으로 **안전한 우회 패턴(티켓 발부식 분산 구조)**을 확립하여 IP 차단을 방어했습니다.

*   `"1번에서 내 순위를 확인하는 건 넣자! 300위 까지 스크롤해서 내 상점이 있다는건 결국 작업할 가치가 잇는 키워드 이니깐"` (대표님 지시)

## 🚨 [CRITICAL POSTMORTEM] 300위 딥서치 단일 IP 트래픽 물리적 한계 (2026-04-01 추가)

> **"두 번 다시 무의미한 동시접속 튜닝 실험을 반복하지 않기 위한 최종 결론"**

1. **상황**: `concurrency=2`로 트래픽 속도를 늦춰 '동시 접속 폭주(Burst Limit)'는 우회했으나, 4개 상점을 연달아 돌렸을 때 3번째 상점에서 네이버 방화벽 `429 에러(IP Rate Limit 차단)`가 또다시 발생함.
2. **원인 (Volume Limit)**: 300위를 검색하려면 화면을 80번 무한 스크롤 내려야 함. 15개 키워드를 3개 상점 연속으로 돌리면 **약 7분 동안 단일 IP에서 3,600번의 스크롤 통신**이 발생. 네이버가 이를 비정상적인 데이터 추출 봇으로 확정 지음.
3. **오판(Mistake) 기록**: 트래픽 신뢰도를 높이겠다고 '세션 쿠키(NID_SES)'를 유지한 채 검색하려고 했으나, 이는 IP 밴이 아닌 **해당 쿠키(계정) 자체가 봇으로 찍혀 영구적인 캡차(Captcha) 루프에 빠지게 만드는 최악의 독약**임. (과거에 시크릿 모드로 각 브라우저를 완벽히 격리시켰던 설계가 100% 정답이었음)
4. **최종 결론**: 1개의 모바일 핫스팟(단일 무로 IP)으로 브라우저를 띄워 300위를 전부 뒤지는 방식은 **상점 2개 분량이 물리적 마지노선**임. 브라우저 속도 제어(Jitter/Concurrency)로는 절대 이 한계를 넘을 수 없음을 확정함.
