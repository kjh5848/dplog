이해하기 어렵고 혼란스럽거나, 엉성하게 쓰인 인터페이스 텍스트를 식별하여 사용자가 훨씬 쉽게 제품을 이해하고 조작할 수 있도록 개선합니다.

## 필수 준비 사항

`frontend-design` 스킬을 활용하세요 — 여기에는 디자인 원칙, 안티 패턴 및 **컨텍스트 수집 프로토콜(Context Gathering Protocol)**이 포함되어 있습니다. 작업을 시작하기 전에 반드시 지침을 따르십시오 — 아직 디자인 컨텍스트 정보가 없다면 무엇보다 먼저 `teach-impeccable`을 실행해야 합니다. 추가 파악 요소: 대상 사용자의 기술적 이해도 및 직면한 상황에서의 심리 상태를 면밀하게 가늠하세요.

## 현재 작성된 카피 평가

어떤 이유로 텍스트가 모호하거나 제 역할을 못하고 있는지 분석해 보세요:

1. **명확성을 해치는 문제 탐색**:
   - **그들만의 은어 (Jargon)**: 일반 사용자는 절대 알아먹을 리 없는 기술적 전문 용어
   - **모호함 (Ambiguity)**: 여러 가지 뜻으로 중의적 해석이 가능한 애매한 문구
   - **수동태 (Passive voice)**: "귀하의 파일이 업로드 처리 되었습니다" (x) → "파일 업로드를 완료했어요" (o)
   - **분량의 실패**: 구구절절 쓸데없이 말이 많거나, 반대로 불친절하게 짧은 경우
   - **지식의 오판 (Assumptions)**: 사용자가 당연히 알 거라고 지레짐작하고 생략해 버린 설명
   - **맥락의 상실**: 그래서 당신들이 왜 이 정보를 나한테 요구하는지 도무지 납득할 수 없음
   - **톤앤매너 불일치**: 너무 딱딱한 관공서 투이거나, 반대로 심각한 시스템 장애 상황에서조차 장난스러운 하프톤 억양

2. **컨텍스트 이해**:
   - 주 이용 고객은 누구인가? (개발자? 할머니? 완전 처음 서비스를 접한 뉴비?)
   - 유저의 지금 심리 상태는? (에러 터져서 멘붕 오고 빡친 상태? 결제 성공해서 설레는 상태?)
   - 유도하려는 액션이 무엇인가? (그래서 결국 유저가 어떤 버튼을 눌러야 하는가?)
   - UI의 물리적 제약은? (글자 수 압박? 모바일 화면 협소?)

**중요(CRITICAL)**: 깔끔하고 명쾌한 카피는 사용자를 곧장 성공으로 이끕니다. 반면 꼬인 카피는 온갖 좌절감과 잦은 오작동을 야기하며 애꿎은 고객센터 전화기만 불나게 만듭니다.

## 카피 개선 전략 수립

명쾌한 커뮤니케이션을 위한 커스텀 전략을 구성하세요:

- **핵심 메시지**: 유저가 무조건 알아야 하는 단 '하나'의 진짜 메시지는 무엇인가?
- **다음 행동 유도**: 그렇다면 이제 무엇을 해야 할 차례인가? (가만히 있어야 하나, 버튼을 눌러야 하나?)
- **톤 앤 보이스**: 어떤 톤으로 다가갈 것인가? (친절하게 오구오구? 정중하게 사과? 힘찬 응원?)
- **물리적 제약**: 최대 허용 글자 수, 브랜드 가이드라인 보이스, 다국어 번역을 대비한 로컬라이징 구조 타당성.

**중요 (IMPORTANT)**: 위대한 UX 라이팅(UX Writing)은 그 자체로 완전히 투명하여 보이지 않습니다. 문장에 시선을 뺏길 겨를도 없이 행동 흐름 자체가 즉각적으로 관통해야 합니다.

## 상황별 체계적인 카피 개선 방안

가장 흔하게 벌어지는 각 컴포넌트 구간에서 텍스트를 담금질하세요:

### 에러 메시지 (Error Messages)
**최악**: "Error 403: Forbidden"
**합격**: "현재 이 페이지에 접근할 권한이 없습니다. 접근 권한이 필요하시다면 관리자에게 요청해주세요."

**최악**: "Invalid input (입력값 오류)"
**합격**: "이메일 주소에는 @ 기호가 포함되어야 합니다. (예: name@example.com)"

**원칙**:
- 원인 규명을 일반인도 알 수 있는 쉬운 말로 풀어서 설명할 것
- 그래서 어떻게 하면 복구할 수 있는지 해결 방안을 함께 쥐여줄 것
- 시스템 오류를 은근슬쩍 유저의 탓으로 돌리지 말 것
- 가능하다면 예시를 시각적으로 보여줄 것
- 필요하다면 즉각적인 고객센터 문의 링크를 동봉할 것

### 입력 폼 레이블 & 지침 (Form Labels & Instructions)
**최악**: "DOB (MM/DD/YYYY)"
**합격**: "생년월일" (플레이스홀더로 YYYY.MM.DD 형식 표기)

**최악**: "여기에 값 입력"
**합격**: "이용 중인 이메일 주소" 혹은 "이력서에 등록될 회사명"

**원칙**:
- 애매한 플레이스홀더로 퉁치지 말고 확실하고 명시적인 라벨을 박아넣을 것
- 어떤 형식을 요구하는지 예제 뷰를 친절하게 보여줄 것
- (직관적이지 않다면) 왜 이 민감한 정보를 구태여 수집하려 하는지 그 이유를 밝힐 것
- 설명글(Instruction)은 인풋 박스 '아래'가 아니라 '위'에 배치하여 먼저 읽고 적게 유도할 것
- 필수 표기(Required) 마크를 누락시키지 말고 선명하게 노출할 것

### 버튼 & CTA 텍스트
**최악**: "여기를 클릭" | "전송" | "확인 (OK)"
**합격**: "계정 만들기" | "변경 사항 저장" | "네, 이해했습니다"

**원칙**:
- 무슨 일이 벌어질지 행동 결과를 동사로 명확히 명시할 것
- "전송됨" 같은 수동태 말고 "작성 완료하기" 식의 주도적 능동태를 쓸 것
- 사용자의 머릿속 멘탈 모델과 완벽하게 부합시킬 것
- 뭉뚱그린 "확인" 버튼 대신 "저장하기" 처럼 확실한 결과를 텍스트에 못 박을 것

### 도움말 & 툴팁 (Help Text & Tooltips)
**최악**: "사용자 아이디를 입력하는 필드입니다." (이미 라벨에 다 있는 얘기 반복)
**합격**: "서비스에서 사용할 닉네임을 지정해 주세요. 나중에 설정에서 언제든 변경할 수 있습니다."

**원칙**:
- 아무 영양가 없는 텍스트 앵무새 반복 절대로 금지 (추가적인 가치를 부여해야 함)
- 유저의 무의식적 질문에 직접적으로 대답할 것 ("이게 대체 뭔데?" 또는 "이걸 왜 쓰라는 건데?")
- 타격감 있게 꽂히면서도 내용은 부족하지 않게
- 내용이 길어질 것 같으면 깔끔하게 상세 가이드라인 외부 다큐먼트 링크로 넘길 것

### 빈 상태 (Empty States)
**최악**: "아이템 없음 (No items)"
**합격**: "아직 등록된 프로젝트가 없습니다. 첫 프로젝트를 생성하고 작업을 시작해 볼까요?"

**원칙**:
- 왜 여기가 텅 비어 있는지 상황을 납득시킬 것 (당연한 게 아니라면)
- 여기서 유저가 당장 뭘 눌러야 하는지 거대한 CTA로 멱살을 잡고 이끌어 줄 것
- 여기가 감옥의 막다른 골목(Dead-end)이 아니라 새로운 출발점인 것처럼 환대(Welcoming)할 것

### 성공 알림 메시지 (Success Messages)
**최악**: "성공"
**합격**: "설정이 저장되었습니다! 변경 사항은 지금 바로 모든 화면에 적용됩니다."

**원칙**:
- 방금 네가 누른 행동으로 무슨 일이 처리되었는지 확신을 줄 것
- 이다음엔 어떤 변화가 따라오는지 미래를 알려줄 것 (해당되는 경우)
- 경박하지 않으면서도 핵심은 다 담을 것
- 유저가 느끼는 감정선에 적절히 동기화해 줄 것 (어려운 과업을 성공했다면 아낌없이 리액션 해주기)

### 로딩 상태 (Loading States)
**최악**: "로딩 중..." (이러고 30초 넘게 뻗어있음)
**합격**: "입력하신 데이터를 실시간 분석 중입니다... 보통 30초에서 1분 정도 소요될 수 있어요."

**원칙**:
- 체감 대기 시간을 명확히 미리 세팅해 줄 것 (얼마나 걸릴까?)
- 이 긴 시간 동안 뒤에서 시스템이 대체 무슨 짓을 하고 있는지 상세하게 썰을 풀어줄 것
- 가능하다면 단순 뺑뺑이(Spinner) 말고 진행 퍼센티지 바(Progress bar)를 제공할 것
- 너무 오래 걸린다 싶으면 언제든 포기하고 빠져나갈 비상 탈출구("취소" 버튼)를 열어둘 것

### 확인 다이얼로그 (Confirmation Dialogs)
**최악**: "정말 확실합니까?"
**합격**: "'알파 프로젝트'를 정말 삭제하시겠습니까? 이 작업은 영구적으로 복구할 수 없습니다."

**원칙**:
- 어떤 대상을 날려버릴 건지 액션을 날카롭게 적시할 것
- 파괴적인 액션이라면 그 비극적 결말을 무서울 정도로 솔직하게 경고할 것
- "네(Yes) / 아니오(No)" 따위의 멍청한 버튼 쓰지 말고 "프로젝트 삭제 / 취소" 등 명확한 액션 레이블을 박을 것
- (사소한 변경 사항마다 저장할 건지 묻는 둥) 짜증 나는 확인 창을 습관적으로 남용하지 말 것

### 내비게이션 & 경로 안내 (Navigation & Wayfinding)
**최악**: "항목" | "목록" | "기타 자료" 같이 게으른 분류 레이블
**합격**: "내 프로젝트 관리" | "초대된 팀원 목록" | "보안 설정" 같이 구체적인 레이블

**원칙**:
- 추상적인 단어 배제하고 명확한 기능 위주의 구체적 어휘 사용
- 내부 직원들끼리나 통할 법한 부서 용어(Jargon) 지양하고 타겟 유저의 언어로 세탁하기
- 공간의 계층 구조를 직관적으로 세팅할 것
- 유저가 현재 어느 골목을 걷고 있는지 정확한 냄새(Information scent)를 피워줄 것 - (브레드크럼, Current location 활성 표기 등)

## 압도적 카피 작성의 대원칙

단 한 줄의 문장이라도 아래의 검열을 반드시 통과시켜야 합니다:

1. **지독할 정도로 구체적인가**: "값 입력" (x) → "가입하신 이메일 주소 입력" (o)
2. **칼춤 추듯 간결한가**: 의미 훼손이 없는 선에서 모든 쓸데없는 수식어와 군더더기 박멸
3. **능동태인가**: "저장될 예정입니다" (x) → "변경 사항 저장" (o)
4. **인간미가 느껴지는가**: "시스템 치명적 오류코드 발생" (x) → "앗, 일시적인 문제가 생겨 연결이 끊어졌어요" (o)
5. **실제 도움이 되는가**: 이미 일어난 참사를 중계만 하지 말고 "그래서 어떻게 해야 해결할 수 있는지" 행동 지침을 함께 쥐여주기
6. **집요하게 일관적인가**: 같은 뜻이면 하나의 단어로 통일 (어휘력 뽐내려고 똑같은 용어를 다르게 변형 금지)

**절대로 하면 안 되는 짓 (NEVER)**:
- 단어 뜻풀이도 없이 불친절하게 개발자 은어 들이밀기
- 시스템 오류를 은근슬쩍 유저 과실로 몰아가며 기분 상하게 하기 ("너 잘못했어" → "어라, 필수 입력 항목이 빠져있네요")
- "오류 발생" 한 마디 퉁 치고 뒤로 숨어버리기 (대체 뭘 어떻게 오류가 났는지 구체적으로 밝힐 것)
- 책임 회피성 수동태 문장 구사하기
- 불필요하게 혓바닥이 긴 장황한 설명 늘어놓기 (숨 막히니까 다 잘라낼 것)
- 서버 터지는 심각한 오류가 났는데 눈치 없이 농담 따먹기 시전 (공감하는 태도로 사과부터 박을 것)
- 유저가 당연히 IT 천재일 것이라고 거만하게 단정 짓기
- 화면마다 부르는 명칭을 다르게 표기하여 헷갈리게 만들기 (제발 단어 하나로 통일할 것)
- 방금 위에서 한 뻔한 소리를 아래에서 또 반복하여 타이핑 낭비하기
- 인풋 라벨 없이 플레이스홀더만 달랑 넣어두기 (사용자가 타이핑하는 순간 안내문이 증발해버려 기억 상실에 걸림)

## 개선 결과 품질 검증

수정된 텍스트 카피가 진짜로 구출 작전에 성공했는지 검수하십시오:

- **이해도 (Comprehension)**: 부연 설명 하나도 없이 이 문장 하나만 던져놔도 알아서 이해하고 행동하는가?
- **액션 유도력 (Actionability)**: 유저가 텍스트를 읽고 나서 이다음에 당장 무슨 버튼을 눌러야 할지 1초 만에 캐치하는가?
- **숨 막히는 간결함 (Brevity)**: 의미 손살 없이 자를 수 있는 모든 단어를 도려냈음에도 문장이 여전히 완벽한 폼을 유지하는가?
- **일관성 (Consistency)**: 여기서 쓰는 이 버튼 이름과 저쪽 세팅 창 메뉴 이름의 톤오브보이스(Tone of Voice)가 동일한 배를 타고 있는가?
- **눈치 있는 태도 (Tone)**: 지금 텍스트의 온도와 말투가 유저가 맞이한 그 빡침의, 혹은 기쁨의 타이밍과 적절히 어우러지는가?
