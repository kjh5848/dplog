기능을 분석하고 이해를 높이며, 피드백을 제공하고, 즐거움을 창출하는 애니메이션과 마이크로 인터랙션을 전략적으로 추가하세요.

## 필수 준비 사항

`frontend-design` 스킬을 사용하세요. 이 스킬에는 디자인 원칙, 안티 패턴 및 **컨텍스트 수집 프로토콜(Context Gathering Protocol)**이 포함되어 있습니다. 작업을 시작하기 전에 프로토콜을 따르세요. 만약 디자인 컨텍스트가 등록되어 있지 않다면 반드시 먼저 `teach-impeccable`을 실행하세요. 추가로 해당 앱의 퍼포먼스 제약 사항도 파악해야 합니다.

## 애니메이션 도입 기회 평가

모션 연출을 통해 체감 품질을 높일 수 있는 영역을 분석하세요:

1. **정적인 영역 식별**:
   - **피드백 부재**: 시각적인 호응 상태가 없는 액션 (버튼 클릭, 폼 제출 등)
   - **어색한 트랜지션**: 너무 갑작스럽고 몰입을 깨는 상태 변화 (노출/숨김, 페이지 로드, 라우트 변경)
   - **불명확한 관계**: 즉각적으로 인지하기 힘든 공간적 계층 구조 및 관계성
   - **무미건조함**: 기능적 작동은 문제없으나 감각적인 즐거움이 결여된 인터랙션
   - **의도 누락**: 사용자의 시선을 유도하거나 앱의 동작 방식을 납득시켜야 하는 곳

2. **컨텍스트 이해**:
   - 앱의 개성 및 성격은 어떤가요? (재치 있는 스타일 vs 진중한 스타일, 역동적 vs 차분함)
   - 퍼포먼스 예산(Performance budget)은 어떤가요? (모바일 우선인가? 데이터가 무거운 페이지인가?)
   - 주요 사용자는 누구인가요? (모션 어지럼증을 겪는 사용자? 즉각적인 속도를 원하는 파워 유저?)
   - 가장 강조해야 할 포인트는 무엇인가요? (하나의 눈에 띄는 진입 애니메이션 vs 다수의 마이크로 인터랙션)

코드베이스를 살펴봐도 이런 점들이 파악되지 않는다면, 즉시 사용자에게 직접 물어보세요.

**중요(CRITICAL)**: `prefers-reduced-motion` 설정을 항상 존중해야 합니다. 디바이스에서 애니메이션 감소를 설정한 사용자를 위한 대체안(Non-animated)을 반드시 제공하세요.

## 애니메이션 전략 수립

목적이 뚜렷한 애니메이션 연출 계획을 세우세요:

- **히어로 모멘트(Hero moment)**: 단 하나의 가장 핵심적인 애니메이션 연출은 무엇인가요? (페이지 로드, 히어로 영역, 핵심 인터랙션 지점)
- **피드백 레이어**: 어떤 인터랙션에 즉각적인 상호작용 피드백을 주어야 하는가?
- **트랜지션 레이어**: 어떤 상태 변화 과정(State changes)을 매끄럽게 연결해야 하는가?
- **즐거움 레이어**: 예상치 못한 작은 기쁨(Delight)을 줄 수 있는 곳은 어디인가?

**중요**: 잘 짜여진 한두 개의 집중된 모션 연출이, 의미 없이 곳곳에 흩뿌려진 산만한 애니메이션보다 훨씬 낫습니다. 효과가 가장 큰 타이밍에 집중하세요.

## 애니메이션 구현

다음 카테고리에 맞춰 체계적으로 모션을 더하세요:

### 진입 애니메이션 (Entrance Animations)
- **페이지 로드 코레오그래피**: 요소들이 등장하는 시간을 살짝 엇갈리게(Stagger) 설정 (100-150ms 지연 시간 배치), fade와 slide 조합
- **히어로 영역**: 핵심 콘텐츠의 드라마틱한 등장을 연출 (스케일 업, 패럴랙스 혹은 크리에이티브 텍스트 효과)
- **콘텐츠 노출 연출**: Intersection Observer를 활용한 스크롤 연계 등장 효과
- **모달/드로어 진입**: 부드러운 Slide + Fade, 배경 딤(Dim) 처리, 자동으로 포커스 이동시키기

### 마이크로 인터랙션 (Micro-interactions)
- **버튼 피드백**:
  - Hover 시: 미세한 크기 확대 (1.02-1.05), 색상 변화, 그림자 범위 증가
  - Click 시: 순식간에 줄어들었다 다시 회복되는 텐션 (0.95 → 1), 리플(Ripple) 효과
  - Loading 시: 텍스트가 스피너 혹은 펄스(Pulse) 애니메이션으로 자연스럽게 교체
- **폼(Form) 인터랙션**:
  - 인풋 포커스: 테두리 색상이 자연스럽게 전환되며 살짝 커지거나 은은한 Glow 발산
  - 유효성 검증 상태: 에러 시 좌우 흔들림(Shake), 성공 시 체크 마크 연출, 자연스러운 색상 전환
- **토글 스위치**: 버튼 슬라이싱과 색상 전환이 겹치는 부드러운 움직임 (200-300ms)
- **체크박스/라디오**: 드로잉 되듯 나타나는 체크 마크, 클릭 지점에서 퍼지는 잔물결 리플 효과
- **좋아요/즐겨찾기**: 튀어 오르는 스케일 업 + 각도 회전, 파티클 그래픽, 색상 변화

### 상태 전환 (State Transitions)
- **보기/숨기기**: 즉각적으로 숨기지 않고 Fade + Slide 처리, 목적에 맞는 시간 할당 (200-300ms)
- **확장/축소 (아코디언)**: 최대 높이(Height)를 예측하여 콘텐츠가 넘치지 않게 대응하며 전환, 아이콘 회전 동반
- **로딩 상태**: 번쩍거리지 않고 부드럽게 스며드는 스켈레톤(Skeleton UI), 로딩 스피너, 프로그레스 바
- **성공/실패 알림**: 배경이나 테두리 색상 전환, 위아래로 튀어 오르는 아이콘 애니메이션 적용
- **활성/비활성 처리**: Opacity를 이용한 투명도 전환과 마우스 커서 변화

### 내비게이션 & 흐름 (Navigation & Flow)
- **페이지 전환**: 다음 라우트로 넘어갈 때 자연스러운 크로스페이드(Crossfade) 및 공유 요소 간 트랜지션
- **탭 스위치**: 움직임을 따라가는 슬라이드 인디케이터 라인, 콘텐츠 Fade/Slide
- **캐러셀/슬라이더**: 부드러운 Transform 제어, 스냅 포인트, 스와이프 시 텐션(Momentum)
- **스크롤 효과**: 패럴랙스 깊이감, 일정 구간 도달 시 끈적하게 달라붙는 스티키 헤더, 스크롤 진행률 표시

### 피드백 및 안내 (Feedback & Guidance)
- **호버 힌트**: 부드럽게 Fade-in 되는 툴팁, 커서 모양 변경, 요소 강조 효과
- **드래그 앤 드롭**: 요소가 떠오르는 부양 효과 (그림자 증가 + 사이즈업), 드랍 존 강조, 놓았을 때 부드러운 원위치
- **복사/붙여넣기**: 완료 시 짧은 섬광(Flash) 시각 효과, '복사됨' 안내 툴팁
- **오토 포커스 동선**: 사용자가 입력해야 하는 워크플로나 폼의 순서를 하이라이트로 집중시킴

### 즐거움을 주는 요소 (Delight Moments)
- **빈 상태(Empty states)**: 일러스트레이션이 공중에 가볍게 둥둥 떠 있는 듯한 애니메이션
- **작업 완료 순간**: 성공을 축하하는 꽃가루(Confetti), 화려하게 그려지는 체크 마크
- **이스터 에그**: 숨겨진 인터랙션을 통한 의외의 긍정적 발견 제공
- **맥락적인 모션**: 날씨, 시간대별 테마, 계절적 요소를 반영한 섬세한 효과

## 기술적 구현

각 상황에 맞는 적절한 기술적 해법을 사용하세요:

### 타이밍 & 이징 (Timing & Easing)

**목적별 지속 시간(Durations):**
- **100-150ms**: 즉각적인 피드백 반응 (버튼 누름, 토글 스위치)
- **200-300ms**: 상태 변화 (호버 작동, 메뉴 열림)
- **300-500ms**: 레이아웃 변화 (아코디언 전개, 모달 창 노출)
- **500-800ms**: 진입 애니메이션 (페이지 랜딩 로드)

**이징 곡선 (브라우저 기본값이 아닌 아래 값 사용):**
```css
/* 권장 - 자연스러운 감속 */
--ease-out-quart: cubic-bezier(0.25, 1, 0.5, 1);    /* 부드럽고 세련됨 */
--ease-out-quint: cubic-bezier(0.22, 1, 0.36, 1);   /* 조금 더 경쾌하고 빠름 */
--ease-out-expo: cubic-bezier(0.16, 1, 0.3, 1);     /* 자신감 있고 명확한 정지 */

/* 피해야 할 값 - 촌스럽고 산만해 보임 */
/* bounce: cubic-bezier(0.34, 1.56, 0.64, 1); */
/* elastic: cubic-bezier(0.68, -0.6, 0.32, 1.6); */
```

**종료(Exit) 애니메이션은 등장(Enter) 애니메이션보다 빠르게 처리합니다.** 진입 시간의 약 75% 수준을 권장합니다.

### CSS 애니메이션
```css
/* 선언형의 간단하고 직관적인 모션에 적합 */
- 상태 변화 시 transitions 활용
- 복잡한 순차 연출 시 @keyframes 사용
- 성능을 위해 transform과 opacity만 조합하여 사용 (GPU 가속 대상)
```

### JavaScript 애니메이션
```javascript
/* 복잡하거나 인터랙션 연계가 필요한 모션에 적합 */
- 프로그래밍 방식의 제어 시 Web Animations API 활용
- React 환경이라면 Framer Motion을 적극 권장
- 더 복잡한 모션 시퀀스라면 GSAP 도입
```

### 퍼포먼스 제어 (Performance)
- **GPU 가속**: layout 속성 (width/height/top/left 등) 조작을 자제하고 `transform`과 `opacity` 위주로 제어
- **will-change**: 모든 곳에 바르지 말고 브라우저의 부하가 예상되는 곳에만 제한적으로 사용
- **리페인트 방지**: 렌더 트리 재계산을 피하고 특정 컴포넌트에 필요하다면 `contain` 속성 부여
- **모니터링 (FPS)**: 타겟 디바이스에서 화면 주사율 60fps가 안정적으로 떨어지는지 확인

### 웹 접근성 (Accessibility)
```css
@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
```

**절대 금지 (NEVER)**:
- bounce나 elastic과 같은 이징 커브 사용 금지 — 촌스럽고 애니메이션 그 자체로만 튀어 보입니다.
- 레이아웃 속성 애니메이팅 금지 — 너비(width), 높이(height), 위치(top/left) 대신 항상 `transform`을 활용하세요.
- 사용자 피드백 애니메이션 시간을 500ms 이상으로 지연시키기 — 랙(렉)이 걸린 것처럼 느껴집니다.
- 목적 없이 움직임 추가하기 — 모든 모션 효과에는 타당한 이유가 존재해야 합니다.
- `prefers-reduced-motion` 무시하기 — 이는 웹 접근성 위반입니다.
- 요소마다 애니메이션 도배하기 — 정보 피로도를 유발하여 오히려 어수선하고 피곤해집니다.
- 연출이 다 끝날 때까지 렌더링을 완전히 막아 사용자의 액션을 차단하기

## 품질 검증

마지막으로 품질을 확인하세요:

- **60fps의 부드러운 프레임**: 대상 디바이스에서 끊김(Jank) 현상 없이 자연스럽게 재생되는가.
- **자연스러움**: 설정된 이징 곡선이 기계적이지 않고 유기적인가.
- **타이밍 밸런스**: 지나치게 재빠르고 경박하거나 (Jarring), 고의적으로 늦춰져서 랙처럼(Laggy) 체감되지는 않는가.
- **애니메이션 감소 옵션**: 접근성 선택에 맞게 필수 사항 외에는 효과가 차단 혹은 단순화되는가.
- **주도권 침해 없음**: 요소가 변화하는 도중이나 직후에도 사용자가 정상적으로 애플리케이션을 조작할 수 있는가.
- **가치 입증**: 인터페이스를 더욱 직관적으로 돕거나 사용자에게 기분 좋은 시각적 가치를 되돌려주었는가.

기억하세요: 모션의 본질적 의미는 단순 장식물이 아닌, 인터페이스의 이해도를 높이면서도 알맞은 피드백을 전달하는 데 있습니다. 분명한 목적을 가지고 퍼포먼스 제약과 웹 접근성을 필시 고려하여 연출하세요. 훌륭한 애니메이션 디자인은 그 자체를 의식할 여유도 없이 마치 현실에 존재하는 물리 세계처럼 그 자체로 완벽하게 어우러지는 상태입니다.
