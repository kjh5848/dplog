영혼 없이 싸구려 기본값처럼 보이거나, 삐뚤빼뚤 일관성이 무너진 그지같은 텍스트 뭉치들을 평가하고 개조하여 의도가 명확하고 장인정신이 깃든(Well-crafted) 고급 타이포그래피 시스템으로 환골탈태 시키십시오.

## 필수 준비 지상명령 (MANDATORY PREPARATION)

`frontend-design` 스킬을 활용하세요 — 여기에는 디자인 원칙, 안티 패턴 및 **컨텍스트 수집 프로토콜(Context Gathering Protocol)**이 포함되어 있습니다. 작업을 시작하기 전에 반드시 지침을 따르십시오 — 아직 디자인 컨텍스트 정보가 없다면 무엇보다 먼저 `teach-impeccable`을 실행해야 합니다.

## 1. 현재 타이포그래피 참사 현장 진단 (Assess Current Typography)

도대체 어느 부분이 촌스럽고(Generic) 시스템이 무너졌는지 날카롭게 분석하십시오:

1. **글꼴 선택의 폭력성 (Font choices)**:
   - 아무 생각 없이 투명인간 같은 브라우저 기본 폰트나 시스템 폰트(Inter, Roboto, Arial, Open Sans)를 남발하고 있는가?
   - 폰트가 앱의 영혼(Brand personality)과 매칭되는가? (개구지고 장난스런 브랜드에 딱딱한 공문서 고딕체를 처박아두진 않았는가?)
   - 미친 폰트 뷔페를 차려놨는가? (한 화면에 폰트 패밀리가 3~4개를 넘어가면 100% 개판입니다).

2. **계급의 실종 (Hierarchy)**:
   - 0.1초 만에 쳐다봤을 때 대가리(Headings)와 몸통(Body)과 발가락(Captions) 텍스트가 한눈에 구별되는가?
   - 폰트 크기 차이가 개미 눈곱만한가? (14px, 15px, 16px 이딴 식으로 쪼개놓으면 계층 구조는 똥물처럼 섞여버립니다).
   - 폰트 두께(Weight) 대비가 살아있는가? (Medium이랑 Regular 두께를 붙여놓으면 시각적 구분이 아예 안 됩니다).

3. **비례 스케일의 무법천지 (Sizing & scale)**:
   - 철저한 사다리표(Type scale) 공식에 의해 크기가 통제되고 있는가, 아니면 개발자가 그날 기분 따라 사이즈 숫자를 찍어발랐는가?
   - 본문 텍스트가 인간의 눈으로 편하게 읽을 수 있는 최소 생존선(16px 이상)을 지키고 있는가?
   - 텍스트 크기 지정 전략이 맥락에 맞는가? (보수적인 앱 UI는 고정 `rem` 스케일을 써야 하고, 화려한 마케팅/랜딩 페이지 대문짝 글씨는 유체 스케일 `clamp()`를 써야 합니다).

4. **망막 파괴 가독성 (Readability)**:
   - 한 줄의 길이가 읽기 짜증날 정도로 긴가? (한 줄당 45~75자가 인간 인내심의 한계입니다).
   - 폰트 종류와 크기에 맞게 행간(Line-height)이 숨 쉴 틈을 주고 있는가?
   - 글자색과 배경색이 보호색을 띠어서 시각 장애를 유발하진 않는가(Contrast)?

5. **일관성 붕괴 (Consistency)**:
   - 이 페이지 제목 폰트 다르고, 저 페이지 제목 폰트 다르게 찌그러져 있는가?
   - 같은 신분의 텍스트인데 폰트 두께가 지 맘대로 왔다 갔다 하는가? (A버튼은 Bold, B버튼은 Semibold).
   - 글자 간격(Letter-spacing)을 치수별로 고정하지 않고 방치하여 간판이 삐뚤어져 보이는가?

**명심하십시오 (CRITICAL)**: 당신의 임무는 텍스트를 "화려하고 예쁘게(Fancier)" 꾸미는 게 아닙니다. 훨씬 더 명징하고(Clearer), 술술 읽히고(Readability), 확고한 질서(Intentional)를 부여하는 것입니다. 훌륭한 타이포그래피는 투명하게 메시지를 전달하지만, 병신 같은 타이포그래피는 똥파리처럼 시선을 교란시킵니다.

## 2. 타이포그래피 시스템 수술 계획 (Plan Typography Improvements)

스케일표, 폰트 짝짓기(Pairing), 로딩 전략에 대한 치밀한 가이드는 `frontend-design` 스킬의 [typography reference](reference/typography.md)를 반드시 참고하십시오.

체계적인 뼈대를 세우십시오:

- **글꼴 체인지업 (Font selection)**: 싹 다 갖다버리고 교체해야 하는가? 브랜드의 영혼을 대변할 최적의 폰트는 무엇인가?
- **계단 스케일 확립 (Type scale)**: 딱 떨어지는 배수 비율(예: 1.25 단위)을 정해서 군기 바짝 든 모듈형 스케일표를 강제하십시오.
- **두께별 계급 부여 (Weight strategy)**: 본문 노예는 Regular 돌격, 캡션은 Medium, 거대한 대가리 헤더는 Bold. 이런 식으로 역할극을 완벽하게 부여하십시오.
- **숨통 규격화 (Spacing)**: 행간, 자간, 그리고 각 텍스트 덩어리 사이의 여백(Margins) 공식을 세우십시오.

## 3. 메스 들고 체계적으로 찢어고치기 (Improve Typography Systematically)

### 글꼴 영입 작전 (Font Selection)
폰트 교체가 승인되었다면:
- 유저 타겟과 브랜드의 성격(Personality)을 100% 대변하는 얼굴마담을 섭외하십시오.
- 두 폰트를 섞어 쓸 거라면(세리프 + 산세리프 콤보, 기하학 폰트 + 휴머니스트 콤보) 확실하게 갭 차이(Contrast)가 나는 극단적 녀석들을 짝지어주거나, 아예 폰트 패밀리 단 하나만 가지고 굵기(Weights) 조절로 승부를 보십시오.
- 브라우저 로딩하다 화면 폰트가 출렁이는 깜빡임(Layout shift) 병크를 터뜨리지 않도록 방어 코드(`font-display: swap`, 뼈대 맞춘 Fallback 폰트)를 확실히 세팅하십시오.

### 서열 5단계 계급화 (Establish Hierarchy)
명확한 타입 스케일을 구축하십시오:
- **딱 5가지 사이즈면 세상의 모든 문자를 통제가능합니다**: 초소형(Caption), 보조(Secondary), 본문(Body), 부제목(Subheading), 대가리(Heading).
- **고정 비율 계단 (Consistent ratio)**: 단계별로 넘어갈 때 일정한 수학적 비율(1.25배, 1.333배, 1.5배 등)을 철저하게 준수하십시오.
- **다차원 믹스 갈구기**: 폰트 사이즈 큰 거 하나만 믿고 나대지 마십시오. 글씨를 줄이더라도 '색상 농도'와 '두께'와 '여백'을 버무리면 충분히 강력한 서열을 창조할 수 있습니다.
- **앱 UI 철칙 (App UIs)**: 모니터 사방 막혀 있는 정보 밀집형 대시보드에서는 고정된 `rem` 스케일을 못 박고 한두 군데(Breakpoints)에서만 꺾이게 하십시오. 미친 듯 치수 변하는 유체(Fluid) 폰트가 이 구조를 개박살 내게 두지 마십시오.
- **마케팅/대문 페이지 (Marketing / Content)**: 여기 대문짝만한 제목들은 브라우저 창 쪼그라들 때 우아하게 같이 축소되는 마법의 유체 스케일 `clamp(최소, 기본, 최대)`를 먹여버리십시오. 단, 본문 텍스트는 유체로 만들지 말고 고정하십시오.

### 시각 파괴 방지 (Fix Readability)
- 글이 끝없이 이어지는 본문 박스는 텍스트 전용 단위 `ch`를 써서 `max-width: 65ch` 정도로 강제 단절시켜 가두십시오.
- 상황에 따라 행간(Line-height)의 숨통을 조였다 푸십시오: 덩치 큰 헤딩은 바짝 조이고(`1.1 - 1.2`), 자잘한 본문 텍스트는 답답하지 않게 확 푸십시오(`1.5 - 1.7`).
- 어두운 다크 모드(Dark theme)에 선명한 하얀 글씨를 박을 땐 텍스트가 번져 보이니 행간 공간을 평소보다 미세하게 더 벌려주십시오.
- 스마트폰 시대에 본문 텍스트가 `16px / 1rem` 밑으로 기어 내려가 눈알을 괴롭히게 방치하지 마십시오.

### 악마의 디테일 교정 (Refine Details)
- 표 안에 숫자나 가격표를 일렬 종대로 엑셀처럼 빡 맞추려면 반드시 등폭 숫자 폰트 세팅(`font-variant-numeric: tabular-nums`)을 켜두십시오.
- 캡션이나 대문자(Uppercase) 범벅 텍스트는 글자 사이 숨 막히니까 자간을 슬쩍 벌려주고(Open letter-spacing), 대문짝(Display text)만한 폰트들은 빈틈이 너무 벙벙하게 보이니까 디폴트 두거나 살짝 마이너스 조이기(Tight)를 타십시오.
- CSS 변수를 지을 때 쓰레기같이 숫자 박힌 이름(`--font-16`) 당장 폐기하고, 이놈의 존재 책무가 담긴 시맨틱 토큰 네임(`--text-body`, `--text-heading`)으로 개조하십시오.
- 여유 있으면 `font-kerning: normal` 박아주고 고오급 OpenType 피처들 있으면 쏵 다 긁어오십시오.

### 두께별 신분증 발급 (Weight Consistency)
- 각 두께별로 "너는 이 역할만 해" 라고 신분증을 끊어주고 그 룰을 사수하십시오.
- 두께 3~4개(Regular, Medium, Semibold, Bold)면 황제가 웹페이지 씹어먹고도 남습니다. 다 섞어 쓰지 마십시오.
- 웹사이트 속도 깎아먹는 주범이니까 진짜 화면에 노출시키는 그 두께 폰트 파일만 딱 골라서 로드해오십시오.

**절대로 하면 안 되는 자살골 (NEVER)**:
- 폰트 패밀리 2~3개 넘겨서 화면을 잡탕밥으로 만들기
- 폰트 사이즈를 느낌 가는 대로 아무 숫자나 던져서 스케일 망가뜨리기
- 본문 텍스트를 쪼잔하게 16px 아래로 찌그러뜨리기
- 가독성 병신인 멋부리기용 특수(Decorative/Display) 폰트를 긴 본문 글줄에 억지로 쓰기
- "아 화면 깨지네 ㅋ" 하면서 모바일 브라우저 강제 줌 옵션 막아버려서 장애인들 엿 먹이기 (`user-scalable=no` 금지)
- 폰트 사이즈 단위에 무식하게 고정 `px` 쓰기 — 유저 보이스웨어 세팅 다 무시하는 적폐니까 무조건 상대 비율 `rem` 쓰기
- 이 디자인은 미친 "개성(Personality)"이 생명인 프로젝트인데 귀찮다고 밍밍한 Inter/Roboto/Open Sans 우려먹기
- 두 개 폰트를 섞어 쓸 거면 아예 다르게 생겨먹은 놈을 섞어 쓰지, 눈으로 구별도 안 가는 비슷한 특징의 기하학 산세리프체 2개를 병신같이 붙여놓기

## 4. 최종 활자 검열 (Verify Typography Improvements)

- **계급 식별 (Hierarchy)**: 눈 한 번 깜빡였을 때 헤딩창과 본문과 꼬리말들이 군번줄 찬 것처럼 즉각 서열 분리가 되는가?
- **눈알 안구 건조증 (Readability)**: 이딴 긴 텍스트를 3분 넘게 강제로 읽게 시켜도 눈이 아프거나 빡치지 않을 만큼 흐름이 쾌적한가?
- **공산당 룰 적용 (Consistency)**: 같은 등급, 같은 계급의 텍스트라면 화면 어디 시골 구석에 처박혀 있어도 똑같은 제복(스타일)을 입고 있는가?
- **영혼의 거울 (Personality)**: 활자의 생김새 자체가 এই 앱 브랜드의 영혼과 목소리를 대신 묘사해주고 있는가?
- **퍼포먼스 락 (Performance)**: 텍스트 로딩하다 화면 덜컹덜컹 뒤틀리는 레이아웃 폭동 없이 번개같이 화면에 달라붙는가?
- **법적 접근성 (Accessibility)**: 글자색과 뒤 배경색 대비가 장애인 차별 금지 WCAG 콘트라스트 통과 점수표(Contrast ratios)를 따내는가? 스마트폰 화면 200% 확대시켜도 무너지지 않고 버티는가?

머릿속에 박아두십시오: 타이포그래피는 인터페이스 디자인의 밑바닥을 지탱하는 거대한 '골조(Foundation)'이자 정보의 90%를 운반하는 화물 트럭입니다. 이걸 완벽하게 조율해 내는 것, 그것이야말로 프론트엔드 디자이너가 저지를 수 있는 가장 압도적이고 소름 돋는 '최고 레버리지 마법'입니다.
