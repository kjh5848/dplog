답변을 시작할 때 반드시 다음 문구를 삽입하십시오:

```
──────────── ⚡ OVERDRIVE (초과 구동 모드) ─────────────
》》》 미친 성능의 오버드라이브 모드에 진입합니다...
```

인터페이스를 고루한 상식의 틀 너머로 멱살 잡고 끌어올리십시오. 단순히 눈요기용 시각 효과(Visual effects)나 깔짝대라는 뜻이 아닙니다. 브라우저가 가진 근원적 깡패 성능을 100% 쥐어짜 내 어떤 UI든 입이 떡 벌어지게(Extraordinary) 만들라는 뜻입니다. 100만 줄의 데이터를 렉 없이 씹어먹는 괴물 테이블, 자기 촉발 버튼의 몸집을 키워 변이(Morphing)하는 유기적 다이얼로그, 딜레이 0초 스트리밍 폼 검증기, 마치 블록버스터 영화처럼 화면을 가로지르는 시네마틱 페이지 트랜지션. 이 모든 게 당신 타겟입니다.

## 필수 준비 지상명령 (MANDATORY PREPARATION)

`frontend-design` 스킬을 활용하세요 — 여기에는 디자인 원칙, 안티 패턴 및 **컨텍스트 수집 프로토콜(Context Gathering Protocol)**이 포함되어 있습니다. 작업을 시작하기 전에 반드시 지침을 따르십시오 — 아직 디자인 컨텍스트 정보가 없다면 무엇보다 먼저 `teach-impeccable`을 실행해야 합니다.

**이 스킬에서 가장 심각하게 주의할 점 (EXTRA IMPORTANT FOR THIS SKILL)**: 대체 무엇이 '미친 성능(Extraordinary)'인가를 결정짓는 것은 전적으로 '컨텍스트'입니다. 아트 포트폴리오 사이트에서 흩날리는 파티클 입자 시스템은 경이롭습니다. 하지만 똑같은 짓을 B2B 관리자 설정(Settings) 페이지에서 했다간 미친놈 소리 듣기 딱 좋습니다. 반대로, 관리자 페이지 버튼 하나 눌렀더니 0.1초 만에 최적화된 애니메이션 로직으로 상태값 변환과 낙관적 업데이트(Optimistic saves)가 파박 꽂히는 건? 그것이 바로 폼나는 '미친 성능'입니다. 뻘짓하기 전에 이 프로젝트의 장르와 성격부터 똑바로 파악하십시오.

### 무턱대고 키보드부터 잡지 마라 (Propose Before Building)

이 스킬은 헛발질(Misfire)할 확률이 미친 듯이 높은 폭탄입니다. 무작정 코드 에디터부터 키면 안 됩니다. **당신은 무조건 다음을 선행해야 합니다:**

1. **2~3개의 완전히 차원이 다른 작전 방향 구상**: 뇌를 쥐어짜서 완전히 다른 레벨의 기술(Techniques)과 극단적 미학(Aesthetic) 접근법 다수를 고안하십시오. 각 방향마다 완성본이 어떻게 동작하고 어떻게 피부에 와닿을지(Feel) 시각적으로 묘사하십시오.
2. **유저에게 직설적으로 캐물어 확답받기**: 당신 마음대로 유추하지 마십시오. 유저에게 구상한 옵션들을 들이밀고, 도대체 뭘 진심으로 원하는지(Pick) 확인받기 전까진 단 한 줄의 코드도 짜지 마십시오. 브라우저 지원 여부, 성능 폭발, 오버엔지니어링 코스트 같은 잔인한 트레이드오프(Trade-offs)도 낱낱이 자백(Explain)하십시오.
3. **유저가 컨펌(Confirm) 때린 딱 그 방향으로만 닥돌하십시오.**

이 3단계를 무시하고 혼자 신나서 만들었다간, 쪽팔림과 함께 코드 뭉치 전체를 쓰레기통에 쳐박아야 할 것입니다.

### 브라우저 시뮬레이션 광기 (Iterate with Browser Automation)

이 따위 야심 찬 고난도 기술들은 태어나서 절대 한 방에 안 돌아갑니다. 기술이 꼬입니다. 브라우저 자동화 툴(Browser automation tools)을 무자비하게 돌려대며 직접 눈으로 프리뷰(Preview)하고 또 매만져야(Iterate) 합니다. "내 코드는 완벽해"라는 등신 같은 착각은 버리십시오. 직접 눈으로 렉 걸리는지, 선이 튀는지 뜯어보십시오. "기술적으로 굴러가는 것" 과 "우와, 쩐다" 사이의 거대한 심연은, 오직 수십 번의 피를 깎는 화면 시각 조율(Visual iteration)만으로 메울 수 있습니다.

## 지금 이 앱에서 "쩌는(Extraordinary) 게" 뭔지 진단하라

무슨 하이테크를 바를지 결정하기 전에 멱살 잡고 스스로 물어보십시오: **"이딴 데서 대체 뭘 해줘야 유저들이 '개쩌네' 라고 육성으로 탄성을 질러줄까?"**

### 시각/세일즈용 마케팅 전시장 (For visual/marketing surfaces)
일반 랜딩 페이지나 포트폴리오 대문 상단 화면(Hero) — 이 바닥에서의 '와우 포인트'는 오직 감각적 시냅스 폭발뿐입니다: 스크롤에 맞춰 심장이 멎을 듯 폭발하는 3D 요소, 셰이더(Shader)가 지배하는 무빙 백그라운드, 블록버스터 뺨치는 페이지 시네마틱 트랜지션, 마우스 커서를 따라 생동하는 생성형 입자 애니메이션.

### 실전 기능용 뼈대 뷰 (For functional UI)
무식한 데이터 테이블, 입력 폼, 다이얼로그, 내비게이션들 — 이 바닥의 '와우 포인트'는 촉감(FEEL)에서 나옵니다: 쪼그만 하단 트리거 버튼이 뷰 트랜지션(View Transitions)을 타며 부드럽게 대형 팝업 창으로 유기적으로 변태(Morphing)하는 것. 10만 줄의 개노답 데이터를 60fps로 매끄럽게 비벼주는 가상화 스크롤. 타자 치자마자 서버 핑 통신조차 안 느껴질 만큼 번개처럼 응답하는 폼 검증. 요소 끌어다 놓을 때 찰지게 감기는 스프링 텐션 드래그 앤 드롭 물리엔진.

### 초당 0.01초 깎는 퍼포먼스 결벽 UI (For performance-critical UI)
여기의 '와우 포인트'는 눈으론 안 보이지만, 세포로 체감하는 영역입니다: 5만 개짜리 리스트에서 검색 필터 넘길 때 픽셀 하나 안 깜빡거리는 것. 스크립트 수십만 줄이 깔려도 메인 스레드 락 걸리는 일 없이 코골며 자고 있는 UI. 방대한 이미지 업로드 처리를 거의 실시간 급으로 후려쳐 버리는 기적. 단 1프레임도 버벅거림(Hesitates)을 허용치 않는 강박증.

### 헤비급 데이터 관제탑 (For data-heavy interfaces)
수십 개의 차트와 대시보드 — 이 바닥의 '와우 포인트'는 유체 이동(Fluidity)입니다: 거대한 데이터를 DOM 하나 없이 캔버스(Canvas)나 WebGL을 태워 GPU 가속으로 그려버리는 스킬. 차트 A에서 차트 B로 덜컥! 바뀌는 게 아니라 세포 분열하듯 유기적으로 스르륵 형태가 바뀌는(Morph) 데이터 애니메이션 로직. 중력에 이끌려 자리를 찾아가는 강제 방향 그래프 레이아웃.

**단 하나의 핵심 통찰**: 당신이 부리는 그 흑마법 같은 구현 레벨이 기존 사람들이 '웹사이트 따위'에서 기대하는 뻔한 한계를 완전히 박살 내버려야 합니다. 그 압도적 기술(Technique)은 오직 이 경험치(Experience)를 최강으로 펌핑업 시켜주기 위해 존재하는 충견일 뿐입니다. 

## 당신의 파괴적 무기고 (The Toolkit)

기술 이름표 따위가 아니라, 우리가 도달해야 할 목적별로 미친 툴들을 분류했습니다.

### 시네마틱 무빙 트랜지션 (Make transitions feel cinematic)
- **View Transitions API** (SPA: 모든 브라우저 됨 / 멀티 페이지: 파폭 안 됨) — 이전 화면과 다음 화면 요소를 하나의 오브젝트처럼 공유해서 스무스하게 찰흙 처럼 버무리기(Morphing). 콩알만 한 썸네일 리스트 조각이 확대되면서 상세 뷰의 간판이 되는 이 미친 기술은 사실상 웹 네이티브급 모션을 씹어먹는 끝판왕.
- **`@starting-style`** (전 기종 평정) — `display: none` 에서 화면에 '뿅' 튀어나오는 요소를, 그지같은 자바스크립트 버벅임 안 타고 순수 CSS 엔진만으로 엔트리 키프레임 구동시켜 깔끔하게 등장시키기.
- **스프링 물리 엔진 타격감 (Spring physics)** — 밋밋하고 촌스러운 cubic-bezier 버리고 미친 질량(Mass), 장력(Tension), 마찰력(Damping) 수학 공식을 갈아 넣은 현실 세상의 쫀득한 물리엔진. (추천 라이브러리: Framer Motion(현 Motion), GSAP, 아니면 직접 공식 짜서 집어넣어라).

### 스크롤 위치를 찢어버리는 직결 엔진 (Tie animation to scroll position)
- **스크롤 타임라인 구동 (`animation-timeline: scroll()`)** — JS 단 한 줄도 안 씀. 순수 CSS 코어 가속빨로 스크롤만 내리면 알아서 타이밍 맞춰 패럴랙스(Parallax), 프로그레스 바, 순차적 오브젝트 진입 효과 발동. (크롬/엣지/사파리 가능; 불쌍한 파폭 유저는 안되니까 반드시 정적인 이미지로 우회로(Fallback) 파놓기).

### CSS 따위를 뛰어넘는 이계의 렌더링 (Render beyond CSS)
- **WebGL** (전 기종 평정) — 미친 셰이더 조명 효과, 후처리 뽀샤시(Post-processing), 파티클 그래픽 시스템. (추천 무기고: Three.js, 가벼운 OGL, regl). 불쌍한 CSS로는 죽었다 깨어나도 못 비빌 비디오 게임 수준의 그래픽 조작질.
- **WebGPU** (크롬/엣지 평정; 사파리 찔끔; 파폭은 플래그 켜야 됨) — WebGL 싸다구 때리는 차세대 GPU 코어 직통 가속 시스템. 존나 막강하지만 아직 세상이 이걸 온전히 못 받아주니 무조건 WebGL2 백업(Fallback) 방어선 쳐놓고 진입.
- **Canvas 2D / OffscreenCanvas** — 지독한 픽셀 그래픽 노가다. 혹은 아예 브라우저 메인 스레드 건드리지도 않고 웹 워커(Web Workers) 뒤에 숨어서 OffscreenCanvas 띄워놓고 변태같이 뒤에서 무거운 화면 그려내기.
- **SVG 지옥 필터 (SVG filter chains)** — 일반적인 CSS 박스 쪼가리들을 일그러뜨리고 산산이 부수고(Morphology) 녹여버리는 이계의(Organic) 픽셀 일그러짐 효과. 심지어 CSS로 애니메이션 키프레임 구동까지 가능.

### 죽어있는 데이터베이스 좀비 살려내기 (Make data feel alive)
- **가상화 스크롤 축지법 (Virtual scrolling)** — 10만 개, 100만 개 데이터 다 부르는 병신 짓 그만두고 현재 유저 눈깔에 보이는 10개만 돌려막기. 좆밥들은 생짜로 짜고, 개빡센 건 TanStack Virtual 깔아서 후려치기.
- **GPU 강행돌파 차트 랜더링 (GPU-accelerated charts)** — 일반 DOM이나 SVG로는 렉 걸려서 뻗을 데이터양 쏟아질 때, 망설임 없이 Canvas나 WebGL로 차트 100만 개 점선 그리기 (deck.gl 등 활용).
- **데이터 흐름 유체 이탈 (Animated data transitions)** — 데이터 갈아 끼울 때 화면 껌뻑이는 구시대적 깜빡이 없이 D3 `transition()`이나 View Transitions로 슬라임처럼 부드럽게 변태 타기(Morph).

### 기형적인 CSS 억지 애니메이션 박아넣기 (Animate complex properties)
- **`@property`** (전 기종 평정) — 원래 CSS로는 중간 변환점 없는 빌어먹을 그라데이션(Gradients) 색상이나 온갖 괴기스러운 난해한 값들을 변수로 강제 세팅해서, 억지로 부드럽게 애니메이션 돌리게 만들기.
- **웹 애니메이션 API 하드코어 조작 (Web Animations API)** — 자바스크립트로 그렸으면서 감히 CSS 네이티브 가동률 코어를 똑같이 빨아먹는 악마의 성능. 애니메이션을 이리저리 조합하고 끊고 리버스 거는 등 미친 안무(Choreography) 레이아웃 짤 때의 근본 뿌리.

### 성능 락 해제 (Push performance boundaries)
- **웹 하청업체 고용 (Web Workers)** — 개무거운 뒤끝 일 처리 싹 다 메인 스레드에서 치워버리기. 이미지 깎고 조이고, 파일 데이터 압축하고, 무식하게 서치인덱스 구동하는 등 브라우저 뻗을 요소 싹 다 뒤로 유배.
- **뒷방 캔버스 조작 (OffscreenCanvas)** — 안 그래도 바빠 죽는 메인 스레드 버리고 Worker 워커 안경갑 뒤에서 무거운 화면 그래픽만 혼자 존나게 그리기.
- **WASM 뇌 이식** — 거의 C++ 네이티브 앱과 맞짱 뜨는 속도로 연산 후려치기. 자바스크립트는 명함도 못 내밀 존나 무거운 영상 파일 변환(Codecs)이나 극악의 물리엔진 시뮬레이션 돌릴 때 필수.

### 유저의 물리 디바이스 강탈 (Interact with the device)
- **Web Audio API** — 일반 mp3 소리 따위가 아님. 유저 마우스나 액션 궤적에 따라 공간감 찢어주는 3D 오디오, 파장 연동형 소리 피드백 튜닝. 단, 유저가 클릭 전엔 브라우저가 처막으니 눈치껏 권한 뚫을 것.
- **기기 심장 꺼내기 (Device APIs)** — 자이로센서(Orientation), 조도 센서(Ambient light), 강제 지리 좌표 동기화. 쓸모없는 데 쓰지 말고 허락 맡고 진짜 필요한 데만 소금 치듯 적용.

**치명적 알림 (NOTE)**: 오버드라이브 스킬은 "이 앱이 어떻게 느끼게 만들 것인가(FEELS)"를 창조하는 것이지, "무얼 하는 기능인가(DOES)"를 건드리는 게 아닙니다. 실시간 소켓 통신을 달고 오프라인 모드를 다는 건 기획자나 백엔드가 할 짓이지 화면 단의 오버드라이브가 아닙니다. 이미 존재하는 이 낡아빠진 기능을 '가장 미친 퀄리티'로 도약시키는 데만 목숨을 거십시오.

## 규율 통제와 집행 (Implement with Discipline)

### 점진적 퇴보는 죽어도 타협 불가한 명령이다 (Progressive enhancement is non-negotiable)

온갖 최첨단 미친 테크를 들이부어도, 그 최첨단 흑마법(API)을 못 받아들이는 구형 브라우저에서도 "화면이 안 찢어지고 원래 목적대로 스무스하게 굴러가게" 떨어지는 백업 방어선(Deagrade gracefully)을 무조건 쳐놔야 합니다.

```css
/* 크롬 신버전 형님들은 이쪽으로 모실게요 스크롤 애니 좐좐 */
@supports (animation-timeline: scroll()) {
  .hero { animation-timeline: scroll(); }
}
/* 저거 지원 안하는 무식한 사파리나 구형 파폭은 아예 이 코드를 읽지 않고 원래 하던 정적 화면 유지 */
```

```javascript
// 최고급 차세대 GPU 사양 돌리는 WebGPU 전용 부하 엔진 
if ('gpu' in navigator) { /* WebGPU 가동 폭주 */ }
// 그런 거 없는 거렁뱅이(?) PC는 무난한 WebGL2 로 구동시켜주기
else if (canvas.getContext('webgl2')) { /* WebGL2 폴백 가동 */ }
/* 진짜 그것마저 다 박살 나면 CSS만 돌려서 최소 밥값 화면 방어 */
```

### 성능 유지 결사대 룰 (Performance rules)

- 무조건 모니터 뚫어지게 **60fps** 방어. 만약 50프레임 이하로 버벅대면 쓰레기니까 잔가지를 쳐내십시오.
- OS에서 **'애니메이션 줄이기(prefers-reduced-motion)' 켰으면 입도 뻥긋 말고 즉시 화려한 이펙트 싹 다 정지**(어기면 처형). 대신 개깐깐하게 정지된 화보 같은 대체(Static alternative) 화면을 깔아둘 것.
- 처음부터 무겁게 WebGL 도화지 다 펼치지 말고 화면 근처로(Viewport) 쓱 밀고 올 때 몰래 로딩 지연시켜 켜두기(Lazy-initialize).
- 스크롤 밖으로 밀려 나간 무거운 렌더링 놈들, 다 잡아다 모가지 따고 메모리 정지(Pause/Kill).
- 당신이 쓰고 있는 M3 맥북이 세상의 표준이라 착각하지 마라. 개나 소나 다 쓰는 저혈압 구형 안드로이드 폰에서도 부드러운지 돌려봐라.

### 악마는 미친 디테일에 숨어있다 (Polish is the difference)

"와 쩐다"와 "시발 개쩐다"의 차이는 마지막 20%의 피 튀기는 노가다에 달렸습니다. 스프링 애니메이션 꿀렁이는 타이밍, 마우스를 뗄 때 쭈욱 미끄러지는 미세한 장력 곡선(Easing curve), 카드 뒤집어질 때 그 뒤를 따라 느릿하게 밀려오는 잔상 그림자 효과. 대충 겉핥기로 '기술 적용 완료 ㅋ' 따위 똥 코드 던지지 마십시오. 이건 무조건 필연적이고 숙명적인 움직임이라는 착각이 들 때까지 지독하게 깎아내십시오(Inevitable).

**절대로 하면 안 되는 자살골 (NEVER)**:
- 명심해라 `prefers-reduced-motion`을 씹고 안 무시하는 것은 선택이 아니라 형사처분 급의 필수 의무(Accessibility)다.
- 삐까번쩍 한답시고 똥폰에서 돌렸더니 브라우저 퍼져버리게 하는 폭탄 투여.
- 작동도 안 하는 개살구 최첨단 API 쓰면서 안전망(Fallback) 안 짜놓고 퇴근하기.
- 유저 클릭도 안 했는데 소리(Sound) 지르며 스피커 터트리기.
- 자기 디자인이나 레이아웃 좆같이 짠 거 무마하려고 화려한 이펙트로 덮어 씌우며 기술적 허세 부리기 (디자인 똥은 `frontend-design` 스킬로 치워라).
- 정신 사납게 한 화면에 미친 효과 5개씩 쑤셔 박기 (미친 포스는 한 방에 몰빵 집중할 때 폭발한다).

## 최후의 심판대 (Verify the Result)

- **입 떡벌림 테스트 (The wow test)**: 이거 생전 처음 보는 사람한테 딱 보여줬을 때 반응이 "어... 예쁘네요" 가 아니라 욕이 나오며 감탄을 터트렸는가?
- **탈것 박탈 테스트 (The removal test)**: 이 화려한 이펙트를 싹 지웠을 때, 갑자기 이 앱이 붕어빵 앙꼬 빠진 것처럼 역체감이 개 심하고 개허무한가? (아무도 눈치 못 챈다면 그딴 연출 거둬치워라).
- **똥기계 내구 테스트 (The device test)**: 휴대폰, 태블릿, 개싸구려 크롬북 다 갖다 대고 비볐는데도 살 떨리게 부드러운가?
- **멀미 환자 구호 테스트 (The accessibility test)**: 애니메이션 정지 옵션 켜놨더니 레이아웃 다 박살 나지 않고 여전히 우아한 자태의 포스터 뷰를 뽑아주는가?
- **주제 파악 테스트 (The context test)**: 씨발 이거 국세청 세금 내는 앱인데 여기다 우주 파티클 폭발시키고 있는 건 아니겠지? 브랜드 톤앤매너 맞는가?

가슴팍 깊숙이 박아두십시오: "기술적 한계 돌파(Technically extraordinary)"란 무슨 어제 나온 따끈따끈한 라이브러리를 써서 뻐기는 게 아닙니다. 평범하기 그지없는 찌그러진 사각형 웹페이지 조각들이 합쳐져 유저가 단 한 번도 상상조차 하지 못한 방식의 아우라를 뿜어내며 모니터 밖으로 튀어나오게 조종하는 겁니다.
