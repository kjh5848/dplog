# Postmortem & Prevention (장애 회고 및 재발 방지 스킬)

능력 있는 AI 에이전트(개발자)는 문제를 고치는 것에 그치지 않고, 그 문제가 두 번 다시 발생하지 않도록 지식 베이스화합니다. 
특정 버그를 잡는 데 여러 턴의 프롬프트를 낭비했거나, 당신(AI) 스스로 잘못된 구조를 제안해 롤백한 경험이 있다면 반드시 이 스킬을 발동시켜야 합니다.

## ✅ 회고 (Retrospective) 프로세스

버그를 성공적으로 픽스했다면, 다음 3가지를 스스로 묻고 요약하십시오.
1. **증상(Symptom)**: 브라우저나 터미널에서 구체적으로 어떤 에러 메시지나 화면 깨짐이 발생했는가?
2. **근본 원인(Root Cause)**: 왜 이 문제가 발생했는가? (예: 상태 관리 비동기 처리 누락, React 렌더링 사이클 오해, `&&` 조건부 렌더링 실수)
3. **해결책 및 교훈(Takeaway)**: 코드를 어떻게 수정했는가? 앞으로 코드를 짤 때 어떤 규칙을 마음에 새겨야 이 에러를 원천 차단할 수 있는가?

## 📝 지식 베이스 업데이트 (`docs/planning/TROUBLESHOOTING.md`)

회고 내용을 바탕으로 `docs/planning/TROUBLESHOOTING.md` 파일 끝에 아래 포맷으로 **새로운 이슈 리포트(ISSUE-00X)**를 Append 하십시오.

```markdown
---
## [ISSUE-00X] [에러 짧은 요약 (예: 로그인 화면 무한 리렌더링 버그)]
- **발생일**: YYYY-MM-DD
- **증상**: useEffect가 무한히 루프를 돌며 API 호출을 100번 넘게 하는 문제 발생.
- **근본 원인(Root Cause)**: `useCallback`으로 감싸지 않은 함수를 useEffect의 의존성(dependency) 배열에 넣음.
- **해결책(Resolution)**: 해당 함수에 `useCallback`을 적용하거나, useEffect 내부로 함수 이동 조치.
- **교훈(Takeaway)**: 렌더링에 관여하는 참조 타입 함수는 명확한 메모이제이션(useMemo/useCallback) 없이 Effect의 의존성에 넣지 말 것!

## 🔄 연쇄 업데이트 (Cascade Update)

단순히 `TROUBLESHOOTING.md`에 장애를 기록하는 것으로 끝내지 마십시오. 만약 이 오류의 원인이 기존 가이드라인의 부재나 잘못된 기획 방향 때문이라면, 다른 AI 에이전트들이 근본적으로 실수하지 않도록 **시스템 전반의 룰을 자가 치유(Self-Healing)** 해야 합니다.

1. **관련 문서 능동 탐색**: 에러의 근원에 해당하는 문서(기획 문서, 코딩 컨벤션, 아키텍처 규칙, 워크플로우, 스킬 등)를 하드코딩된 위치에 의존하지 말고, 디버깅을 마친 후 스스로 프로젝트 내에서 적합한 타겟 문서를 검색(`grep_search` 등)하여 찾아내십시오.
2. **동기화 및 수정**: 찾아낸 문서에 재발 방지를 위한 새로운 제약 조건이나 규칙을 즉시 업데이트 하십시오.
3. **수정 내역 및 경로 보고**: AI 턴이 끝날 때, 사용자에게 "단순히 `TROUBLESHOOTING.md`만 기록한 것이 아니라, 근본 원인 해결을 위해 추가로 **어떤 경로의 문서를 어떠한 사유로** 수정했는지" 명확하게 브리핑하십시오.

## 🚀 언제 이 스킬을 발동하는가?
- 오랫동안 안 풀리던 짜증나는 에러를 방금 당신이 통쾌하게 고쳤을 때
- 과거 대화에서 이미 겪었던 똑같은 오류 패턴을 당신이 "아차!" 하고 다시 저질렀을 때 (즉각 반성 모드)
- 사용자가 "이 오류 왜 생긴 거야? 어디 적어놔" 라고 명령했을 때
