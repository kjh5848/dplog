# 로드맵 재정비 (Update Roadmap) 워크플로우

이 워크플로우는 프로젝트 진행 도중 기능이 추가/삭제되거나, 우선순위가 바뀌거나, 일정이 지연되어 **전체 로드맵을 다시 짜야 할 때** 사용합니다.
단일 문서만 수정하면 문서 간 불일치(정합성 오류)가 발생하므로, 아래 순서대로 4개의 기획 문서를 유기적으로 업데이트해야 합니다.

## 0. 변경 사항 파악 및 Sequential Thinking 설계 (필수)

가장 먼저 사용자가 제시한 변경 사항을 바탕으로, 기존 문서와 어떤 점이 충돌하는지 Sequential Thinking 도구로 분석합니다.
- 무엇이 연기되었는가? (예: 결제 시스템을 뒤로 미루고 대시보드를 앞으로 당김)
- 새로운 기능이 프론트/백엔드 중 어디에 속하는가?
- 일정(기간)은 얼마나 늘어나는가?

모든 변경 결과를 정리하고, 사용자에게 변경될 요약본을 먼저 제시하여 **컨펌(`notify_user`)**을 받으십시오. 승인된 후 1단계로 진입합니다.

## 1. JSON 스키마 현행화 파악 (`roadmap_status.json`)

기계적으로 관리되는 최상단 마스터 JSON 파일을 첫 번째로 업데이트합니다.
- 파일 위치: `docs/planning/roadmap_status.json`
- `overallProgress.phases` 배열 내의 `id`, `name`, `duration`, `status`를 재배치합니다.
- 새로운 기능이라면 `frontend.features` 또는 `backend` 요소에 항목을 추가합니다.
- 이 파일은 나머지 Markdown 문서들을 업데이트하기 위한 '진실의 원천(Source of Truth)' 역할을 합니다.

## 2. 마스터 로드맵 수정 (`ROADMAP.md`)

- 전체 타임라인 테이블(Phase 요약표와 시각적 타임라인 바)을 업데이트합니다.
- 예상 총 기간(예: 약 9~12주)이 변경되었다면 반영합니다.
- JSON의 `overallProgress.phases`와 1:1로 일치해야 합니다.

## 3. 백엔드 상세 로드맵 수정 (`ROADMAP_BACKEND.md`)

- 기능이 백엔드에 영향을 미친다면 해당 Phase 항목으로 이동하거나 새로운 Phase를 추가합니다.
- 신규 기능 관련 DTO/테스트/인프라 요구사항 체크리스트(`- [ ]`)를 추가합니다.

## 4. 프론트엔드 상세 로드맵 수정 (`ROADMAP_FRONTEND.md`)

- 기능이 프론트엔드에 영향을 미친다면 해당 Phase 항목으로 재배치합니다.
- UI 컴포넌트, 상태 관리 로직, 페이지 라우트 등 신규 기능 작업 목록(`- [ ]`)을 추가합니다.

## 5. 검증 및 브리핑

모든 문서 수정이 완료되면, 변경된 Phase 타임라인을 짧게 요약하여 사용자에게 최종 브리핑합니다.
- JSON 구조에 오류가 없는지 `view_file`로 최종 점검합니다.
- 서로 다른 4개 파일의 Phase 번호와 일정이 모두 동일한지 교차 검증합니다.
